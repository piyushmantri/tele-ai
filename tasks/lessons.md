# Lessons Learned

## [2026-04-28] ‚Äî Neon serverless driver: no `unsafe`, no `query`, no multi-statement SQL
**Rule**: With `@neondatabase/serverless`, the SQL function exposes only two forms: the tagged template (`` sql`SELECT ${x}` ``) and the direct-call form (`sql(text, params)`). There is no `sql.unsafe(...)`, no `sql.query(...)`, and no built-in `sql.transaction([...])` for arbitrary multi-statement strings. A migrator that reads a `.sql` file and tries to send it as one string will throw at runtime.
**Why**: The plan's first migration runner used `sql.unsafe(fileContents)` and the `db/index.ts` helper used `sql.query(text, params)` ‚Äî both compiled cleanly as `any`-typed access but failed at runtime. The verifier caught this only after running `tsc` against a real `index.d.ts`. Migrations would have crashed on the very first boot, blocking acceptance criterion 1.
**Apply when**: Writing any DB code against the Neon serverless driver ‚Äî especially custom migrators, schema scripts, or any "run this raw SQL blob" helper. Split multi-statement files on `;` and execute each via `sql(stmt, [])`. Make DDL idempotent (`CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`) since you cannot wrap the whole file in a single transaction with this driver.

## [2026-04-28] ‚Äî GramJS `EntityLike` rejects `bigint`
**Rule**: GramJS `client.sendMessage(peer, ...)` and similar APIs type `peer` as `EntityLike = string | number | Entity`. Passing `BigInt(chatId)` is a TypeScript error and may or may not coerce at runtime depending on the call path. Convert Telegram IDs to `Number(...)` (safe for IDs < 2^53) or to a string (`peer.toString()`) before handing them to GramJS.
**Why**: Plan stored Telegram chat IDs as `BIGINT` in Postgres (correct ‚Äî they can exceed 2^31), and the natural reflex was to pass `BigInt(chatId)` back into GramJS calls. That fails to typecheck and is fragile at runtime.
**Apply when**: Bridging persisted Telegram IDs (typically `string` after coming back from a `BIGINT` column) into any GramJS call. Use `Number(id)` for entity arguments; keep the string/bigint form only for DB writes and audit logs.

## [2026-04-28] ‚Äî Gemini SDK `Part` union needs a widened type predicate
**Rule**: The `@google/generative-ai` `Part` type is a discriminated union; filtering with `(p): p is { functionCall: FunctionCall } => !!p.functionCall` is too narrow and trips TS2677 ("type predicate's type must be assignable to its parameter's type"). Use `(p): p is Part & { functionCall: NonNullable<Part["functionCall"]> } => !!p.functionCall` instead, or just filter without a predicate and use a non-null assertion downstream.
**Why**: A naive narrow predicate fails compilation and cascades into "possibly undefined" errors at every downstream `fc.functionCall.name` access. The function-call loop is hot-path code; this needs to compile cleanly.
**Apply when**: Implementing a Gemini tool/function-call loop that filters `response.candidates[0].content.parts` for `functionCall` parts. Same pattern applies to filtering for `functionResponse` or `text` parts in the SDK.

## [2026-04-28] ‚Äî `packageManager: "pnpm@9"` is rejected by pnpm v10
**Rule**: The `packageManager` field in `package.json` requires a full semver, not a major-version shorthand. `"pnpm@9"` causes pnpm v10 to warn `Cannot switch to pnpm@9: "9" is not a valid version` and skip the version switch. Use `"pnpm@9.0.0"` (or whatever exact version you want) instead.
**Why**: Corepack and pnpm's self-update logic both expect a parseable semver. The shorthand silently degrades behavior ‚Äî install still works but the pin is non-functional, which defeats the purpose.
**Apply when**: Setting up `packageManager` in any monorepo root `package.json`. Pin to a real version (e.g. `9.15.0`) and update intentionally.

## [2026-04-28] ‚Äî Build workspace packages before type-checking their consumers
**Rule**: In a pnpm workspace where `apps/server` depends on a local `packages/shared`, running `tsc --noEmit` on `apps/server` fails with `Cannot find module '@tele/shared'` until `packages/shared` has been built (its `dist/` exists with `.d.ts` files). Always build shared first.
**Why**: Workspace symlinks resolve to `packages/shared/package.json`, whose `"main"` / `"types"` fields point at `dist/`. Without a build step the compiler sees the package but no types. The error is misleading ‚Äî it looks like a missing dependency, not a missing build artifact.
**Apply when**: Running per-package type checks in CI or locally in a monorepo with `tsc`-built shared packages. Either build shared packages first (`pnpm -F @tele/shared build && pnpm -F @tele/server typecheck`), or configure TS project references with `composite: true` so `tsc -b` handles the build order automatically.

## [2026-04-29] ‚Äî GramJS Poll API: `TextWithEntities` for question/answers, `correctAnswers` on `InputMediaPoll`
**Rule**: In current GramJS versions, `Api.Poll.question` and `Api.PollAnswer.text` are `Api.TextWithEntities({ text, entities: [] })` objects ‚Äî not plain strings. Correct answer for quiz polls goes in `Api.InputMediaPoll.correctAnswers: Buffer[]`, not in `Api.Poll` itself. `Poll.id` and `SendMedia.randomId` must be `bigInt.BigInteger` ‚Äî use `helpers.generateRandomLong()` from the `telegram` package.
**Why**: Telegram migrated from plain strings to `TextWithEntities` across poll fields. Code assuming plain strings compiles if you bypass typechecking but produces a GramJS runtime error or malformed request. The `correctAnswers` placement on `InputMediaPoll` (not `Poll`) is non-obvious from docs alone.
**Apply when**: Any code that constructs `Api.Poll`, `Api.PollAnswer`, or `Api.InputMediaPoll`. Always read the installed `api.d.ts` before writing ‚Äî never assume the field shapes from memory or older docs.

## [2026-04-29] ‚Äî Persisting poll state as synthetic outbound message avoids schema migration
**Rule**: When adding a Telegram poll-send feature, persist the poll as a plain text message `[Poll: <question>] opt1 | opt2 | ...` using the existing `messages` table and `insertMessage` helper. This lets the dashboard render it without schema changes or new UI components.
**Why**: Adding a structured `polls` table would require a migration, new repo functions, new API routes, new dashboard components, and vote-tracking infrastructure ‚Äî all out of scope for an "AI can send polls" feature. The `[Poll: ...]` convention follows the established `[Image: ...]` / `[File: ...]` patterns already in sender.ts.
**Apply when**: Adding new Telegram media types (stickers, polls, location shares, etc.) where the primary goal is AI-initiated sending, not vote/response tracking. Use the synthetic message pattern; defer structured tracking to when vote results must feed back into the AI context.

## [2026-04-30] ‚Äî FK to internal `chats(id)` UUID, not raw `tg_chat_id` BIGINT
**Rule**: When a new table needs to reference a "person/chat" (e.g. `kanban_tasks.assignee_chat_id`, `reminders.target_chat_id`), make the FK reference `chats(id)` (UUID) ‚Äî never `chats.tg_chat_id` (BIGINT). Resolve display names via a `LEFT JOIN chats c ON c.id = t.assignee_chat_id` and a `COALESCE(NULLIF(TRIM(CONCAT_WS(' ', c.first_name, c.last_name)), ''), c.username, c.tg_chat_id::text)` expression.
**Why**: `chats.id` is the established internal PK pattern; using it makes joins trivial and avoids any GramJS coercion (the BIGINT path forces `Number(id)` / `String(id)` conversions in every consumer). It also lets `ON DELETE SET NULL` / `CASCADE` semantics work cleanly. The same pattern is in `reminders.target_chat_id` ‚Äî follow it for parity.
**Apply when**: Adding any new relational column that points at a person, chat, or contact. Always FK to `chats(id)`, never to `tg_chat_id`. Pre-validate the FK in the AI tool handler (`getChatById(id)` ‚Üí `{ ok: false, error: "unknown contact id; call lookup_contacts first" }`) so the model learns in-loop instead of hitting a 500 on the server.

## [2026-04-30] ‚Äî Distinguish "key absent" from "key present, value null" in PATCH/PUT repos
**Rule**: For an `updateTask(id, patch)` repo that supports clearing a nullable FK (e.g. `assignee_chat_id: null` to unassign), check `Object.prototype.hasOwnProperty.call(patch, "assignee_chat_id")` ‚Äî not `patch.assignee_chat_id !== undefined` and not a truthy check. The presence of the key (regardless of value) means "write this field"; absence means "leave it alone". Pair the repo with a zod schema like `z.string().uuid().nullable().optional()` so `null` survives validation.
**Why**: Truthy checks (`if (patch.assignee_chat_id)`) silently drop `null`, making "unassign" impossible. `!== undefined` works only if the JSON parser preserves the distinction (it does), but `hasOwnProperty` is the explicit, intent-revealing form. Also: if the dynamic SET builder finds no real fields it must reject the update rather than running `UPDATE ... SET updated_at = now()` with nothing else ‚Äî Gemini will sometimes call `update_task(id)` with no other args.
**Apply when**: Writing any partial-update repo where some fields are nullable FKs/strings the caller may want to clear. Same pattern applies to AI tool handlers receiving partial JSON args from Gemini.

## [2026-04-30] ‚Äî Tool-call response payloads must be trimmed to keep Gemini context small
**Rule**: When a Gemini tool returns a list (`list_tasks`, `list_chats`, etc.), trim each row to the minimum fields the model needs to act ‚Äî typically `(id, title, status, assignee_id, assignee_name, comment_count)` ‚Äî and explicitly drop heavy fields like `description`, `body`, attachments, full timestamps. Document this in the tool description so the model knows to call a separate `get_*(id)` tool for the full row when needed.
**Why**: Tool responses are echoed back into the model's context on every turn. A board with 200 tasks √ó full descriptions can blow past token budgets and degrade response quality. Trimming at the tool boundary (not in the repo) keeps the REST API rich while keeping the AI loop lean.
**Apply when**: Authoring any AI tool that returns a list. Default to a trimmed projection; only return full rows from singular `get_*(id)` tools. If filters could be expressed in SQL (e.g. `assignee_chat_id`, `status`), prefer SQL over post-fetch JS filtering once the table grows past a few hundred rows.

## [2026-04-30] ‚Äî `WsEvent` discriminated union: verify consumers have no silent `default` branches
**Rule**: When extending the `WsEvent` union in `packages/shared/src/types.ts` with a new `{ type: "..."; payload: ... }` variant, audit existing `useWsEvent` / switch consumers to confirm none have a `default:` branch that would silently swallow the new type. The `useWsEvent<T extends WsEvent["type"]>` generic picks up new variants automatically ‚Äî but only if no consumer has a catch-all that drops them.
**Why**: TypeScript's exhaustiveness checking only fires on `switch` statements with a `never`-typed default. A `default: return;` or missing `default` in a hand-written dispatcher will compile fine and silently drop the new event at runtime ‚Äî invisible until QA notices the dashboard doesn't refresh.
**Apply when**: Adding any new variant to a discriminated union consumed in multiple places. Grep for the union name and audit each consumer; prefer the generic `useWsEvent<"specific:type">(handler)` pattern over a hand-rolled switch so new variants are opt-in by name.

## [2026-04-30] ‚Äî Split AI tool pairs into trimmed-list + full-get; never one tool that does both
**Rule**: When the AI needs both "discover what's available" and "load the full thing", expose two tools: `list_X` returns only `(id, name, description)` for every row, and `load_X({ id?, name? })` returns the full payload for one row. Don't add a `verbose: true` flag to `list_X` ‚Äî the model will pass it.
**Why**: A single `list_X(verbose=true)` lets Gemini one-shot a 200-row √ó full-content response into context, blowing the budget. Two distinct tools force the model to choose `id` first, then load only what it needs. The discoverability tool stays cheap; the load tool is intentionally heavy but scoped to one item. This is the deliberate counterpart to the existing trim-list rule ‚Äî same goal, structural enforcement instead of convention.
**Apply when**: Designing any tool surface for content the model needs to *select among, then read in full* (skills, prompts, documents, templates, runbooks). The list/get split is the right shape; resist the temptation to merge them.

## [2026-04-30] ‚Äî AI tool args: accept both `id` and `name`, prefer `id`, error if neither
**Rule**: For lookup tools where the model might know either an opaque `id` or a friendly `name` (skills, contacts, chats, channels), accept both as optional args, prefer `id` when both are present, and return `{ ok: false, error: "provide id or name" }` when neither is given. Don't make either required.
**Why**: Gemini often remembers a friendly name from earlier in the conversation (`"grail-explorer"`) but not the UUID. Forcing `id`-only makes the model call `list_*` again to look up the id ‚Äî wasted turn. Forcing `name`-only breaks when the user renames. Accepting both with a documented precedence is the friction-free path. The explicit "provide id or name" error teaches Gemini in-loop instead of silently returning an empty result.
**Apply when**: Any AI tool that resolves a single record by some user-meaningful identifier. Same pattern works for `chat`, `contact`, `template`, etc. Document the precedence in the tool description so the model can plan around it.

## [2026-04-30] ‚Äî Path-takes-precedence-over-inline at load time, not at write time
**Rule**: When a record can hold either inline content or a filesystem path pointing at the source of truth (e.g. `skills.content` vs `skills.path`), store both columns but resolve at *load* time: if `path` is non-null, read the file fresh; otherwise use `content`. Do NOT sync `content` from `path` on write ‚Äî leave `content = ''` when `path` is set.
**Why**: Editing a SKILL.md file on disk should automatically be reflected the next time the AI loads the skill. If you eagerly cache the file's contents into `content` on write, the row goes stale the moment the file changes. Storing `content = ''` alongside a non-null `path` makes the precedence visually obvious in the DB and prevents drift.
**Apply when**: Any "ref-or-inline" data model ‚Äî skills, prompt templates, config blobs, image-by-URL-or-binary. Document the precedence in the schema comment and enforce at the load boundary, not the write boundary.

## [2026-04-30] ‚Äî Catch Postgres unique-violation (`23505`) and return 409, never let it bubble as 500
**Rule**: When a route inserts/updates a row with a `UNIQUE` column the user controls (e.g. `skills.name`, `chats.username`), wrap the DB call in `try { ... } catch (err) { if (err?.code === "23505") return res.status(409).json({ error: "name already exists" }); throw err; }`. Apply on both POST (create) and PUT (update ‚Äî rename collisions hit the same constraint).
**Why**: Postgres throws `23505` for unique violations; without the catch, Express returns 500 with a stack trace and the dashboard shows "Server error" ‚Äî useless to the user, who just typed a duplicate name. A targeted 409 with a friendly message lets the form display the real problem ("name already exists") inline. Forgetting to apply this on PUT is the common miss ‚Äî only POST gets remembered.
**Apply when**: Any route writing to a column with a `UNIQUE` constraint that's user-supplied. Audit both create AND update paths. Same pattern for FK violations (`23503`) ‚Üí 400 "referenced row does not exist".

## [2026-04-30] ‚Äî Mirror create-time validation in update routes ‚Äî partial PATCH/PUT often skips invariants
**Rule**: When a CREATE route enforces "at least one of A or B must be provided" (e.g. `content` or `path` for skills), the corresponding UPDATE/PUT route MUST re-check the invariant after applying the patch. A naive `PUT { content: "", path: null }` will happily clear both and leave the row in a state CREATE would have rejected. Read the current row, apply the patch in memory, then re-validate before persisting.
**Why**: The verifier flagged this exact gap on the skills PUT route ‚Äî POST rejected `{content: "", path: null}` but PUT accepted it, leaving a skill that loads to empty instructions. Partial-update endpoints inherit none of the create-time invariants for free. Either re-validate after merging the patch, or document the invariant as "create-only" and reject the field in the update schema.
**Apply when**: Any UPDATE/PATCH/PUT route on a table whose CREATE has cross-field validation (mutual exclusivity, "at least one of", "exactly one of"). Walk through every CREATE-time check and confirm it still holds after the partial update.

## [2026-04-30] ‚Äî `enabled`/visibility checks must be re-applied at every consumption point, not just at list time
**Rule**: When a row has an `enabled` flag and the dashboard hides it from a `list_*` tool by filtering on `enabled = true`, the corresponding `load_*` / `get_*` / `invoke_*` consumer MUST re-check `enabled` after the lookup and refuse if disabled. Hiding from list does not protect against stale ids ‚Äî the model may have remembered an id from earlier in the conversation, before the user toggled it off.
**Why**: Verifier flagged this on `load_skill`: `list_skills` filters `enabled=true`, but `load_skill(id)` does not, so a model with a stale id from earlier in the same conversation can still load a now-disabled skill. The same applies to MCP servers, prompt templates, scheduled jobs ‚Äî anything with a soft-disable flag. Defense-in-depth at every consumer, not just the discovery surface.
**Apply when**: Any soft-disable flag (`enabled`, `active`, `archived = false`) on a table consumed by multiple tools or routes. Treat the flag as an authorization check that must run at every read path, not just the listing path.

## [2026-04-30] ‚Äî `pnpm -F <pkg> typecheck` fails silently when no `typecheck` script exists
**Rule**: `pnpm -F @tele/server typecheck` exits with `ERR_PNPM_RECURSIVE_RUN_NO_SCRIPT` if the package has no `"typecheck"` script in its `package.json`. The error is easy to mistake for "no typecheck needed" or "passed". Always either add `"typecheck": "tsc --noEmit"` to every package's `package.json`, or fall back to `cd <pkg> && npx tsc --noEmit` and check the exit code explicitly.
**Why**: A plan step that says "run typecheck" can read as PASS when it actually never ran. The verifier on this task caught it only by re-running with `npx tsc --noEmit` directly. Repo hygiene fix: add the script to every package; CI fix: assert the script exists before running.
**Apply when**: Writing or following a plan step that invokes `pnpm -F` with a script name. Either confirm the script is defined first (grep `package.json`) or use the direct `npx tsc --noEmit` form. Same trap applies to `lint`, `test`, etc. ‚Äî pnpm recursive scripts are silent on absence.

## [2026-04-30] ‚Äî Seed rows with machine-specific paths are fine if idempotent and user-editable
**Rule**: It's acceptable to ship a migration with a hard-coded `INSERT ... ON CONFLICT DO NOTHING` containing a developer-local path (e.g. `/Users/foo/Downloads/grail-explorer/SKILL.md`) provided: (1) the seed is idempotent so re-runs on a clean DB don't error, (2) the dashboard exposes a way to delete or re-point the row, and (3) the path is documented as "edit me on a fresh install" in the plan. Don't gate the migration on the path's existence at boot.
**Why**: For a single-user tool, blocking the migration on a missing local file would break onboarding for any other developer. `ON CONFLICT DO NOTHING` plus an editable dashboard is the pragmatic middle ground ‚Äî the seed is convenience, not correctness. The AI tool that consumes the path should fail gracefully ("path not readable: <path>") so the model surfaces the issue to the user.
**Apply when**: Seeding any developer-local resource (file paths, MCP server commands, local URLs) into a migration. Don't put hard-coded local paths into seeds for multi-tenant or production deployments ‚Äî only for single-user dev tools where the user can fix the seed via the UI.

## [2026-05-02] ‚Äî Return `{ handled, type }` (not just `boolean`) from interceptors so the caller can vary follow-up side effects
**Rule**: When an "intercept-or-fall-through" dispatcher (slash command router, middleware shim, command parser) sits in front of a pipeline that already fires its own side effects (reactions, telemetry, status updates), return a discriminated result like `{ handled: true, type: "ai_prompt" }` rather than a bare `true`. The caller uses `type` to decide whether to fire its own outer side effect or let the inner branch's side effect stand alone. Never return only `boolean` if any branch reuses the existing pipeline.
**Why**: For slash commands, the `ai_prompt` branch calls `generateAndReply`, which already fires `reaction_done` on success. If the router *also* fires `reaction_done` after seeing `handled: true`, the user gets two reactions on the same message ‚Äî a visible bug. The discriminated return lets the router skip the outer reaction precisely for `ai_prompt` and fire it for `shell` / `message` (which use only `sendReply` and don't double-fire).
**Apply when**: Adding any pre-pipeline interceptor where some branches go through the existing pipeline and some don't. The signature `{ handled: boolean; type?: Variant }` is the minimum needed; richer payloads are fine if downstream needs them. Document which branches are "pass-through to inner side effects" vs "self-contained" in the dispatcher's doc comment.

## [2026-05-02] ‚Äî Optional `opts` param with `??` fallback is the safe way to add per-call overrides without breaking callers
**Rule**: When you need to add a per-call override to an existing function (e.g. `systemInstructionOverride` on `generateAndReply`), append an optional `opts?: { ... }` parameter at the end and use `opts?.field ?? defaultExpression` at the use site. Do NOT change the existing parameter list, do NOT make `opts` required, and do NOT mutate any module-level state to thread the override through. Existing call sites continue to work with `opts` defaulting to `undefined`.
**Why**: Mutable module-level overrides (e.g. setting a `currentSystemPrompt` global before calling) leak across concurrent calls and create surprising action-at-a-distance. A required `opts` argument forces every existing caller to be updated in the same diff, blowing up the blast radius. The `?? default` pattern keeps the override scoped to one call and adds zero behavioral change for callers that don't pass it.
**Apply when**: Extending any existing function with a new "optional behavior tweak" used by exactly one new caller. The pattern is also the right shape for dependency injection in tests ‚Äî pass `opts.clock`, `opts.now`, `opts.logger` for overrides without restructuring the signature.

## [2026-05-02] ‚Äî Soft-rejecting parse failures (not just lookup failures) lets shorthand syntax coexist with real content
**Rule**: When a feature reserves a syntax prefix (e.g. `/` for slash commands, `@` for mentions, `:` for emoji), the dispatcher must distinguish "matches the prefix but fails to parse as our syntax" from "parses but no row found" ‚Äî and BOTH should fall through to the default handler, not error. For slash commands, `/path/to/file` starts with `/` but does not match `^([a-z0-9_-]+)(?:\s+([\s\S]*))?$/i`; the dispatcher returns `{ handled: false }` so the AI receives `/path/to/file` as ordinary user text rather than getting a 400 or a "command not found" reply.
**Why**: Users casually type things like `/Users/foo/...`, `/etc/hosts`, or `/r/subreddit` in chat. Hard-rejecting any `/`-prefixed text as a command attempt would block normal conversation. The two-stage filter (regex parse ‚Üí DB lookup ‚Üí both fall through on miss) means the feature only activates on syntactically clean command names that also resolve to a configured row.
**Apply when**: Designing any prefix-triggered command/mention/macro system where the prefix character also occurs in normal content (paths, URLs, user handles). Treat the regex parse as a soft filter; never throw, never reply with an error ‚Äî only intercept on a positive match all the way through.

## [2026-05-02] ‚Äî `child_process.spawn(shell, ["-c", cmd])` with explicit timeout + SIGKILL is the correct shell-exec primitive
**Rule**: For server-side "run this user-authored shell command and capture output" features, use `spawn("zsh", ["-c", cmd])` (or `/bin/sh`) with a manual `setTimeout` + `child.kill("SIGKILL")` for hard timeouts, accumulate stdout/stderr in string buffers via `data` events, and resolve a `{ code, stdout, stderr, timedOut }` shape from the `close` event. Do NOT use `exec` / `execSync` (no streaming, no clean timeout) and do NOT use `execFile` (no shell expansion ‚Äî defeats the point for user-authored commands). Always handle the `error` event in addition to `close` to catch spawn failures (e.g. zsh missing on the host).
**Why**: `exec` buffers everything in memory with a default 1MB cap and silently truncates beyond that; its `timeout` option sends SIGTERM, which a stuck process can ignore. `spawn` + `setTimeout` + SIGKILL guarantees the process dies on the deadline. The `code` may be `null` (process killed without exit) ‚Äî coalesce to a sentinel like `-1` so the caller can detect it. Truncate the stdout/stderr at the *output* boundary (e.g. 3500 chars for a Telegram reply), not the buffer boundary, so you keep the tail of the output for short-but-useful commands and only clip the noisy ones.
**Apply when**: Implementing any "run a shell command from a config row" feature (slash commands, custom hooks, webhook actions). The `{ timedOut: true }` branch is the easy-to-forget one ‚Äî make sure the reply explicitly says "Command timed out after Ns" rather than reporting an empty stdout as success. For untrusted input contexts, this whole pattern is unsafe ‚Äî that's a different problem (sandboxing/allowlisting), not a different exec API.

## [2026-05-02] ‚Äî `String.replaceAll(literal, value)` is fine for trusted templates; resist over-engineering to `{{handlebars}}`
**Rule**: For a single-token substitution like `{args}` in a user-authored template, `template.replaceAll("{args}", value ?? "")` is the right primitive. Don't reach for Handlebars, Mustache, or a hand-rolled tokenizer unless the feature needs conditionals, loops, or escaping. Document the limitation ("the literal string `{args}` is unsupported in output") in the dashboard help text and move on.
**Why**: A real template engine adds a dependency, a security surface (HTML escaping defaults vary), and a learning curve for the user ‚Äî all to support features no one asked for. `replaceAll` is one line, has no escaping behavior to misconfigure, and the "you can't emit a literal `{args}`" trade-off is acceptable for a single-user tool. If the feature later needs `{user}` and `{date}` too, a 5-line `Object.entries` loop still beats pulling in Handlebars.
**Apply when**: Adding any single-or-few-token substitution in a config string the user authors. Default to `replaceAll` per token; only escalate to a real engine when you need conditionals or escaping. Same logic for env-var-style `${VAR}` substitution in non-shell contexts.

## [2026-05-02] ‚Äî `as any` cast for SDK fields not in published types ‚Äî anchor to a comment naming the SDK version
**Rule**: When a Google/OpenAI SDK accepts a field at the REST boundary that is not yet in the TypeScript types (e.g. `generationConfig.responseModalities` in `@google/generative-ai` v0.21 for the image-preview model), the right escape hatch is a localized `as any` cast on just the offending object ‚Äî `generationConfig: { responseModalities: ["TEXT", "IMAGE"] } as any`. Always co-locate a one-line comment naming the SDK version and stating that the SDK forwards the field to the REST body unchanged. Do NOT widen the cast to the whole call, do NOT `// @ts-expect-error` (it'll flip the wrong way on SDK upgrades), and do NOT fork or re-declare the SDK's types.
**Why**: Vendor SDKs lag the REST surface by weeks, especially for preview features. A bare `as any` with no comment becomes a mystery in three months when an SDK upgrade tightens the type and someone wonders why we had to dodge it. The comment is the documentation. A `@ts-expect-error` is worse: when the SDK adds the field, the directive becomes the new error and breaks the build, but a plain `as any` keeps working silently ‚Äî which is what you want for a forward-compatible cast.
**Apply when**: Any time you're calling a vendor SDK with a preview/beta field that is documented in the REST API but missing from the typed surface. Cast at the smallest possible scope (the single object literal, not the function call), comment with the SDK version, and ideally gate the field behind a config flag so a future operator can flip it without code changes if the SDK rejects it.

## [2026-05-02] ‚Äî Gate fast-changing vendor model names behind an env var with a sane default
**Rule**: For vendor model identifiers that are known to drift (Gemini image-preview model has gone through `gemini-2.0-flash-exp-image-generation` ‚Üí `gemini-2.0-flash-preview-image-generation` ‚Üí `gemini-2.5-flash-image-preview` in months), expose the model name as an env-var with a default in the config schema (`GEMINI_IMAGE_MODEL: z.string().default("gemini-2.0-flash-preview-image-generation")`) and reference `config.GEMINI_IMAGE_MODEL` at the call site. Do NOT hard-code the string in the tool file.
**Why**: When the model is renamed or deprecated, the user can flip the env var and restart ‚Äî no code change, no redeploy of a new build. Hard-coding forces a code change on every drift event. The default keeps the zero-config experience intact for the common case. Same pattern applies to embedding models, vision models, and any preview-tier name that the vendor hasn't committed to long-term.
**Apply when**: Adding any new vendor model integration where the model id is in a preview/experimental tier, or where the vendor has historically renamed models. Always pair the env var with a default; document the rename history in a comment near the schema entry if it's especially churny.

## [2026-05-02] ‚Äî Temp-file write-then-send pattern: `try/finally` with `unlink().catch(() => {})`
**Rule**: For a tool that materializes binary data (image bytes from a model, downloaded blob) just long enough to hand off to a transport that needs a path (Telegram `sendFile`, ffmpeg, image-resize CLIs), the shape is: write to `os.tmpdir()` with a randomized filename (`Date.now()`-`randomBytes(4).toString("hex")`-`.ext`), wrap the transport call in `try { await transport(tmpPath); return success; } finally { await fs.unlink(tmpPath).catch(() => {}); }`. The `.catch(() => {})` on unlink is intentional ‚Äî failures to clean up should never mask the real error from the transport.
**Why**: A naive `await fs.unlink(tmpPath)` after the transport will throw if the file is already gone (e.g. on EBADF, or if a previous cleanup ran), and an unhandled throw from `unlink` would replace the meaningful error from `sendFile` with a confusing "ENOENT on cleanup". The randomized filename avoids collisions if two concurrent tool calls race. `os.tmpdir()` is OS-managed so leaked files (process killed between write and unlink) get reaped on reboot ‚Äî don't over-engineer a custom GC for a single-user dev tool.
**Apply when**: Any tool/handler that produces a blob and feeds it to a path-based downstream API. The `try/finally + swallow on unlink` shape is the minimum-surprise pattern; the cost of a leaked tmp file on a process crash is acceptable, the cost of a confusing stack trace on every cleanup hiccup is not.

## [2026-05-02] ‚Äî Multimodal SDK responses: trust `mimeType` for extension, default safely on unknown
**Rule**: When extracting binary parts from a multimodal LLM response (`response.candidates[0].content.parts[].inlineData`), pick the file extension from `inlineData.mimeType` (`image/png` ‚Üí `.png`, `image/jpeg` ‚Üí `.jpg`, `image/webp` ‚Üí `.webp`) and default to `.png` for any unknown or missing mime. Don't sniff magic bytes; don't enforce a single output format on the model. Telegram (and most consumers) renders all three correctly regardless of extension, so the default is forgiving.
**Why**: The model picks the format based on its own heuristics ‚Äî forcing `.png` always works, but uses larger files for photographic content where JPEG would be 1/10th the size. Trusting `mimeType` keeps the file small and the content-type accurate. Defaulting to `.png` on unknown is safe because PNG is the universal "lossless image" format and Telegram doesn't actually look at the extension ‚Äî but downstream tools (S3, browsers via Content-Disposition, image processors) do.
**Apply when**: Any code extracting `inlineData` from a Gemini/OpenAI multimodal response, or any handler that writes vendor-supplied bytes to disk and needs an extension. Don't add a magic-byte sniffer; the mime field is reliable when present, and the safe default covers the rest.

## [2026-05-02] ‚Äî Prefer extending the existing `ai_prompt` slash branch with a system-prompt override over adding a new dispatcher branch
**Rule**: When adding a new slash command that should trigger AI behavior with a tweaked instruction (e.g. `/draw` should force a `generate_image` tool call), seed the row with `type='ai_prompt'` and put the directive in the `action` column as a system-prompt override. Do NOT add a new dispatcher branch (`type='draw'`, `type='generate'`) ‚Äî the `ai_prompt` branch already passes `action` as `systemInstructionOverride` to `generateAndReply`, which is exactly the right hook. Each new slash command becomes one migration row, zero code changes to the dispatcher.
**Why**: Adding a new `type` value would require: (1) a CHECK constraint migration to widen the allowed types, (2) a new dispatcher branch, (3) new tests, (4) a new entry in the `{ handled, type }` discriminated return. Versus: one INSERT row that uses the existing `ai_prompt` plumbing. The system-prompt override is the universal "make the AI do this specific thing" lever ‚Äî resist the urge to specialize.
**Apply when**: Any new slash command whose behavior is "have the AI do X with this instruction." Seed it as `ai_prompt` with the action being the override prompt. Only add a new `type` if the behavior genuinely cannot be expressed as "AI with this system override" ‚Äî e.g. a pure shell-out, a webhook trigger, a synchronous DB query that must short-circuit the AI loop entirely.

## [2026-05-04] ‚Äî Public webhook routing via auth-hook scope, not per-route allowlist
**Rule**: When adding a public (unauthenticated) HTTP endpoint to a Fastify app whose auth is enforced by a global `onRequest` hook, register the route OUTSIDE the hook's URL prefix scope (e.g. mount at `/webhook/:id` when the hook only fires on `/api/*`). Do NOT widen the hook to add a `PUBLIC_PATHS` allowlist; do NOT decorate the route with a "skip auth" flag. The path itself is the auth boundary.
**Why**: A `PUBLIC_PATHS` allowlist is an action-at-a-distance config that someone editing the route forgets to update. The path-prefix split is enforced by URL routing ‚Äî impossible to mis-configure on the route side. The trade-off is the public route can't share the `/api/*` namespace, but for webhooks called by external services (Telegram, Stripe, GitHub) that's the right separation anyway. ALSO: add a top-of-file comment in the public route noting the dependency on the auth hook's scope, so a future operator widening the hook to `/*` gets a breadcrumb.
**Apply when**: Any new endpoint a third party calls without your auth cookie/header (webhooks, OAuth callbacks, health checks for external monitors). Mount outside the protected prefix; comment the dependency on the hook's scope at both ends (the route and the hook).

## [2026-05-04] ‚Äî `timingSafeEqual` needs length-mismatch handling ‚Äî wrap, don't try/catch
**Rule**: `crypto.timingSafeEqual(a, b)` THROWS `RangeError: Input buffers must have the same byte length` when buffers differ in size ‚Äî it does not return `false`. Wrap calls in a helper like `safeEqualHeader(actual, expected)` that converts both inputs to Buffers, returns `false` on length mismatch BEFORE calling `timingSafeEqual`, and returns `timingSafeEqual(a, b)` otherwise. Do NOT try/catch the throw ‚Äî pre-checking the length is faster, more readable, and avoids the perception that any other crypto error is being swallowed.
**Why**: A naive `timingSafeEqual(Buffer.from(headerVal), Buffer.from(expected))` crashes the request handler with an uncaught throw the moment an attacker (or a misconfigured client) sends a header of the wrong length ‚Äî turning what should be a 401 into a 500. The pre-length-check is the cleanest fix; try/catch works but loses the chance to detect genuine crypto failures. The helper also normalizes the type ‚Äî header values can arrive as `string | string[] | undefined` from Fastify.
**Apply when**: Any time you compare a user-supplied secret/token header against a stored value. The pattern is identical for HMAC signatures, webhook secrets, API keys passed in headers. Default to a small `safeEqualHeader` helper in a shared util file rather than inlining at each call site.

## [2026-05-04] ‚Äî Webhook handlers: always 200 except for the ONE error code the sender uses to back off
**Rule**: For an HTTP webhook handler, return `200` for nearly every outcome ‚Äî bad request body, unknown bot/resource id, disabled row, parse failure, internal exception during async work ‚Äî and reserve non-200 codes for the one signal the sender actually acts on. For Telegram, that signal is `401` on `secret_token` mismatch (Telegram stops retrying); every other condition uses `200` with silent drop. Wrap the AI/business logic in `.catch(err => logger.error(...))` AFTER the 200 has been returned so a Gemini failure or DB hiccup does not leak into the HTTP response.
**Why**: Webhook senders (Telegram, Stripe, GitHub) retry on non-200 with exponential backoff. Returning `400` for "unknown bot id" makes Telegram retry the same broken update for hours, flooding logs and possibly racing with cleanup. Returning `500` on a transient downstream error (Gemini timeout) makes Telegram resend the same user message ‚Äî duplicating the AI response and any side effects (sent messages, DB writes). The "always 200" rule sounds wrong from a REST hygiene standpoint, but webhooks have inverted semantics: you are the *consumer*, the sender is the *server*, and your job is to acknowledge receipt fast.
**Apply when**: Any inbound webhook handler. Map error responses to: 401 = "stop retrying, you're misconfigured" (the signal); 200 = literally everything else. Document the chosen non-200 codes inline so a future maintainer doesn't add a 500 path on instinct.

## [2026-05-04] ‚Äî Fire-and-forget ACK before async work ‚Äî `void` + `.catch`, never `await`
**Rule**: When an inbound message has a UI-acknowledgment requirement that the upstream service uses to clear a loading state (Telegram `callback_query.id` ‚Üí `answerCallbackQuery`, Slack `response_url`, Discord interaction tokens), call the ACK with `void someApi(...).catch(err => logger.warn(...))` ‚Äî explicit `void` prefix, `.catch` tail to swallow errors ‚Äî and IMMEDIATELY proceed to the actual work. Do NOT `await` the ACK before the AI/handler logic runs.
**Why**: `await ack(); await ai();` serializes a network round-trip in front of the slow path, leaving the user staring at a spinner for hundreds of extra milliseconds. The ACK has no return value the handler needs and its failures are non-fatal (the spinner just stays a few extra seconds). The `void` prefix is load-bearing ‚Äî without it, lint rules like `no-floating-promises` will flag the line and the next person "fixes" it by adding `await`, regressing the latency. The `.catch` is not optional ‚Äî an unhandled promise rejection in a `void`-discarded promise will crash Node 15+.
**Apply when**: Any handler whose upstream protocol has a "clear the spinner" or "I got it" signal separate from the actual response. Always: `void ack().catch(log); doRealWork();`. If TS or lint complains, the lint config is the bug ‚Äî add a comment, not an `await`.

## [2026-05-04] ‚Äî Reuse the route/import path when replacing a feature; let the seam be the file contents
**Rule**: When deleting and replacing a feature that owns a sidebar entry, frontend route, page filename, and API path (e.g. swapping the persona-bot CRUD for a Telegram-bot CRUD that occupies the same "Bots" name), KEEP the existing identifiers ‚Äî `/bots` route, `pages/Bots.tsx` filename, `/api/bots` REST path, `qk.bots` query key, `{ to: "/bots", label: "Bots" }` sidebar entry. Replace ONLY the file contents. The router (`App.tsx`), the sidebar (`Sidebar.tsx`), and the query-key barrel become zero-edit files.
**Why**: Editing the router and sidebar to point at a renamed file/route doubles the diff for no behavioral gain and makes the rollback bigger. Reusing the names also lets the dashboard's bookmark/URL stability survive the swap ‚Äî a user who had `/bots` in browser history still lands on the same page, just with new contents. The trade-off is git blame on the page file shows a sudden total rewrite (good ‚Äî that IS the change), and the internal server filenames keep a longer prefix (`telegramBots.ts`) to disambiguate from the deleted persona-bot file in version-control history.
**Apply when**: Any feature swap where the new feature occupies the same conceptual slot as the old one. Keep the user-facing names (URL, sidebar label, query key) identical; let the new behavior live entirely inside the file contents. Add longer disambiguating prefixes only on internal server filenames where git-blame archaeology matters.

## [2026-05-04] ‚Äî Atomic delete-then-recreate phases: do NOT typecheck or commit between
**Rule**: When a multi-phase plan deletes a file in an early phase and recreates it (with new contents at the same path) in a later phase ‚Äî e.g. Phase 0 deletes `pages/Bots.tsx` and Phase 7 writes a new `pages/Bots.tsx` ‚Äî treat the delete-and-recreate phases as ONE atomic change set. Do NOT run `tsc --noEmit` between them, do NOT commit between them, do NOT push between them. The `App.tsx` import will be dangling for the duration of the gap. If parallelism is desired, do the recreate step IMMEDIATELY after the delete step so the import is never broken on disk.
**Why**: A typecheck run between Phase 0 and Phase 7 will fail loudly with "Cannot find module './pages/Bots'", which is technically correct but uselessly distracting ‚Äî the plan acknowledged the gap and the recreate is queued. A commit between the two phases creates a build-broken commit in history, which (a) breaks `git bisect` going forward and (b) blocks anyone who checks out that SHA from running the dev server. The atomic-set rule sidesteps both. Document the atomicity in the plan ("treat Phases 0‚Äì7 as one change set; do not commit between them") so a future operator doesn't split them across PRs by accident.
**Apply when**: Any plan where a file is deleted in one step and a same-pathed replacement is written in another. The atomicity declaration belongs in the plan's Risks section AND in a comment on the deletion step itself, not just in the executor's head.

## [2026-05-04] ‚Äî Webhook + CRUD route registration ordering vs catch-all plugins
**Rule**: When Fastify has `setNotFoundHandler` or a static-file plugin that catches `/*`, register every explicit route ‚Äî including the public webhook ‚Äî BEFORE those catch-all registrations. The webhook route at `/webhook/:botId` is no exception, even though it lives outside `/api/*`: the static plugin's catch-all will swallow it if registered first. Order in `api/index.ts` should be: explicit API routes ‚Üí explicit public routes (webhooks) ‚Üí static plugin ‚Üí `setNotFoundHandler`.
**Why**: Fastify resolves routes in registration order for catch-all conflicts. A `staticPlugin` mounted at `/` with `wildcard: true` will eat any request that doesn't match an already-registered route. If you register webhook routes AFTER the static plugin, the webhook POST gets a 404 (or worse, a "not found" HTML page returned to Telegram, which retries forever because non-200). The fix is registration order, not a `before:` hook or a `priority` config ‚Äî Fastify doesn't have those. Put a comment near both the static plugin and the webhook registration noting the ordering dependency.
**Apply when**: Any new public route in a Fastify app that uses static-file serving or a global not-found handler. Always register explicit routes first; never assume route specificity wins over registration order with catch-all plugins.

## [2026-05-04] ‚Äî `BIGINT` vs JS Number: Telegram chat ids fit in 2^53, but persist as text-safe form anyway
**Rule**: Telegram chat ids are `int64` on the wire and exceed `2^31`, but currently fit comfortably in `2^53` (the JS-safe integer range). Persist them as `BIGINT` in Postgres and read them back as `string` (most pg drivers do this by default to preserve precision). When passing them BACK into a third-party API (GramJS `EntityLike`, Bot API `chat_id` JSON field), `Number(idStr)` is safe today ‚Äî but prefer `String(idStr)` if the API accepts string form, since the cost of a future Telegram id-space expansion is "all your existing code is silently wrong" with no error.
**Why**: The Bot API actually accepts `chat_id` as either number OR string in JSON, and GramJS accepts string for `EntityLike`. Defaulting to string-form propagation through the stack is the forward-compatible choice for ~zero cost. The only place you MUST convert to number is when feeding the id into a JS `Map` key alongside numeric keys (don't mix), or comparing to a value the SDK has already coerced to number. The lesson is to prefer the string form whenever the API allows it ‚Äî a one-character change today saves a debug session in 2030.
**Apply when**: Any code path that pulls a Telegram chat id from the DB and hands it to an external API. Default to string. Only call `Number(id)` at the explicit API boundary where the SDK demands a number, with a comment naming the SDK and version. Never use `BigInt(id)` for GramJS ‚Äî the type rejection is documented in 2026-04-28's lesson; for Bot API the string-or-number flexibility makes BigInt unnecessary.

## [2026-05-04] ‚Äî Webhook secret rotation: explicit delete + recreate, not in-place mutation
**Rule**: For per-row webhook secrets (or any rotateable credential stored alongside other config), document that rotation requires DELETING the row and recreating it ‚Äî do NOT add a "rotate secret" PATCH endpoint or auto-regenerate on update. The secret is a one-time set at create-time; the dashboard's edit form does not expose it as editable.
**Why**: A "rotate" endpoint introduces a window where the new secret is registered with the third party (Telegram `setWebhook`) but the old secret is still in flight on a pending request ‚Äî opening a brief auth-failure gap. More importantly, it's surface area no one asked for: rotation is rare (compromise, lost backup), and delete-and-recreate is a 30-second operator action that handles the registration cleanly. The simpler model also keeps the row's identity tied to its secret ‚Äî useful for audit logs ("secret X was used during this incident" maps to a single row, not "row Y at time Z had secret X1, then X2").
**Apply when**: Any per-row credential where rotation frequency is "almost never" and the operator is a human, not an automated service. If rotation needs to be automated (e.g. compliance-mandated quarterly rotation), the model is different: PATCH endpoint, dual-secret window, callbacks accept both for a grace period. For the single-user dev-tool case, "delete + recreate" wins on simplicity.

## [2026-05-07] ‚Äî `botAuthToken` MTProto client beats webhook for single-user bot integration
**Rule**: When adding a Telegram bot to an existing GramJS user-account app, run a SECOND `TelegramClient` in the same process authenticated via `client.start({ botAuthToken: () => Promise.resolve(token), onError })` instead of registering a webhook with `setWebhook`. The bot client speaks MTProto over the existing GramJS connection plumbing, listens via `NewMessage` and `CallbackQuery` event handlers, and requires zero public URL, zero TLS termination, zero webhook secret. Persist its session to a SEPARATE file (`data/bot-session.txt`) ‚Äî never share with the user-account `data/session.txt`.
**Why**: The webhook path forces a `PUBLIC_URL`, a path-prefix carve-out for unauthenticated traffic, a `secret_token` rotation strategy, `setWebhook` lifecycle calls, and the "always 200 except 401" handler discipline (lessons-2026-05-04). All of that disappears when the bot client uses the same GramJS primitives the user-account already does ‚Äî one extra module-level `TelegramClient`, one extra session file, one extra event handler pair. The trade-off is the bot is only online while the server process is alive (no message buffering during downtime), which for a single-user dev tool is a feature not a bug.
**Apply when**: Adding a Telegram bot to any app that already runs GramJS for a user account. Default to the second-client MTProto pattern; only escalate to webhook if you need (a) message delivery while the server is down, (b) horizontal scaling across multiple processes, or (c) you don't already have GramJS in the dependency tree. For greenfield bot-only apps without a user-account client, webhook is still the right default.

## [2026-05-07] ‚Äî Two GramJS clients in one process: isolate session files, share nothing else
**Rule**: When running both a user-account `TelegramClient` and a bot `TelegramClient` in the same Node process, give each its own `StringSession` file (`data/session.txt` vs `data/bot-session.txt`), its own module-level singleton (`_client` in `client.ts`, `_botClient` in `botClient.ts`), and its own start/stop lifecycle. Do NOT try to share a `StringSession` instance, do NOT cross-import the singletons, and do NOT register handlers on one client that call into the other's state. Both clients can share the same `apiId`/`apiHash` from config ‚Äî those are app credentials, not session state.
**Why**: GramJS `StringSession` holds the auth-key for a specific account. Sharing it between clients corrupts the session file (last-writer-wins on `client.session.save()`) and crashes one or both clients with auth errors. Separate singletons also let the bot client be stopped/restarted independently when the user toggles the dashboard `enabled` flag, without affecting the user-account client's connection. The shared `apiId`/`apiHash` is fine because Telegram allows one app's credentials to authenticate multiple sessions concurrently ‚Äî that's the entire point of session strings.
**Apply when**: Any multi-account GramJS setup in one process ‚Äî bot + user, multiple users, user + multiple bots. The pattern generalizes: one module per client, one session file per client, one lifecycle pair (start/stop) per client. If the count exceeds ~3, consider a `Map<accountId, TelegramClient>` registry instead of N singleton modules ‚Äî but for the bot-plus-user case, two singletons keep the import graph trivially readable.

## [2026-05-07] ‚Äî `chat_type='bot'` + composite unique `(tg_chat_id, chat_type)` for dual-channel chats
**Rule**: When the same Telegram user can talk to BOTH the user-account client AND a bot client in the same app, the `chats` table needs to allow two rows for the same `tg_chat_id` ‚Äî one for each channel. Drop the single-column `UNIQUE(tg_chat_id)` constraint, add a composite `CREATE UNIQUE INDEX chats_tg_chat_id_chat_type_key ON chats(tg_chat_id, chat_type)`, and widen the CHECK constraint to include `'bot'`. The `upsertChat` query becomes `ON CONFLICT (tg_chat_id, chat_type) DO UPDATE`.
**Why**: A user with Telegram id `123` who DMs your user-account is a "private" chat; the same user `123` who also messages your bot is a separate logical conversation ‚Äî different message history, different unread count, possibly different display name (their bot-facing name). Forcing them into one row collapses the two histories together and makes the dashboard show a confusing mix. The composite unique index is the minimum schema change that preserves both rows; the CHECK widening is mandatory because the old constraint silently rejects the bot-channel insert at runtime with no useful error.
**Apply when**: Any chat-storage schema where one external user identity can flow through multiple ingress channels (bot + user, multiple bots, bot + email-bridge). Default to `(external_id, channel_type)` as the composite uniqueness key; never rely on `external_id` alone if multiple channels exist or might be added.

## [2026-05-07] ‚Äî Singleton config tables via `CREATE UNIQUE INDEX ON ((TRUE))`
**Rule**: For a config table that should hold AT MOST one row (e.g. `telegram_bot_config`, app-wide settings, single-tenant flags), enforce the singleton at the schema level with `CREATE UNIQUE INDEX <name>_singleton ON <table>((TRUE))`. The `(TRUE)` expression evaluates to the same value for every row, so the unique index allows zero or one row but rejects the second insert with a `23505` violation. Pair with a repo `setX(patch)` function that does INSERT-or-UPDATE based on whether `getX()` returned a row.
**Why**: The alternative ‚Äî a CHECK constraint like `id = 1` plus enforcing `id=1` in every INSERT ‚Äî leaks the magic constant through every consumer and breaks the moment someone forgets to set it. The `((TRUE))` trick is a documented Postgres idiom that lives entirely in the schema, requires zero discipline from consumers, and produces a normal unique-violation error path the route layer already handles (lessons-2026-04-30). It also leaves the row's `id` as a normal UUID, so audit logs and FKs work without special-casing.
**Apply when**: Any table whose semantic cardinality is "exactly zero or one" ‚Äî global config, singleton secrets, per-app feature toggles. Don't reach for it when cardinality is "exactly one" (that's a CHECK constraint plus a seed migration); the index allows zero rows on purpose so the app can boot before config is set.

## [2026-05-07] ‚Äî `ReplyAdapter` interface lets one AI loop serve multiple egress channels without forking
**Rule**: When the same `generateAndReply(chat, text, opts)` function needs to send replies through different transports (user-account `client.sendMessage` vs bot `client.sendMessage` vs future webhook), introduce a small `ReplyAdapter { sendText, persistOutbound }` interface and accept it via `opts.replyAdapter`. The default branch (no adapter) keeps the existing user-account behavior including reactions and prefix prepending; the adapter branch sends raw text and persists via the adapter's hooks. Do NOT fork the AI loop; do NOT add an `if (isBot)` branch at every send site ‚Äî funnel both through the adapter.
**Why**: Forking `generateAndReply` into `generateAndReplyForBot` duplicates the tool-loop, the system-prompt assembly, the error handling, and the message-persist logic ‚Äî every future change has to be made twice. The adapter pattern keeps one source of truth for the AI loop and isolates the transport differences (which buttons are supported, which reactions to skip, which prefix to prepend) into the adapter implementation. The `opts?.replyAdapter ?? defaultBehavior` shape (lessons-2026-05-02) means existing call sites are zero-edit.
**Apply when**: Any AI/orchestration loop that currently has one egress transport baked in and needs to gain a second. The `{ sendText, persistOutbound }` minimal interface is enough for most chat transports; add `sendImage`/`sendFile`/`react` only as the second adapter actually needs them. Default: empty interface, grow only on demand.

## [2026-05-07] ‚Äî CallbackQuery ‚Üí synthetic `[Button: <data>]` user message re-enters AI loop
**Rule**: When a Telegram bot inline button is pressed, the `CallbackQuery` event handler should: (1) fire-and-forget `void event.answer().catch(log)` to clear the spinner (lessons-2026-05-04), (2) `insertMessage` with text `[Button: <callback_data>]` and `direction='in'` as if the user had typed it, (3) call `generateAndReply` with the same opts as a regular incoming message. Do NOT add a separate code path that handles button presses with bespoke logic ‚Äî re-entering the AI loop with a synthetic message keeps the conversation history complete and lets the model react to the button as a normal turn.
**Why**: A separate "button handler" branch fragments the conversation history (the AI sees the button-driven response but no record of WHICH button was pressed), forces every tool to know about both real-message and synthetic-button ingress, and makes the chat dashboard show a mysterious bot reply with no preceding user turn. The synthetic `[Button: X]` message is the same pattern as `[Image: ...]` and `[Poll: ...]` already established in the codebase (lessons-2026-04-29) ‚Äî uniform treatment of non-text turns as bracketed sentinel text. The model handles them naturally because the system prompt explains the convention.
**Apply when**: Any UI element whose interaction needs to feed back into the AI loop (inline buttons, reactions, slash menu picks, voice transcripts). Convert to a synthetic `[Type: payload]` user message; let the AI loop handle it like any other turn. The convention scales: every new ingress type adds a new bracket prefix and zero new code paths.

## [2026-05-07] ‚Äî Resolve global tool dependencies at INVOCATION time, not registration time
**Rule**: When a tool definition needs access to a singleton client/connection that may be reconnected during the bot's lifetime (`getBotClient()`, `getDbConnection()`, `getCurrentUser()`), call the resolver INSIDE the tool's handler ‚Äî not at `buildTools(...)` time. Captured-at-registration references go stale the moment the singleton is replaced; lazy resolution always sees the current value. Cost: one extra function call per tool invocation. Benefit: tool keeps working across `stopBotClient() ‚Üí startBotClient()` cycles without rebuilding the tool registry.
**Why**: When the dashboard toggles the bot off and back on, `botClient.ts` calls `client.disconnect()` and replaces `_botClient` with a new instance. Tools that captured the old `client` reference at `buildTools` time will silently send messages on a disconnected client (or throw "client not connected"). Re-resolving via `getBotClient()` at invocation time picks up the new client. Same logic applies to any singleton that has a "rebuild" path ‚Äî config-driven DB pools, session managers, feature-flag clients.
**Apply when**: Authoring tools that depend on a globally-managed connection or service singleton. The lazy-resolve pattern is the default; the early-resolve pattern is fine ONLY if the singleton is provably immutable for the process lifetime (rare). Document the lazy resolution in the tool's source with a one-line comment so a future "optimization" doesn't hoist the lookup out of the handler.

## [2026-05-07] ‚Äî Boot-time bot start failures must be non-fatal ‚Äî log and proceed
**Rule**: When the server boots and reads `telegram_bot_config` to auto-start the bot client, wrap `startBotClient(token)` in `.catch(err => logger.error(...))`. Do NOT let a bot-auth failure (revoked token, network blip during MTProto handshake) crash the process. The dashboard MUST stay reachable so the operator can fix or clear the config; a crash loop turns a recoverable misconfiguration into a "console-into-the-server" emergency.
**Why**: The bot is one feature among many in this app; the dashboard, the user-account client, the AI loop, and the slash commands all work fine without it. Crashing the whole process on bot-start failure inverts the priority ‚Äî a single peripheral feature takes down the operator's ONLY recovery surface. The error log is enough signal for the operator to notice, and the dashboard's `/bots` page lets them fix the token, toggle off, or delete the config without restarting. The same logic applies to any optional feature initialized at boot from config ‚Äî log loudly, fail soft, keep the recovery surface alive.
**Apply when**: Any boot-time initialization of an optional service (bot clients, MCP servers, scheduled-job runners, integrations) where misconfiguration is recoverable via the dashboard. Wrap the start call in `.catch(log)` and continue boot. Reserve hard crashes for the irrecoverable subset: missing required env vars, DB unreachable, migrations failed.

## [2026-05-07] ‚Äî `prefix` and `sendReaction` belong inside the user-account branch, not the shared loop
**Rule**: When extending an AI loop to support a second egress channel via `ReplyAdapter`, AUDIT every side-effect that currently runs unconditionally ‚Äî message prefixes (`prefix + reply`), upstream reactions (`sendReaction(...)`), read-receipts, typing indicators ‚Äî and gate each on `!opts?.replyAdapter`. The bot path almost always wants none of them: bots don't typically prefix every reply, can't react with the user-account's emoji set, and don't have the same "typing" semantics. Default behavior stays unchanged for existing user-account callers; bot path gets a clean send-only experience.
**Why**: Reactions sent through a bot client may use a different emoji set or fail outright (some emoji are user-only). Prefixes that make sense for the user-account ("[bot]: " or similar disambiguators) are noise on a dedicated bot channel where every message is from the bot by definition. Failing to gate these creates subtle UX bugs (double-prefixed bot replies, reaction errors in logs) that are easy to ship and hard to notice in QA. The audit is a one-time scan of the responder; the gates add ~3 lines.
**Apply when**: Any time you split a single-channel AI loop into multi-channel via an adapter pattern. Walk through every side effect in the loop, ask "does this make sense for channel B?", and gate accordingly. Document the gates inline with a short comment ("bot channel skips reactions; user-account-only feature") so a future refactor doesn't accidentally re-enable them.

## [2026-05-08] ‚Äî Default-block via DB column DEFAULT flip, not via app-layer INSERT change
**Rule**: When the requirement is "all NEW rows of table T should default to state S, but existing rows are untouched", express it as a single `ALTER TABLE T ALTER COLUMN c SET DEFAULT S` migration ‚Äî NOT as an application-code change to the INSERT path. This works ONLY if the existing `upsert*` helper deliberately omits column `c` from its INSERT column list (so the DB default applies on first INSERT) and from the `ON CONFLICT DO UPDATE` SET list (so existing rows preserve their value). Verify both halves before shipping the migration.
**Why**: The schema-only change is atomic, idempotent, retroactively safe (no `UPDATE` needed), and requires zero application logic. The pre-condition ‚Äî `upsertChat` not touching `is_blocked` in either INSERT columns or the conflict-update ‚Äî is load-bearing: if a future refactor adds `is_blocked: false` to the INSERT defaults "for clarity", the DB default is silently overridden and new chats arrive UNblocked. Pin this coupling with a top-of-migration comment naming the upsert function and the columns it deliberately omits, so a future maintainer reading the migration sees the dependency. Same pattern works for any "default state for new rows" feature (default-archive, default-pending, default-private).
**Apply when**: Any "new rows default to X" requirement where existing rows must NOT be retroactively updated. Always check the upsert/insert path doesn't override the default; document the coupling in the migration comment AND in the upsert function with a one-line "do not add column X here ‚Äî DB default applies" reminder.

## [2026-05-08] ‚Äî Silent-drop on auth-style gate to prevent enumeration; reply only on success
**Rule**: When a feature gates access on a user-supplied secret/name/token (e.g. `/unblock <ai_username>`, password-protected channel, invite-code redemption), the wrong-value branch must silently drop ‚Äî no reply, no error message, no "wrong code" feedback ‚Äî while the correct-value branch sends a confirmation. The inbound attempt is still persisted to the dashboard so the operator can see attempts, but the sender learns nothing about whether the gate exists or what the right value would be.
**Why**: Replying "wrong username" or "access denied" lets an attacker iterate over guesses with feedback ("the gate exists; let me try `bot`, `assistant`, `ai`..."); silent drop forces them to operate blind, dramatically raising the cost of brute force. The honest-user UX cost is real (a confused human might type `/unblock bot` and see nothing) but is mitigated by communicating the right username out-of-band (via the bot's profile bio, a `/start` reply, or operator instructions). The asymmetric "loud on success, silent on failure" is the standard pattern for auth gates and should be the default unless the threat model explicitly tolerates enumeration.
**Apply when**: Any user-supplied-secret gate (passwords, invite codes, magic words, named slash commands). Always: success ‚Üí confirmation reply + state change; failure ‚Üí silent drop + persist the attempt for audit. Log at info-level (`"blocked, dropping unblock attempt"`) not error-level so the dashboard isn't noisy. Same pattern for HMAC verification on webhooks: 401 only when the SENDER needs the signal to back off; otherwise drop.

## [2026-05-08] ‚Äî Pre-pipeline gate ordering: anti-loop > authz-bypass > authz-deny
**Rule**: When a request handler has multiple pre-pipeline checks ‚Äî anti-self-loop guards (skip own outbound echoes), authz-bypass commands (`/unblock` to flip the gate), and the actual authz-deny (block check) ‚Äî the ordering is: (1) anti-loop FIRST so we don't process our own outbound, (2) authz-bypass parser SECOND so the unblock command can run even on a currently-blocked chat, (3) authz-deny THIRD so it only applies when the message wasn't an unblock command. Inserting the bypass after the deny means a blocked chat can NEVER unblock itself; inserting it before the anti-loop means the bot's own echo of `/unblock x` would re-trigger the bypass.
**Why**: The order is a load-bearing invariant: each check filters out a class of messages from reaching the next. A naive implementation puts the new bypass at the bottom near the AI call ("smallest blast radius") and finds the gate is unbypassable. Document the ordering with a numbered comment block at the top of the dispatcher (`// 1) skip self-echoes 2) try unblock 3) check block 4) AI`) so future additions know where to slot in. The same ordering applies to slash-command dispatchers, mute/unmute pairs, and any "command that affects whether the next step runs" feature.
**Apply when**: Adding any pre-pipeline check to a router/dispatcher that already has gates. Identify which existing checks the new one must come BEFORE (because it counters them) vs AFTER (because it depends on them passing). Anti-loop almost always stays first; authz-bypass goes between anti-loop and authz-deny; logging/metrics typically come last.

## [2026-05-08] ‚Äî Pure parser + caller-decides-side-effects keeps gate logic testable across channels
**Rule**: For a command-parser used by multiple ingress paths (user-account router AND bot event handler), the parser MUST be a pure function returning a discriminated result (`{ matched: boolean; correct: boolean }`) and the CALLER decides what to do ‚Äî emit events, send confirmations, update DB, log. Do NOT bake `setChatBlocked()` or `eventBus.emit()` into the parser. Each caller can vary its side effects (user-account uses `sendReply` with `source='manual'`; bot uses `getBotClient().sendMessage` + `insertMessage`) without forking the parser.
**Why**: A parser that does its own DB writes ties every consumer to the same side-effect order, the same event payload, the same confirmation text. The user-account and bot paths legitimately differ ‚Äî user-account has `sendReply` infrastructure for source-tagging; bot has the bot client and a different persist path. Returning the discriminated result and having both callers handle the three cases (`!matched`, `matched && correct`, `matched && !correct`) keeps one source of truth for the parsing rule and one source of truth per channel for the side effects. Also makes the parser unit-testable with no mocks.
**Apply when**: Any cross-channel command parser (slash commands, mention triggers, hashtag handlers) where multiple ingress paths need to dispatch the same syntax. Always: parser returns a result; caller acts. Mirror the discriminated-return shape from lessons-2026-05-02 (`{ handled, type }`) ‚Äî both serve the same goal of "parser informs, caller decides".

## [2026-05-08] ‚Äî Async UI prompt round-trip via DB-token + synthetic re-entry message
**Rule**: When the AI needs to ask the user a discrete question and resume on their answer (inline keyboards, polls, choice prompts), do NOT block the AI loop waiting for the response. Instead: (1) the tool issues the prompt and returns IMMEDIATELY with `{ ok, sent }`; (2) the AI's turn ends; (3) the question's lookup state lives in a DB table (`pending_choices` here) keyed by a short token embedded in the callback payload (`c:<token>:<idx>`); (4) the user's tap fires a callback handler that atomically claims the token via `UPDATE ... SET consumed_at = now() WHERE token=$1 AND consumed_at IS NULL AND expires_at > now() RETURNING ...`; (5) the handler inserts a synthetic `[Choice: <label>]` user message into the SOURCE chat (where the AI originally asked) and re-enters the AI loop via the normal `generateAndReply` path. The AI sees the answer as a fresh turn, just like any other user message.
**Why**: A "wait for user input" tool that blocks the AI's runToolLoop holds the LLM API connection open for hours, breaks streaming, and leaks resources. The token+re-entry pattern decouples the issue from the answer cleanly: the AI's process can crash, restart, scale across machines ‚Äî the token in Postgres survives. The synthetic `[Choice: ‚Ä¶]` reuses the existing `[Button: ‚Ä¶]`, `[Image: ‚Ä¶]`, `[Poll: ‚Ä¶]` bracket convention so dashboard rendering, conversation history, and AI context all stay uniform with no special cases. Source-chat-id stored in the token row matters: future versions might deliver the prompt to a different channel than the source (e.g. a dedicated approvals channel), but the AI continues the conversation where it started.
**Apply when**: Any AI-tool that needs a structured user response to drive the next turn ‚Äî choice prompts, approval gates, scheduling pickers, multi-step forms. Default to fire-and-resume via DB token + synthetic message; reserve "blocking tool" semantics for fast deterministic computations (calculator, lookup, format conversion). The token TTL (24h here) protects against stale taps; the `consumed_at` column makes the claim idempotent.

## [2026-05-08] ‚Äî Reserved callback_data prefix for system-issued buttons coexisting with AI-authored freeform buttons
**Rule**: When a callback handler must distinguish system-issued buttons (where the system owns the round-trip semantics) from AI-authored freeform buttons (where the AI invents the callback_data), reserve a short prefix for the system's format (e.g. `c:<token>:<idx>` for `ask_user_choice`) and BRANCH on the prefix BEFORE the freeform fallback. The decode is a strict regex (`^c:([A-Za-z0-9_-]{1,32}):(\d+)$`) ‚Äî on miss OR on token-lookup-fail, fall through to the freeform `[Button: <data>]` synthetic-message path. NEVER fall through after a regex match ‚Äî a stale system token tap should silently drop, not inject `c:expired:0` into the AI's history as a freeform button.
**Why**: Two concurrent button systems share the same Telegram callback channel ‚Äî there is exactly one `CallbackQuery` event type. The prefix is the only at-rest signal that lets the handler route correctly. Strict regex (not just `startsWith('c:')`) prevents an AI-authored freeform `c:approve:42` from accidentally hitting the system path; the freeform tool's description should ALSO warn the model to avoid the reserved prefix, but the regex strictness is the primary defense. The asymmetric "fall through on prefix-miss but DROP on prefix-match-with-bad-token" is load-bearing: a matched-but-stale token represents a UI element the system once owned but no longer does, and treating it as freeform would create a phantom AI turn from a user's accidental re-tap.
**Apply when**: Any time a single callback/event channel is shared between system-managed and user/AI-managed payloads. Reserve a prefix for the system; document it in the freeform tool's description; enforce strict regex matching in the dispatcher; drop silently on match-but-stale rather than falling through. Same pattern works for any shared event channel ‚Äî slash commands vs typed text, mention syntax vs URL paths, etc.

## [2026-05-08] ‚Äî "End your turn" steering needs both the tool description AND the system prompt
**Rule**: When a tool has fire-and-resume semantics (returns immediately, model must NOT continue generating in the same turn), put the "END YOUR TURN" directive in BOTH the tool's description string AND a Guidelines bullet in the system prompt. The tool description alone is insufficient ‚Äî the model sees it only when calling the tool, but the inhibitory instruction needs to also live in the persistent system context so the model plans the turn correctly from the start. Conversely, the system-prompt bullet alone is insufficient ‚Äî by the time the model is generating after the tool response, the system prompt's instruction may not be foregrounded; the tool description (echoed in tool definitions every turn) reinforces it.
**Why**: Gemini's behavior on tool-call sequencing depends on both the structured tool definition and the natural-language instructions. A tool description saying "returns immediately" without "END YOUR TURN" can be read as "this is fast, you can immediately do more" ‚Äî exactly the wrong inference. A system prompt bullet without a corresponding tool-description directive relies on the model carrying global context into a tool-specific decision, which is unreliable across model versions. The doubled instruction is cheap (two sentences) and the cost of getting it wrong is duplicate AI replies ‚Äî the buttons appear AND the model emits trailing "I asked you to pick‚Ä¶" text, confusing the user.
**Apply when**: Authoring any AI tool with fire-and-resume / async-prompt semantics. Always pair the directive in two places. If observed empirically that the model still emits trailing chatter after such a tool, the deeper fix is post-tool early-exit (drop trailing text in the responder when the last tool was a fire-and-resume tool) ‚Äî but the doubled instruction handles 90% of cases at zero code cost.

## [2026-05-08] ‚Äî Source chat in token row, NOT callback chat ‚Äî for cross-channel question/answer flows
**Rule**: When a system-issued button can be DELIVERED to a different chat than the one where the AI was responding (e.g. AI is talking in user-account group X, but the prompt is delivered via the bot to that same group X ‚Äî same chat today, but design for divergence), persist BOTH `source_chat_id` (where the AI was responding) AND `delivery_chat_id` (where the button was sent) in the token row. The callback handler resolves the SOURCE chat from the token (via `getChatById(claimed.source_chat_id)`), inserts the synthetic `[Choice: <label>]` message into THAT chat, and calls `generateAndReply` on the source chat ‚Äî NOT the callback's sender chat. In v1 source and delivery are equal, but the columns are separated for future divergence (e.g. delivering to a dedicated "decisions" channel while the conversation continues in the original source).
**Why**: The "obvious" implementation pulls chat info from the callback event itself (`event.chatId`), which works as long as the prompt was delivered to the same chat the conversation is in. The moment that assumption breaks ‚Äî operator wants approvals routed to an admin channel, multi-channel orchestration, etc. ‚Äî the conversation history fragments into the wrong chat. Storing source_chat_id in the token row makes the conversation-continuity rule explicit at the schema level: "this token, when claimed, belongs to THIS conversation regardless of where the button physically lives." The cost is one extra column and one extra `getChatById` lookup per claim ‚Äî trivial. The benefit is the cross-channel design space stays open without a future migration.
**Apply when**: Any UI element (button, form, poll) that is issued by an AI in one logical conversation and rendered in a possibly-different physical channel. Persist source AND delivery separately even if they're equal in v1; resolve the source at claim time; insert any synthetic re-entry message into the source. Same pattern applies to OAuth callback flows, email-link confirmations, SMS reply codes ‚Äî wherever the "where to resume" answer differs from the "where the click happened" answer.

## [2026-05-08] ‚Äî `PASS-by-inspection` is a legitimate verifier verdict ‚Äî name the unrunnable boundary explicitly
**Rule**: When verification of a behavior requires a live external service (Telegram round-trip, real LLM call, third-party webhook delivery) that the verifier cannot run in-environment, the right verdict for that criterion is `PASS-by-inspection` ‚Äî meaning the code path was traced against the plan and each branch maps correctly, but the live behavior was not directly observed. Name the unrunnable criteria in a "Caveats / not-runnable in this environment" section AT THE BOTTOM of the verify report so the operator knows what to smoke-test before declaring production-ready. Do NOT mark as PASS without qualification (overstates confidence) and do NOT mark as FAIL (the code is correct, the test infrastructure is the gap).
**Why**: A binary PASS/FAIL forces the verifier to either lie (PASS for unverified live paths) or block on infrastructure they don't control (FAIL for paths that look correct). The `PASS-by-inspection` middle verdict is honest: it says "I read the code, it matches the plan, I cannot run the integration test." The Caveats section pairs it with an actionable handoff ‚Äî the operator knows EXACTLY which criteria need a manual smoke test. Without naming the boundary explicitly, a future reader of the verify report has no idea which "PASS" was observed live and which was inferred from inspection. The discipline also prevents scope creep: the verifier doesn't try to mock the entire Telegram + Gemini stack just to convert a PASS-by-inspection into a PASS ‚Äî they document the limitation and move on.
**Apply when**: Any verification task where some acceptance criteria require live external integrations the verifier can't reproduce. Use the trichotomy `PASS / PASS-by-inspection / FAIL`; pair PASS-by-inspection items with a Caveats section listing what the operator must smoke-test. Pattern generalizes to any pre-prod gate: code review approval ("approved subject to smoke test of the OAuth flow"), security review ("approved; runtime verification of CSP requires staging deploy"), performance review ("approved; load test requires real traffic profile").

## [2026-05-08] ‚Äî Logger wrap is the zero-edit hook for cross-cutting metrics side-effects
**Rule**: When adding cross-cutting instrumentation (error counters, request counters, structured event emission) that should fire at every existing `logger.error`/`logger.warn` site, modify the LOGGER itself ‚Äî wrap the existing `error`/`warn` body so it ALSO calls the side-effect AFTER the original console/file write ‚Äî instead of editing every call site. The added call MUST be wrapped in a `try { sideEffect(...) } catch {}` so a bug in the side-effect can never crash the logger. Combine with TARGETED `incCounter` calls at hot paths that need stable named buckets (`router.message_received`, `gemini.call.ok`) ‚Äî the logger wrap covers the long tail; the named counters cover the dashboard-prominent buckets.
**Why**: Hand-instrumenting every `logger.error` site is N edits with high miss rate (some files get touched, some don't); the dashboard then shows a partial picture and operators don't know which gaps are "no errors" vs "instrumentation missing". The logger wrap captures EVERY `logger.error` automatically ‚Äî including future ones added after the instrumentation lands. The try/catch is load-bearing: the file's existing "logger must never throw" comment means any side-effect that bubbles an exception out is a regression of that invariant. The hybrid (logger-wrap + targeted named counters) gives both completeness and naming discipline without forcing every call site to opt in.
**Apply when**: Adding any "fire at every existing log call" instrumentation ‚Äî error metrics, structured-log shipping, OpenTelemetry span attributes derived from log fields, audit trails. Always: wrap inside try/catch, document the invariant inline (`"metrics must never throw out of the logger"`), and confirm no circular import (the side-effect module must not transitively import the logger).

## [2026-05-08] ‚Äî Metrics module must use `console.*`, not `logger.*`, to keep the import graph acyclic
**Rule**: A metrics/instrumentation module that the logger calls into (per the logger-wrap pattern) MUST NOT import the project logger ‚Äî it would create a cycle (`logger.ts ‚Üí metrics.ts ‚Üí logger.ts`). Use bare `console.warn`/`console.log` for the rare diagnostics the metrics layer itself needs to emit (config-missing notices, write-failure callbacks). Document the constraint at the top of both files: in `logger.ts` ("metrics module imported below; that module must not import this one back") and in `metrics.ts`/`influx.ts` ("uses console.* instead of project logger to avoid circular import with logger.ts").
**Why**: The cycle is silent at compile time (TS allows it) but fragile at runtime ‚Äî module-init order can leave one of the two modules with `undefined` exports during the brief window the cycle resolves, producing bizarre "X is not a function" errors only at startup. The `console.*` escape hatch costs nothing (the noise is bounded ‚Äî config notices fire once at boot; write callbacks are infrequent) and keeps the import graph a DAG. Same pattern applies to any "logger calls into X, X needs to log too" situation: X uses `console`, not the logger.
**Apply when**: Designing any module that the logger transitively depends on (metrics, telemetry buffers, panic handlers). Identify the cycle BEFORE writing the code; pick the module to break the cycle (usually the lower-level one); use `console.*` there with a documenting comment.

## [2026-05-08] ‚Äî Synchronous coherent snapshot before any `await` for in-process metric capture
**Rule**: When persisting an in-process metrics state (Maps of counters/gauges/histograms) to an external store, capture the entire snapshot SYNCHRONOUSLY in one JS tick BEFORE the first `await` ‚Äî `const snap = { counters: Object.fromEntries(counters), gauges: Object.fromEntries(gauges), histograms: getHistograms(), errors: [...errorBuckets.values()] }` ‚Äî then build serialization payloads from `snap`. JavaScript is single-threaded, so any concurrent `incCounter` calls that arrive during the subsequent `await writePoints(...)` land in the NEXT snapshot, not a half-written current one. NEVER iterate a Map and `await` per item ‚Äî the second iteration's values will not match the first's.
**Why**: Iterating `for (const [name, value] of counters) { await write(...) }` exposes the writer to concurrent mutation between iterations: a counter incremented after iteration N starts but before iteration N+1 reads it produces a corrupt snapshot where some counters reflect a later state than others. The `Object.fromEntries(map)` capture takes microseconds (single tick, no yield) and produces a frozen-in-time view. Persistence proceeds against the snapshot; the live maps keep ticking. No locking, no copy-on-write, no atomic primitives needed ‚Äî JS's single-threaded model is the locking primitive when used correctly.
**Apply when**: Any "drain in-process state to external store" code ‚Äî metrics persistence, log batching, event buffer flushing, queue draining. Always: synchronous capture FIRST, then serialize/await against the captured snapshot. The cost (one extra map allocation) is trivial; the alternative (subtle inter-iteration drift bugs that only show up under load) is not.

## [2026-05-08] ‚Äî Fire-and-forget batch writers: empty batch early-return is correct, document the `lastWriteAt=null` consequence
**Rule**: For a fire-and-forget batch writer (`writePoints(points: Point[])`), early-returning when `points.length === 0` is the right behavior ‚Äî there is literally nothing to write, no point in calling the underlying SDK. The consequence is that "last write timestamp" tracking (e.g. `lastWriteAt` updated by a `writeSuccess` callback) stays NULL when the snapshot is empty (no counters incremented yet). The dashboard tile reading `server.snapshot_at` will show "never persisted" even though the persistence interval IS running and would write if there was data. Document this in BOTH the metrics persistence function AND in the verify-results doc ‚Äî "empty snapshots are no-ops by design; `snapshot_at` stays null until first non-empty snapshot."
**Why**: The "always write a heartbeat" alternative (push at least one dummy point per snapshot to update `lastWriteAt`) pollutes the time-series with synthetic data and confuses any downstream Flux query that treats the bucket as "real metrics only". The empty-snapshot no-op preserves the bucket's semantic cleanliness at the cost of one observable quirk: a freshly-booted server with no traffic shows `snapshot_at: null` even after the interval has fired. Operators reading the dashboard in this state get confused ("is persistence broken?"); the inline doc / verify note prevents the wasted debugging cycle. Same pattern applies to any "drain queue" interval that legitimately has nothing to drain on most ticks.
**Apply when**: Implementing any periodic-flush writer where the batch size can legitimately be zero. Choose: (a) early-return + document the observable consequence (preferred), or (b) emit a synthetic heartbeat point + accept the bucket pollution. Default to (a); reserve (b) for cases where downstream consumers REQUIRE a "still alive" signal (heartbeat-driven liveness checks).

## [2026-05-08] ‚Äî Restore counters/gauges/errors on boot, but never restore histogram rings
**Rule**: When persisting in-memory metrics to an external store and restoring on boot, restore COUNTERS (monotonic totals ‚Äî losing them resets the operator's long-running count), GAUGES (last-known scalar ‚Äî useful to show "subscribers: 3" immediately on boot rather than 0), and ERROR BUCKETS (the operator's "what was failing" view should survive restart). Do NOT restore HISTOGRAM RINGS ‚Äî they're a sliding window of recent samples; restoring stale samples produces misleading p95/p99 values that mix yesterday's data with today's. Histograms start fresh; the SUMMARIES are still queryable from the time-series store for charts.
**Why**: Counters and gauges are summary statistics where the historical value IS the truth (total messages received since boot, current subscriber count). Histograms are sample distributions where freshness IS the value ‚Äî a p95 latency from 30 minutes ago combined with the last 5 minutes of new samples produces a meaningless mixed distribution. The persistence layer can still write histogram summaries (count, p50, p95, p99, max, mean) per snapshot for time-series CHARTS, but the live ring buffer must not be reseeded from those summaries. Document the asymmetry inline (`// histograms intentionally start fresh; rings would mix old and new samples`).
**Apply when**: Designing the boot-restore path for any in-memory metric system. Categorize each metric type: monotonic totals ‚Üí restore; current scalars ‚Üí restore; sliding-window samples ‚Üí start fresh. Document the rule in the restore function's docstring; don't let a future "restore everything for completeness" refactor smash the asymmetry.

## [2026-05-08] ‚Äî Single Flux composite query restores multiple measurements; iterate and switch on `_measurement`
**Rule**: When restoring multiple metric kinds (counters, gauges, errors) from a time-series store at boot, run ONE composite Flux query with an OR-filter across measurements ‚Äî `from(bucket) |> range(start: -1d) |> filter(fn: (r) => r._measurement == "tele_counter" or r._measurement == "tele_gauge" or r._measurement == "tele_error") |> last()` ‚Äî and SWITCH on `_measurement` while iterating the row stream to dispatch each row to its target Map. Do NOT issue three separate queries; do NOT issue one query per metric.
**Why**: Each Flux query is a network round-trip and a server-side compile + execute. Three separate queries triple the boot-time restore latency for no benefit; one query per metric is pathological at hundreds of metrics. The composite query lets InfluxDB's TSI index do the work in a single scan; the client-side switch on `_measurement` is microseconds. The `|> last()` aggregator returns one row per unique tag combination, so the result size stays bounded by the cardinality (counter count + gauge count + error-bucket count) regardless of how much history exists. Same pattern works for any time-series store with a discriminator column (Prometheus `__name__`, OpenTSDB metric name).
**Apply when**: Any boot-restore or batch-load against a time-series store with multiple "kinds" of metrics. Default to one composite query + client-side dispatch; only split into separate queries if a kind needs a fundamentally different time range or aggregation (e.g. errors want `last 100 per bucket` while counters want `last 1 per name`).

## [2026-05-08] ‚Äî Counter naming discipline is the load-bearing precondition for time-series persistence
**Rule**: Before adding ANY external time-series persistence layer (InfluxDB, Prometheus, Datadog), audit the counter/gauge/histogram naming convention to confirm cardinality is BOUNDED at the source. The rule: `<subsystem>.<event>[.<variant>]`, depth ‚â§ 3, and ZERO dynamic high-cardinality segments ‚Äî no `chat_id`, no `user_id`, no UUID, no per-message label as part of the metric NAME. Tags follow the same rule. Document the cardinality budget in the persistence design ("expect ~80 unique counter names, ~50 histogram names, ‚â§100 error buckets ‚Äî well under any free-tier rate limit"). Before adding a new counter that interpolates a runtime value into the name, ask: "what is the upper bound on the distinct values this can produce?" If it's "unbounded" or "scales with users/messages/data", reject the design and find a different shape (a tag with a known small enum, or a separate gauge per fixed category).
**Why**: Time-series stores price by series cardinality, not by point volume ‚Äî a few thousand series writing 1 point/min each is cheaper than ten series writing 1 million points/min. A single bug like `incCounter("tool.invoked." + chatId + ".ok")` can blow up the series count from ~80 to "one per chat ever seen" within hours, breaking the rate limit and silently dropping all metrics until the bug is caught. The naming discipline must be a CODE REVIEW CHECK, not just a documentation aspiration ‚Äî every PR adding `incCounter(name)` should have the name's cardinality justified inline. **Apply when**: Adding any external metrics persistence layer to an existing app, OR adding any new metric to an app that already has external persistence. ## [2026-05-08] ‚Äî Cost accumulation in micro-USD integers avoids float drift across thousands of calls
**Rule**: When accumulating per-call USD cost into a long-running counter, store the counter as `Math.round(prompt_tokens * input_per_1m_usd + completion_tokens * output_per_1m_usd)` ‚Äî this resolves to MICRO-USD ($0.000001 units) as integers, NOT float USD. The dimensional sanity check: `tokens * (USD per 1M tokens)` = `tokens √ó USD √ó 10^-6 √ó 10^6 / 10^6` = micro-USD per call. The dashboard divides by `1_000_000` only at render time. Counter type stays `number` (JS); 2^53 ‚âà $9 quadrillion of headroom is plenty.
**Why**: Floating-point accumulation drifts: $0.0001234 added 10,000 times produces noticeable error. `Math.round` to nearest micro-USD per call bounds the drift at 1 micro-USD per call regardless of how many calls accumulate. Storing in integer-units in the counter Map preserves the JS-`number` arithmetic shape (no BigInt complexity) while sidestepping the FP issue. The render-time divide by 1e6 produces clean `$0.0000`-formatted strings without `.toFixed(20)` rounding artifacts. Same trick (integer base unit + render-time scale) works for any small-fractional accumulator: micro-bytes, milli-events, microseconds.
**Apply when**: Any per-call accumulator where the per-event value is small and the accumulated total spans many events. Choose the smallest integer unit that fits 1+ digits per event (micro-USD for cost, microseconds for fast latency, bytes for size); round once at the per-event boundary; scale once at the render boundary. Document the unit in the counter NAME (`gemini.cost_micro_usd`, NOT `gemini.cost`) so a future reader can't misinterpret the magnitude.

## [2026-05-08] ‚Äî Stale-gated post-boot one-shot + long-period interval is the right cadence shape for slow-changing external state
**Rule**: For features that periodically refresh external state where the external state changes slowly (vendor pricing, GeoIP DB, currency rates), pair a long-period `setInterval(refresh, 24*60*60*1000)` with a STALE-GATED post-boot `setTimeout(refresh, 2*60*1000)` ‚Äî the one-shot only fires if the cached state is missing OR `fetched_at < now - <staleness threshold>` (e.g. 7 days). Don't refresh on every boot (wastes vendor API calls when the operator restarts frequently); don't skip the one-shot entirely (a fresh dev environment never gets data); don't tighten the interval below the actual change cadence (noise + cost without benefit).
**Why**: A naive "refresh on every boot" pattern punishes operators for restarting ‚Äî every dev-loop iteration burns a vendor API call. A naive "interval-only with no boot one-shot" leaves a fresh database with no data until the first interval tick (24h later). The stale-gated one-shot reconciles both: long-running prod processes don't re-fetch on restart, fresh dev environments self-heal within 2 minutes. The 7-day threshold tolerates a missed daily refresh (process crashed during the timer window) and forces a re-fetch on truly stale data. The 2-minute delay (not immediate) lets the rest of the boot sequence settle before competing for the network.
**Apply when**: Any boot-time + periodic refresh of slow-changing external state. Tune the three knobs to the change cadence: interval ‚âà how often it actually changes; staleness ‚âà how long is "too old"; one-shot delay ‚âà how long to let boot settle. Document all three values inline at the registration site so a future tuning pass has the rationale.

## [2026-05-08] ‚Äî Flux delta over a window: two queries (`first()` + `last()`) on a monotonic counter
**Rule**: To compute "cost in last 24h" from a monotonic counter persisted to InfluxDB, run TWO separate Flux queries on the same `range(start: -24h)` filter ‚Äî one with `|> first()` and one with `|> last()` ‚Äî and subtract `Math.max(0, last - first)` server-side in JS. Wrap the whole block in try/catch so a Flux blip returns `null` for that field rather than crashing the metrics endpoint. If the older sample is missing (server uptime < window), the field still falls back to `null` cleanly because `first` will be `null`.
**Why**: A single composite Flux query that returns both `first` and `last` in one pipeline is possible but more brittle (uses `pivot()` or `union()` that's harder to debug). Two queries are ~100ms total, dispatched serially ‚Äî negligible on a 5s dashboard poll. Subtracting in JS is one line vs. learning the Flux subtraction syntax. The `Math.max(0, ...)` guards against the edge case where the counter was reset between samples (operator reset, DB wipe) ‚Äî without it, a reset produces a negative "delta" which renders as gibberish. The try/catch + null fallback follows the standard "external query failure shouldn't crash the endpoint" pattern.
**Apply when**: Any windowed delta computation on a monotonic counter from a time-series store. Default to two queries + subtract in app code; reserve composite Flux pipelines for when the windowing logic is genuinely complex (multiple groupings, derived rates). Always: clamp to `>= 0` to handle resets; null-fallback on query failure; document the "reset = negative" edge case inline.

**Rule**: When a value can come from either an auto-fetcher or an operator manual override (vendor pricing, default permissions, theme), store BOTH in dedicated columns on the SAME row ‚Äî `input_per_1m_usd` (auto) and `override_input` (manual) ‚Äî and resolve precedence at LOAD time in the accessor: "if both override columns are non-null ‚Üí use overrides + `is_override=true`; else if both auto columns are non-null ‚Üí use auto + `is_override=false`; else null." **Why**: Separate columns let the operator clear the override (`PUT { override_input: null, override_output: null }`) and fall back instantly to the cached auto-fetched values without re-running the fetch. The "load-time precedence" pattern keeps every consumer's read path identical (`getCurrentPricing()` returns one shape regardless of source) ‚Äî only the accessor knows about the override resolution. The `is_override` boolean surfaces in the API response so the dashboard can show an "override" badge, providing operator feedback without requiring the consumer to reason about both columns. The "must set both or neither" zod refinement on the override PUT prevents a malformed half-override state.
**Apply when**: Any data point with both an automated source-of-truth and an operator-override path. Always: separate columns for each source, single accessor that resolves precedence, boolean flag in the read shape, "must set both or neither" validation if the override has paired fields. Same pattern works for default-vs-custom themes, system-vs-user permissions, ML-predicted-vs-human-corrected labels.

## [2026-05-08] ‚Äî Strict-JSON LLM prompts: regex-strip code fences as the parse fallback, not the parse default
**Rule**: When prompting an LLM for strict JSON output (data extraction, structured tool input, pricing fetch), the prompt should EXPLICITLY say "no markdown, no code fences, no commentary ‚Äî JUST the JSON object." Then in the parser: try `JSON.parse(text.trim())` FIRST; on failure, try ONE fallback that strips markdown code fences via `text.replace(/^```(?:json)?\s*/, "").replace(/```\s*$/, "").trim()` and re-parse; if both fail, log the first 500 chars of the raw response at warn level and return without writing. Do NOT make code-fence stripping the default path (hides the model's deviation from the prompt); do NOT add multiple fallback regex layers (each layer is a place for malformed input to slip through).
**Why**: Models occasionally wrap JSON in ```json ... ``` fences despite the prompt explicitly forbidding them ‚Äî a single fallback regex catches this without legitimizing the deviation. The "try strict first, fallback once, log+skip" pattern preserves observability: the warn-log raw-text-prefix tells the operator EXACTLY what the model returned, making prompt-tuning iterations debuggable. Multiple fallback layers obscure the original failure mode and let new model misbehaviors silently slip through. Same pattern works for any "ask the LLM for structured output" parser; the "log first 500 chars" heuristic is enough to diagnose 90% of real failures without flooding logs.
**Apply when**: Any LLM-as-structured-data-source. Always: strict prompt explicitly forbidding markdown, primary parse on raw text, ONE fallback for fence-stripping, log+skip on double failure. zod validation runs AFTER successful parse to catch shape errors (NaN, negative numbers, missing fields, wrong types). Pair with the manual override path (per the override-columns lesson) so the operator can recover when both parse paths fail.

## [2026-05-08] ‚Äî Atomic object-reference swap for hot-path config: read once synchronously, never field-by-field
**Rule**: When a hot-path (per-AI-call, per-request) reader needs config that can be refreshed asynchronously (vendor pricing, feature flags, rate limits), the accessor MUST return a CAPTURED OBJECT REFERENCE ‚Äî `let cached: ActivePricing | null = null;` at module scope ‚Äî and `getCurrentPricing()` returns `cached` (the entire object). **Why**: JS is single-threaded so the swap (`cached = newObj`) is atomic at the assignment-statement level, but field-by-field reads (`pricing.input_per_1m_usd` at one site, `pricing.output_per_1m_usd` at another site, with an async tick or unrelated function call between them) expose the reader to a refresh landing between the reads. The pattern generalizes the snapshot-before-await rule (existing 2026-05-08 lesson) to async-refreshed config: the rule isn't "synchronous capture before await" ‚Äî it's "single read of a captured reference, regardless of awaits." Same mechanism, different surface.
**Apply when**: Any module-level `let cached = ...` that's refreshed async and read on hot paths. Always: return the whole object reference from one accessor, never split into per-field accessors that re-read the cache. The refresh path swaps the whole object, never mutates fields in place. Document the invariant in the accessor's docstring (`"returns a captured snapshot ‚Äî destructure at call site for coherent reads"`) so a future "let me add a getInputRate helper" refactor sees the warning.

## [2026-05-08] ‚Äî Client-side rolling ring buffer for short-window sparklines avoids the time-series round-trip

## [2026-05-11] ‚Äî Flux operator selection by metric kind: counters use `derivative`, histograms use `aggregateWindow(last)`, gauges use `aggregateWindow(last)`, errors use `derivative` grouped by tag
**Rule**: When authoring Flux queries against a self-instrumented persistence layer that stores `tele_counter` (monotonic), `tele_gauge` (last-known scalar), `tele_histogram` (per-snapshot summary fields like `p95`/`p99`), and `tele_error` (per-source rolling buckets), pick the aggregation operator strictly by metric kind: COUNTERS ‚Üí `|> derivative(nonNegative: true, unit: 1m)` (rate per minute) or `|> last() - first()` for windowed delta; HISTOGRAMS ‚Üí `|> filter(_field == "p95" or _field == "p99") |> aggregateWindow(every: v.windowPeriod, fn: last, createEmpty: false)` ‚Äî NEVER `derivative` on histograms; GAUGES ‚Üí `|> aggregateWindow(every: v.windowPeriod, fn: last, createEmpty: false)`; ERRORS ‚Üí `|> derivative(nonNegative: true, unit: 5m)` grouped by `source` tag. Document the asymmetry in a top-of-file comment in the dashboard JSON or its accompanying README.
**Why**: Histograms in this app's persistence layer are PER-SNAPSHOT summaries, not cumulative ‚Äî each snapshot writes the current p95/p99 of the ring buffer. A `derivative` on `p95` would compute "how much did p95 change between snapshots" ‚Äî meaningless and confusing (a stable p95 of 50ms produces a derivative of 0, not "the actual p95"). Counters are monotonic ‚Äî `derivative` produces the per-unit rate, which is what "messages per minute" charts want; `nonNegative: true` clamps the post-restart blip to 0 (per the existing reset lesson). Gauges are last-known ‚Äî `last()` over each window grid cell gives the closest-to-real-time value without averaging across rapid changes. Errors-by-source benefits from the `derivative` pattern (rate of new errors) grouped by the bounded `source` tag (capped at 100 distinct sources by the in-process bucket cap). Choosing the wrong operator for the kind produces charts that LOOK correct (lines plotted, axes drawn) but show meaningless data.
**Apply when**: Authoring any Flux query (or equivalent in another time-series query language) against a metrics persistence layer with mixed metric kinds. Match the operator to the kind, not to the visual the panel is supposed to show. If unsure, the rule is: monotonic ‚Üí `derivative` for rates, `last - first` for windowed deltas; per-snapshot scalar ‚Üí `aggregateWindow(last)`; sliding-window summary ‚Üí never derivative, always `last` or `mean` over the aggregation window. Same logic applies to Prometheus PromQL: counters use `rate()` / `increase()`; gauges and histogram summary fields use raw values or `avg_over_time`.

## [2026-05-11] ‚Äî Top-N regex filter on counter names (`name =~ /^prefix\..+\.suffix$/`) beats enumerating known values
**Rule**: When a "top tools" / "top events" dashboard panel needs to surface the top N of a metric family with an extensible naming convention (e.g. counters named `tool.invoked.<name>.ok` where new tools land continuously), use a Flux regex filter ‚Äî `|> filter(fn: (r) => r._measurement == "tele_counter" and r.name =~ /^tool\.invoked\..+\.ok$/) |> last() |> group(columns: ["name"]) |> sort(columns: ["_value"], desc: true) |> limit(n: 15)` ‚Äî INSTEAD of enumerating known tool names with `name == "tool.invoked.search.ok" or name == "tool.invoked.read_file.ok" or ...`. ## [2026-05-12] ‚Äî Local edit-state effect dep `chat.id` (not `chat.field`) protects in-flight typing from WS clobbers
**Rule**: When a React component holds local edit state (`textarea` value, draft form) seeded from a prop (`chat.ai_context`) that can also update via WebSocket push, the `useEffect` that reseeds the local state MUST depend on `chat.id` ONLY ‚Äî not on `chat.ai_context`. The trade-off is concurrent edits in a different tab don't push into this tab's textarea ‚Äî acceptable because typing locally takes precedence over remote echoes.

**Apply when**: Any controlled-input component whose initial value comes from a prop that can also update asynchronously (WebSocket, polling, optimistic update from a sibling). ## [2026-05-12] ‚Äî Trim-empty-to-NULL at the route boundary collapses empty UI input into the schema's "absent" state

The schema MUST allow null (`z.string().max(8000).nullable()`) so the explicit-null path (slash command `/context clear` sending `null`) still works.

Doing the trim at the route boundary (not in the repo or the slash branch) keeps it in one place ‚Äî the slash branch's explicit-null path still works because it bypasses the trim entirely. **Apply when**: Any nullable-text PATCH/PUT route fed by a dashboard textarea or input field. Always: trim at the route boundary; map empty-trimmed-string ‚Üí null; preserve null (`.nullable()`) in the zod schema for explicit clears. If the field is non-nullable (must always have a value), reject empty strings at zod with `.min(1)` instead ‚Äî pick one model and stick with it.
