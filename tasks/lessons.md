# Lessons Learned

## [2026-04-28] — Neon serverless driver: no `unsafe`, no `query`, no multi-statement SQL
**Rule**: With `@neondatabase/serverless`, the SQL function exposes only two forms: the tagged template (`` sql`SELECT ${x}` ``) and the direct-call form (`sql(text, params)`). There is no `sql.unsafe(...)`, no `sql.query(...)`, and no built-in `sql.transaction([...])` for arbitrary multi-statement strings. A migrator that reads a `.sql` file and tries to send it as one string will throw at runtime.
**Why**: The plan's first migration runner used `sql.unsafe(fileContents)` and the `db/index.ts` helper used `sql.query(text, params)` — both compiled cleanly as `any`-typed access but failed at runtime. The verifier caught this only after running `tsc` against a real `index.d.ts`. Migrations would have crashed on the very first boot, blocking acceptance criterion 1.
**Apply when**: Writing any DB code against the Neon serverless driver — especially custom migrators, schema scripts, or any "run this raw SQL blob" helper. Split multi-statement files on `;` and execute each via `sql(stmt, [])`. Make DDL idempotent (`CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`) since you cannot wrap the whole file in a single transaction with this driver.

## [2026-04-28] — GramJS `EntityLike` rejects `bigint`
**Rule**: GramJS `client.sendMessage(peer, ...)` and similar APIs type `peer` as `EntityLike = string | number | Entity`. Passing `BigInt(chatId)` is a TypeScript error and may or may not coerce at runtime depending on the call path. Convert Telegram IDs to `Number(...)` (safe for IDs < 2^53) or to a string (`peer.toString()`) before handing them to GramJS.
**Why**: Plan stored Telegram chat IDs as `BIGINT` in Postgres (correct — they can exceed 2^31), and the natural reflex was to pass `BigInt(chatId)` back into GramJS calls. That fails to typecheck and is fragile at runtime.
**Apply when**: Bridging persisted Telegram IDs (typically `string` after coming back from a `BIGINT` column) into any GramJS call. Use `Number(id)` for entity arguments; keep the string/bigint form only for DB writes and audit logs.

## [2026-04-28] — Gemini SDK `Part` union needs a widened type predicate
**Rule**: The `@google/generative-ai` `Part` type is a discriminated union; filtering with `(p): p is { functionCall: FunctionCall } => !!p.functionCall` is too narrow and trips TS2677 ("type predicate's type must be assignable to its parameter's type"). Use `(p): p is Part & { functionCall: NonNullable<Part["functionCall"]> } => !!p.functionCall` instead, or just filter without a predicate and use a non-null assertion downstream.
**Why**: A naive narrow predicate fails compilation and cascades into "possibly undefined" errors at every downstream `fc.functionCall.name` access. The function-call loop is hot-path code; this needs to compile cleanly.
**Apply when**: Implementing a Gemini tool/function-call loop that filters `response.candidates[0].content.parts` for `functionCall` parts. Same pattern applies to filtering for `functionResponse` or `text` parts in the SDK.

## [2026-04-28] — `packageManager: "pnpm@9"` is rejected by pnpm v10
**Rule**: The `packageManager` field in `package.json` requires a full semver, not a major-version shorthand. `"pnpm@9"` causes pnpm v10 to warn `Cannot switch to pnpm@9: "9" is not a valid version` and skip the version switch. Use `"pnpm@9.0.0"` (or whatever exact version you want) instead.
**Why**: Corepack and pnpm's self-update logic both expect a parseable semver. The shorthand silently degrades behavior — install still works but the pin is non-functional, which defeats the purpose.
**Apply when**: Setting up `packageManager` in any monorepo root `package.json`. Pin to a real version (e.g. `9.15.0`) and update intentionally.

## [2026-04-28] — Build workspace packages before type-checking their consumers
**Rule**: In a pnpm workspace where `apps/server` depends on a local `packages/shared`, running `tsc --noEmit` on `apps/server` fails with `Cannot find module '@tele/shared'` until `packages/shared` has been built (its `dist/` exists with `.d.ts` files). Always build shared first.
**Why**: Workspace symlinks resolve to `packages/shared/package.json`, whose `"main"` / `"types"` fields point at `dist/`. Without a build step the compiler sees the package but no types. The error is misleading — it looks like a missing dependency, not a missing build artifact.
**Apply when**: Running per-package type checks in CI or locally in a monorepo with `tsc`-built shared packages. Either build shared packages first (`pnpm -F @tele/shared build && pnpm -F @tele/server typecheck`), or configure TS project references with `composite: true` so `tsc -b` handles the build order automatically.

## [2026-04-29] — GramJS Poll API: `TextWithEntities` for question/answers, `correctAnswers` on `InputMediaPoll`
**Rule**: In current GramJS versions, `Api.Poll.question` and `Api.PollAnswer.text` are `Api.TextWithEntities({ text, entities: [] })` objects — not plain strings. Correct answer for quiz polls goes in `Api.InputMediaPoll.correctAnswers: Buffer[]`, not in `Api.Poll` itself. `Poll.id` and `SendMedia.randomId` must be `bigInt.BigInteger` — use `helpers.generateRandomLong()` from the `telegram` package.
**Why**: Telegram migrated from plain strings to `TextWithEntities` across poll fields. Code assuming plain strings compiles if you bypass typechecking but produces a GramJS runtime error or malformed request. The `correctAnswers` placement on `InputMediaPoll` (not `Poll`) is non-obvious from docs alone.
**Apply when**: Any code that constructs `Api.Poll`, `Api.PollAnswer`, or `Api.InputMediaPoll`. Always read the installed `api.d.ts` before writing — never assume the field shapes from memory or older docs.

## [2026-04-29] — Persisting poll state as synthetic outbound message avoids schema migration
**Rule**: When adding a Telegram poll-send feature, persist the poll as a plain text message `[Poll: <question>] opt1 | opt2 | ...` using the existing `messages` table and `insertMessage` helper. This lets the dashboard render it without schema changes or new UI components.
**Why**: Adding a structured `polls` table would require a migration, new repo functions, new API routes, new dashboard components, and vote-tracking infrastructure — all out of scope for an "AI can send polls" feature. The `[Poll: ...]` convention follows the established `[Image: ...]` / `[File: ...]` patterns already in sender.ts.
**Apply when**: Adding new Telegram media types (stickers, polls, location shares, etc.) where the primary goal is AI-initiated sending, not vote/response tracking. Use the synthetic message pattern; defer structured tracking to when vote results must feed back into the AI context.

## [2026-04-30] — FK to internal `chats(id)` UUID, not raw `tg_chat_id` BIGINT
**Rule**: When a new table needs to reference a "person/chat" (e.g. `kanban_tasks.assignee_chat_id`, `reminders.target_chat_id`), make the FK reference `chats(id)` (UUID) — never `chats.tg_chat_id` (BIGINT). Resolve display names via a `LEFT JOIN chats c ON c.id = t.assignee_chat_id` and a `COALESCE(NULLIF(TRIM(CONCAT_WS(' ', c.first_name, c.last_name)), ''), c.username, c.tg_chat_id::text)` expression.
**Why**: `chats.id` is the established internal PK pattern; using it makes joins trivial and avoids any GramJS coercion (the BIGINT path forces `Number(id)` / `String(id)` conversions in every consumer). It also lets `ON DELETE SET NULL` / `CASCADE` semantics work cleanly. The same pattern is in `reminders.target_chat_id` — follow it for parity.
**Apply when**: Adding any new relational column that points at a person, chat, or contact. Always FK to `chats(id)`, never to `tg_chat_id`. Pre-validate the FK in the AI tool handler (`getChatById(id)` → `{ ok: false, error: "unknown contact id; call lookup_contacts first" }`) so the model learns in-loop instead of hitting a 500 on the server.

## [2026-04-30] — Distinguish "key absent" from "key present, value null" in PATCH/PUT repos
**Rule**: For an `updateTask(id, patch)` repo that supports clearing a nullable FK (e.g. `assignee_chat_id: null` to unassign), check `Object.prototype.hasOwnProperty.call(patch, "assignee_chat_id")` — not `patch.assignee_chat_id !== undefined` and not a truthy check. The presence of the key (regardless of value) means "write this field"; absence means "leave it alone". Pair the repo with a zod schema like `z.string().uuid().nullable().optional()` so `null` survives validation.
**Why**: Truthy checks (`if (patch.assignee_chat_id)`) silently drop `null`, making "unassign" impossible. `!== undefined` works only if the JSON parser preserves the distinction (it does), but `hasOwnProperty` is the explicit, intent-revealing form. Also: if the dynamic SET builder finds no real fields it must reject the update rather than running `UPDATE ... SET updated_at = now()` with nothing else — Gemini will sometimes call `update_task(id)` with no other args.
**Apply when**: Writing any partial-update repo where some fields are nullable FKs/strings the caller may want to clear. Same pattern applies to AI tool handlers receiving partial JSON args from Gemini.

## [2026-04-30] — Tool-call response payloads must be trimmed to keep Gemini context small
**Rule**: When a Gemini tool returns a list (`list_tasks`, `list_chats`, etc.), trim each row to the minimum fields the model needs to act — typically `(id, title, status, assignee_id, assignee_name, comment_count)` — and explicitly drop heavy fields like `description`, `body`, attachments, full timestamps. Document this in the tool description so the model knows to call a separate `get_*(id)` tool for the full row when needed.
**Why**: Tool responses are echoed back into the model's context on every turn. A board with 200 tasks × full descriptions can blow past token budgets and degrade response quality. Trimming at the tool boundary (not in the repo) keeps the REST API rich while keeping the AI loop lean.
**Apply when**: Authoring any AI tool that returns a list. Default to a trimmed projection; only return full rows from singular `get_*(id)` tools. If filters could be expressed in SQL (e.g. `assignee_chat_id`, `status`), prefer SQL over post-fetch JS filtering once the table grows past a few hundred rows.

## [2026-04-30] — `WsEvent` discriminated union: verify consumers have no silent `default` branches
**Rule**: When extending the `WsEvent` union in `packages/shared/src/types.ts` with a new `{ type: "..."; payload: ... }` variant, audit existing `useWsEvent` / switch consumers to confirm none have a `default:` branch that would silently swallow the new type. The `useWsEvent<T extends WsEvent["type"]>` generic picks up new variants automatically — but only if no consumer has a catch-all that drops them.
**Why**: TypeScript's exhaustiveness checking only fires on `switch` statements with a `never`-typed default. A `default: return;` or missing `default` in a hand-written dispatcher will compile fine and silently drop the new event at runtime — invisible until QA notices the dashboard doesn't refresh.
**Apply when**: Adding any new variant to a discriminated union consumed in multiple places. Grep for the union name and audit each consumer; prefer the generic `useWsEvent<"specific:type">(handler)` pattern over a hand-rolled switch so new variants are opt-in by name.

## [2026-04-30] — Split AI tool pairs into trimmed-list + full-get; never one tool that does both
**Rule**: When the AI needs both "discover what's available" and "load the full thing", expose two tools: `list_X` returns only `(id, name, description)` for every row, and `load_X({ id?, name? })` returns the full payload for one row. Don't add a `verbose: true` flag to `list_X` — the model will pass it.
**Why**: A single `list_X(verbose=true)` lets Gemini one-shot a 200-row × full-content response into context, blowing the budget. Two distinct tools force the model to choose `id` first, then load only what it needs. The discoverability tool stays cheap; the load tool is intentionally heavy but scoped to one item. This is the deliberate counterpart to the existing trim-list rule — same goal, structural enforcement instead of convention.
**Apply when**: Designing any tool surface for content the model needs to *select among, then read in full* (skills, prompts, documents, templates, runbooks). The list/get split is the right shape; resist the temptation to merge them.

## [2026-04-30] — AI tool args: accept both `id` and `name`, prefer `id`, error if neither
**Rule**: For lookup tools where the model might know either an opaque `id` or a friendly `name` (skills, contacts, chats, channels), accept both as optional args, prefer `id` when both are present, and return `{ ok: false, error: "provide id or name" }` when neither is given. Don't make either required.
**Why**: Gemini often remembers a friendly name from earlier in the conversation (`"grail-explorer"`) but not the UUID. Forcing `id`-only makes the model call `list_*` again to look up the id — wasted turn. Forcing `name`-only breaks when the user renames. Accepting both with a documented precedence is the friction-free path. The explicit "provide id or name" error teaches Gemini in-loop instead of silently returning an empty result.
**Apply when**: Any AI tool that resolves a single record by some user-meaningful identifier. Same pattern works for `chat`, `contact`, `template`, etc. Document the precedence in the tool description so the model can plan around it.

## [2026-04-30] — Path-takes-precedence-over-inline at load time, not at write time
**Rule**: When a record can hold either inline content or a filesystem path pointing at the source of truth (e.g. `skills.content` vs `skills.path`), store both columns but resolve at *load* time: if `path` is non-null, read the file fresh; otherwise use `content`. Do NOT sync `content` from `path` on write — leave `content = ''` when `path` is set.
**Why**: Editing a SKILL.md file on disk should automatically be reflected the next time the AI loads the skill. If you eagerly cache the file's contents into `content` on write, the row goes stale the moment the file changes. Storing `content = ''` alongside a non-null `path` makes the precedence visually obvious in the DB and prevents drift.
**Apply when**: Any "ref-or-inline" data model — skills, prompt templates, config blobs, image-by-URL-or-binary. Document the precedence in the schema comment and enforce at the load boundary, not the write boundary.

## [2026-04-30] — Catch Postgres unique-violation (`23505`) and return 409, never let it bubble as 500
**Rule**: When a route inserts/updates a row with a `UNIQUE` column the user controls (e.g. `skills.name`, `chats.username`), wrap the DB call in `try { ... } catch (err) { if (err?.code === "23505") return res.status(409).json({ error: "name already exists" }); throw err; }`. Apply on both POST (create) and PUT (update — rename collisions hit the same constraint).
**Why**: Postgres throws `23505` for unique violations; without the catch, Express returns 500 with a stack trace and the dashboard shows "Server error" — useless to the user, who just typed a duplicate name. A targeted 409 with a friendly message lets the form display the real problem ("name already exists") inline. Forgetting to apply this on PUT is the common miss — only POST gets remembered.
**Apply when**: Any route writing to a column with a `UNIQUE` constraint that's user-supplied. Audit both create AND update paths. Same pattern for FK violations (`23503`) → 400 "referenced row does not exist".

## [2026-04-30] — Mirror create-time validation in update routes — partial PATCH/PUT often skips invariants
**Rule**: When a CREATE route enforces "at least one of A or B must be provided" (e.g. `content` or `path` for skills), the corresponding UPDATE/PUT route MUST re-check the invariant after applying the patch. A naive `PUT { content: "", path: null }` will happily clear both and leave the row in a state CREATE would have rejected. Read the current row, apply the patch in memory, then re-validate before persisting.
**Why**: The verifier flagged this exact gap on the skills PUT route — POST rejected `{content: "", path: null}` but PUT accepted it, leaving a skill that loads to empty instructions. Partial-update endpoints inherit none of the create-time invariants for free. Either re-validate after merging the patch, or document the invariant as "create-only" and reject the field in the update schema.
**Apply when**: Any UPDATE/PATCH/PUT route on a table whose CREATE has cross-field validation (mutual exclusivity, "at least one of", "exactly one of"). Walk through every CREATE-time check and confirm it still holds after the partial update.

## [2026-04-30] — `enabled`/visibility checks must be re-applied at every consumption point, not just at list time
**Rule**: When a row has an `enabled` flag and the dashboard hides it from a `list_*` tool by filtering on `enabled = true`, the corresponding `load_*` / `get_*` / `invoke_*` consumer MUST re-check `enabled` after the lookup and refuse if disabled. Hiding from list does not protect against stale ids — the model may have remembered an id from earlier in the conversation, before the user toggled it off.
**Why**: Verifier flagged this on `load_skill`: `list_skills` filters `enabled=true`, but `load_skill(id)` does not, so a model with a stale id from earlier in the same conversation can still load a now-disabled skill. The same applies to MCP servers, prompt templates, scheduled jobs — anything with a soft-disable flag. Defense-in-depth at every consumer, not just the discovery surface.
**Apply when**: Any soft-disable flag (`enabled`, `active`, `archived = false`) on a table consumed by multiple tools or routes. Treat the flag as an authorization check that must run at every read path, not just the listing path.

## [2026-04-30] — `pnpm -F <pkg> typecheck` fails silently when no `typecheck` script exists
**Rule**: `pnpm -F @tele/server typecheck` exits with `ERR_PNPM_RECURSIVE_RUN_NO_SCRIPT` if the package has no `"typecheck"` script in its `package.json`. The error is easy to mistake for "no typecheck needed" or "passed". Always either add `"typecheck": "tsc --noEmit"` to every package's `package.json`, or fall back to `cd <pkg> && npx tsc --noEmit` and check the exit code explicitly.
**Why**: A plan step that says "run typecheck" can read as PASS when it actually never ran. The verifier on this task caught it only by re-running with `npx tsc --noEmit` directly. Repo hygiene fix: add the script to every package; CI fix: assert the script exists before running.
**Apply when**: Writing or following a plan step that invokes `pnpm -F` with a script name. Either confirm the script is defined first (grep `package.json`) or use the direct `npx tsc --noEmit` form. Same trap applies to `lint`, `test`, etc. — pnpm recursive scripts are silent on absence.

## [2026-04-30] — Seed rows with machine-specific paths are fine if idempotent and user-editable
**Rule**: It's acceptable to ship a migration with a hard-coded `INSERT ... ON CONFLICT DO NOTHING` containing a developer-local path (e.g. `/Users/foo/Downloads/grail-explorer/SKILL.md`) provided: (1) the seed is idempotent so re-runs on a clean DB don't error, (2) the dashboard exposes a way to delete or re-point the row, and (3) the path is documented as "edit me on a fresh install" in the plan. Don't gate the migration on the path's existence at boot.
**Why**: For a single-user tool, blocking the migration on a missing local file would break onboarding for any other developer. `ON CONFLICT DO NOTHING` plus an editable dashboard is the pragmatic middle ground — the seed is convenience, not correctness. The AI tool that consumes the path should fail gracefully ("path not readable: <path>") so the model surfaces the issue to the user.
**Apply when**: Seeding any developer-local resource (file paths, MCP server commands, local URLs) into a migration. Don't put hard-coded local paths into seeds for multi-tenant or production deployments — only for single-user dev tools where the user can fix the seed via the UI.

## [2026-05-02] — Return `{ handled, type }` (not just `boolean`) from interceptors so the caller can vary follow-up side effects
**Rule**: When an "intercept-or-fall-through" dispatcher (slash command router, middleware shim, command parser) sits in front of a pipeline that already fires its own side effects (reactions, telemetry, status updates), return a discriminated result like `{ handled: true, type: "ai_prompt" }` rather than a bare `true`. The caller uses `type` to decide whether to fire its own outer side effect or let the inner branch's side effect stand alone. Never return only `boolean` if any branch reuses the existing pipeline.
**Why**: For slash commands, the `ai_prompt` branch calls `generateAndReply`, which already fires `reaction_done` on success. If the router *also* fires `reaction_done` after seeing `handled: true`, the user gets two reactions on the same message — a visible bug. The discriminated return lets the router skip the outer reaction precisely for `ai_prompt` and fire it for `shell` / `message` (which use only `sendReply` and don't double-fire).
**Apply when**: Adding any pre-pipeline interceptor where some branches go through the existing pipeline and some don't. The signature `{ handled: boolean; type?: Variant }` is the minimum needed; richer payloads are fine if downstream needs them. Document which branches are "pass-through to inner side effects" vs "self-contained" in the dispatcher's doc comment.

## [2026-05-02] — Optional `opts` param with `??` fallback is the safe way to add per-call overrides without breaking callers
**Rule**: When you need to add a per-call override to an existing function (e.g. `systemInstructionOverride` on `generateAndReply`), append an optional `opts?: { ... }` parameter at the end and use `opts?.field ?? defaultExpression` at the use site. Do NOT change the existing parameter list, do NOT make `opts` required, and do NOT mutate any module-level state to thread the override through. Existing call sites continue to work with `opts` defaulting to `undefined`.
**Why**: Mutable module-level overrides (e.g. setting a `currentSystemPrompt` global before calling) leak across concurrent calls and create surprising action-at-a-distance. A required `opts` argument forces every existing caller to be updated in the same diff, blowing up the blast radius. The `?? default` pattern keeps the override scoped to one call and adds zero behavioral change for callers that don't pass it.
**Apply when**: Extending any existing function with a new "optional behavior tweak" used by exactly one new caller. The pattern is also the right shape for dependency injection in tests — pass `opts.clock`, `opts.now`, `opts.logger` for overrides without restructuring the signature.

## [2026-05-02] — Soft-rejecting parse failures (not just lookup failures) lets shorthand syntax coexist with real content
**Rule**: When a feature reserves a syntax prefix (e.g. `/` for slash commands, `@` for mentions, `:` for emoji), the dispatcher must distinguish "matches the prefix but fails to parse as our syntax" from "parses but no row found" — and BOTH should fall through to the default handler, not error. For slash commands, `/path/to/file` starts with `/` but does not match `^([a-z0-9_-]+)(?:\s+([\s\S]*))?$/i`; the dispatcher returns `{ handled: false }` so the AI receives `/path/to/file` as ordinary user text rather than getting a 400 or a "command not found" reply.
**Why**: Users casually type things like `/Users/foo/...`, `/etc/hosts`, or `/r/subreddit` in chat. Hard-rejecting any `/`-prefixed text as a command attempt would block normal conversation. The two-stage filter (regex parse → DB lookup → both fall through on miss) means the feature only activates on syntactically clean command names that also resolve to a configured row.
**Apply when**: Designing any prefix-triggered command/mention/macro system where the prefix character also occurs in normal content (paths, URLs, user handles). Treat the regex parse as a soft filter; never throw, never reply with an error — only intercept on a positive match all the way through.

## [2026-05-02] — `child_process.spawn(shell, ["-c", cmd])` with explicit timeout + SIGKILL is the correct shell-exec primitive
**Rule**: For server-side "run this user-authored shell command and capture output" features, use `spawn("zsh", ["-c", cmd])` (or `/bin/sh`) with a manual `setTimeout` + `child.kill("SIGKILL")` for hard timeouts, accumulate stdout/stderr in string buffers via `data` events, and resolve a `{ code, stdout, stderr, timedOut }` shape from the `close` event. Do NOT use `exec` / `execSync` (no streaming, no clean timeout) and do NOT use `execFile` (no shell expansion — defeats the point for user-authored commands). Always handle the `error` event in addition to `close` to catch spawn failures (e.g. zsh missing on the host).
**Why**: `exec` buffers everything in memory with a default 1MB cap and silently truncates beyond that; its `timeout` option sends SIGTERM, which a stuck process can ignore. `spawn` + `setTimeout` + SIGKILL guarantees the process dies on the deadline. The `code` may be `null` (process killed without exit) — coalesce to a sentinel like `-1` so the caller can detect it. Truncate the stdout/stderr at the *output* boundary (e.g. 3500 chars for a Telegram reply), not the buffer boundary, so you keep the tail of the output for short-but-useful commands and only clip the noisy ones.
**Apply when**: Implementing any "run a shell command from a config row" feature (slash commands, custom hooks, webhook actions). The `{ timedOut: true }` branch is the easy-to-forget one — make sure the reply explicitly says "Command timed out after Ns" rather than reporting an empty stdout as success. For untrusted input contexts, this whole pattern is unsafe — that's a different problem (sandboxing/allowlisting), not a different exec API.

## [2026-05-02] — `String.replaceAll(literal, value)` is fine for trusted templates; resist over-engineering to `{{handlebars}}`
**Rule**: For a single-token substitution like `{args}` in a user-authored template, `template.replaceAll("{args}", value ?? "")` is the right primitive. Don't reach for Handlebars, Mustache, or a hand-rolled tokenizer unless the feature needs conditionals, loops, or escaping. Document the limitation ("the literal string `{args}` is unsupported in output") in the dashboard help text and move on.
**Why**: A real template engine adds a dependency, a security surface (HTML escaping defaults vary), and a learning curve for the user — all to support features no one asked for. `replaceAll` is one line, has no escaping behavior to misconfigure, and the "you can't emit a literal `{args}`" trade-off is acceptable for a single-user tool. If the feature later needs `{user}` and `{date}` too, a 5-line `Object.entries` loop still beats pulling in Handlebars.
**Apply when**: Adding any single-or-few-token substitution in a config string the user authors. Default to `replaceAll` per token; only escalate to a real engine when you need conditionals or escaping. Same logic for env-var-style `${VAR}` substitution in non-shell contexts.

## [2026-05-02] — `as any` cast for SDK fields not in published types — anchor to a comment naming the SDK version
**Rule**: When a Google/OpenAI SDK accepts a field at the REST boundary that is not yet in the TypeScript types (e.g. `generationConfig.responseModalities` in `@google/generative-ai` v0.21 for the image-preview model), the right escape hatch is a localized `as any` cast on just the offending object — `generationConfig: { responseModalities: ["TEXT", "IMAGE"] } as any`. Always co-locate a one-line comment naming the SDK version and stating that the SDK forwards the field to the REST body unchanged. Do NOT widen the cast to the whole call, do NOT `// @ts-expect-error` (it'll flip the wrong way on SDK upgrades), and do NOT fork or re-declare the SDK's types.
**Why**: Vendor SDKs lag the REST surface by weeks, especially for preview features. A bare `as any` with no comment becomes a mystery in three months when an SDK upgrade tightens the type and someone wonders why we had to dodge it. The comment is the documentation. A `@ts-expect-error` is worse: when the SDK adds the field, the directive becomes the new error and breaks the build, but a plain `as any` keeps working silently — which is what you want for a forward-compatible cast.
**Apply when**: Any time you're calling a vendor SDK with a preview/beta field that is documented in the REST API but missing from the typed surface. Cast at the smallest possible scope (the single object literal, not the function call), comment with the SDK version, and ideally gate the field behind a config flag so a future operator can flip it without code changes if the SDK rejects it.

## [2026-05-02] — Gate fast-changing vendor model names behind an env var with a sane default
**Rule**: For vendor model identifiers that are known to drift (Gemini image-preview model has gone through `gemini-2.0-flash-exp-image-generation` → `gemini-2.0-flash-preview-image-generation` → `gemini-2.5-flash-image-preview` in months), expose the model name as an env-var with a default in the config schema (`GEMINI_IMAGE_MODEL: z.string().default("gemini-2.0-flash-preview-image-generation")`) and reference `config.GEMINI_IMAGE_MODEL` at the call site. Do NOT hard-code the string in the tool file.
**Why**: When the model is renamed or deprecated, the user can flip the env var and restart — no code change, no redeploy of a new build. Hard-coding forces a code change on every drift event. The default keeps the zero-config experience intact for the common case. Same pattern applies to embedding models, vision models, and any preview-tier name that the vendor hasn't committed to long-term.
**Apply when**: Adding any new vendor model integration where the model id is in a preview/experimental tier, or where the vendor has historically renamed models. Always pair the env var with a default; document the rename history in a comment near the schema entry if it's especially churny.

## [2026-05-02] — Temp-file write-then-send pattern: `try/finally` with `unlink().catch(() => {})`
**Rule**: For a tool that materializes binary data (image bytes from a model, downloaded blob) just long enough to hand off to a transport that needs a path (Telegram `sendFile`, ffmpeg, image-resize CLIs), the shape is: write to `os.tmpdir()` with a randomized filename (`Date.now()`-`randomBytes(4).toString("hex")`-`.ext`), wrap the transport call in `try { await transport(tmpPath); return success; } finally { await fs.unlink(tmpPath).catch(() => {}); }`. The `.catch(() => {})` on unlink is intentional — failures to clean up should never mask the real error from the transport.
**Why**: A naive `await fs.unlink(tmpPath)` after the transport will throw if the file is already gone (e.g. on EBADF, or if a previous cleanup ran), and an unhandled throw from `unlink` would replace the meaningful error from `sendFile` with a confusing "ENOENT on cleanup". The randomized filename avoids collisions if two concurrent tool calls race. `os.tmpdir()` is OS-managed so leaked files (process killed between write and unlink) get reaped on reboot — don't over-engineer a custom GC for a single-user dev tool.
**Apply when**: Any tool/handler that produces a blob and feeds it to a path-based downstream API. The `try/finally + swallow on unlink` shape is the minimum-surprise pattern; the cost of a leaked tmp file on a process crash is acceptable, the cost of a confusing stack trace on every cleanup hiccup is not.

## [2026-05-02] — Multimodal SDK responses: trust `mimeType` for extension, default safely on unknown
**Rule**: When extracting binary parts from a multimodal LLM response (`response.candidates[0].content.parts[].inlineData`), pick the file extension from `inlineData.mimeType` (`image/png` → `.png`, `image/jpeg` → `.jpg`, `image/webp` → `.webp`) and default to `.png` for any unknown or missing mime. Don't sniff magic bytes; don't enforce a single output format on the model. Telegram (and most consumers) renders all three correctly regardless of extension, so the default is forgiving.
**Why**: The model picks the format based on its own heuristics — forcing `.png` always works, but uses larger files for photographic content where JPEG would be 1/10th the size. Trusting `mimeType` keeps the file small and the content-type accurate. Defaulting to `.png` on unknown is safe because PNG is the universal "lossless image" format and Telegram doesn't actually look at the extension — but downstream tools (S3, browsers via Content-Disposition, image processors) do.
**Apply when**: Any code extracting `inlineData` from a Gemini/OpenAI multimodal response, or any handler that writes vendor-supplied bytes to disk and needs an extension. Don't add a magic-byte sniffer; the mime field is reliable when present, and the safe default covers the rest.

## [2026-05-02] — Prefer extending the existing `ai_prompt` slash branch with a system-prompt override over adding a new dispatcher branch
**Rule**: When adding a new slash command that should trigger AI behavior with a tweaked instruction (e.g. `/draw` should force a `generate_image` tool call), seed the row with `type='ai_prompt'` and put the directive in the `action` column as a system-prompt override. Do NOT add a new dispatcher branch (`type='draw'`, `type='generate'`) — the `ai_prompt` branch already passes `action` as `systemInstructionOverride` to `generateAndReply`, which is exactly the right hook. Each new slash command becomes one migration row, zero code changes to the dispatcher.
**Why**: Adding a new `type` value would require: (1) a CHECK constraint migration to widen the allowed types, (2) a new dispatcher branch, (3) new tests, (4) a new entry in the `{ handled, type }` discriminated return. Versus: one INSERT row that uses the existing `ai_prompt` plumbing. The system-prompt override is the universal "make the AI do this specific thing" lever — resist the urge to specialize.
**Apply when**: Any new slash command whose behavior is "have the AI do X with this instruction." Seed it as `ai_prompt` with the action being the override prompt. Only add a new `type` if the behavior genuinely cannot be expressed as "AI with this system override" — e.g. a pure shell-out, a webhook trigger, a synchronous DB query that must short-circuit the AI loop entirely.

## [2026-05-04] — Public webhook routing via auth-hook scope, not per-route allowlist
**Rule**: When adding a public (unauthenticated) HTTP endpoint to a Fastify app whose auth is enforced by a global `onRequest` hook, register the route OUTSIDE the hook's URL prefix scope (e.g. mount at `/webhook/:id` when the hook only fires on `/api/*`). Do NOT widen the hook to add a `PUBLIC_PATHS` allowlist; do NOT decorate the route with a "skip auth" flag. The path itself is the auth boundary.
**Why**: A `PUBLIC_PATHS` allowlist is an action-at-a-distance config that someone editing the route forgets to update. The path-prefix split is enforced by URL routing — impossible to mis-configure on the route side. The trade-off is the public route can't share the `/api/*` namespace, but for webhooks called by external services (Telegram, Stripe, GitHub) that's the right separation anyway. ALSO: add a top-of-file comment in the public route noting the dependency on the auth hook's scope, so a future operator widening the hook to `/*` gets a breadcrumb.
**Apply when**: Any new endpoint a third party calls without your auth cookie/header (webhooks, OAuth callbacks, health checks for external monitors). Mount outside the protected prefix; comment the dependency on the hook's scope at both ends (the route and the hook).

## [2026-05-04] — `timingSafeEqual` needs length-mismatch handling — wrap, don't try/catch
**Rule**: `crypto.timingSafeEqual(a, b)` THROWS `RangeError: Input buffers must have the same byte length` when buffers differ in size — it does not return `false`. Wrap calls in a helper like `safeEqualHeader(actual, expected)` that converts both inputs to Buffers, returns `false` on length mismatch BEFORE calling `timingSafeEqual`, and returns `timingSafeEqual(a, b)` otherwise. Do NOT try/catch the throw — pre-checking the length is faster, more readable, and avoids the perception that any other crypto error is being swallowed.
**Why**: A naive `timingSafeEqual(Buffer.from(headerVal), Buffer.from(expected))` crashes the request handler with an uncaught throw the moment an attacker (or a misconfigured client) sends a header of the wrong length — turning what should be a 401 into a 500. The pre-length-check is the cleanest fix; try/catch works but loses the chance to detect genuine crypto failures. The helper also normalizes the type — header values can arrive as `string | string[] | undefined` from Fastify.
**Apply when**: Any time you compare a user-supplied secret/token header against a stored value. The pattern is identical for HMAC signatures, webhook secrets, API keys passed in headers. Default to a small `safeEqualHeader` helper in a shared util file rather than inlining at each call site.

## [2026-05-04] — Webhook handlers: always 200 except for the ONE error code the sender uses to back off
**Rule**: For an HTTP webhook handler, return `200` for nearly every outcome — bad request body, unknown bot/resource id, disabled row, parse failure, internal exception during async work — and reserve non-200 codes for the one signal the sender actually acts on. For Telegram, that signal is `401` on `secret_token` mismatch (Telegram stops retrying); every other condition uses `200` with silent drop. Wrap the AI/business logic in `.catch(err => logger.error(...))` AFTER the 200 has been returned so a Gemini failure or DB hiccup does not leak into the HTTP response.
**Why**: Webhook senders (Telegram, Stripe, GitHub) retry on non-200 with exponential backoff. Returning `400` for "unknown bot id" makes Telegram retry the same broken update for hours, flooding logs and possibly racing with cleanup. Returning `500` on a transient downstream error (Gemini timeout) makes Telegram resend the same user message — duplicating the AI response and any side effects (sent messages, DB writes). The "always 200" rule sounds wrong from a REST hygiene standpoint, but webhooks have inverted semantics: you are the *consumer*, the sender is the *server*, and your job is to acknowledge receipt fast.
**Apply when**: Any inbound webhook handler. Map error responses to: 401 = "stop retrying, you're misconfigured" (the signal); 200 = literally everything else. Document the chosen non-200 codes inline so a future maintainer doesn't add a 500 path on instinct.

## [2026-05-04] — Fire-and-forget ACK before async work — `void` + `.catch`, never `await`
**Rule**: When an inbound message has a UI-acknowledgment requirement that the upstream service uses to clear a loading state (Telegram `callback_query.id` → `answerCallbackQuery`, Slack `response_url`, Discord interaction tokens), call the ACK with `void someApi(...).catch(err => logger.warn(...))` — explicit `void` prefix, `.catch` tail to swallow errors — and IMMEDIATELY proceed to the actual work. Do NOT `await` the ACK before the AI/handler logic runs.
**Why**: `await ack(); await ai();` serializes a network round-trip in front of the slow path, leaving the user staring at a spinner for hundreds of extra milliseconds. The ACK has no return value the handler needs and its failures are non-fatal (the spinner just stays a few extra seconds). The `void` prefix is load-bearing — without it, lint rules like `no-floating-promises` will flag the line and the next person "fixes" it by adding `await`, regressing the latency. The `.catch` is not optional — an unhandled promise rejection in a `void`-discarded promise will crash Node 15+.
**Apply when**: Any handler whose upstream protocol has a "clear the spinner" or "I got it" signal separate from the actual response. Always: `void ack().catch(log); doRealWork();`. If TS or lint complains, the lint config is the bug — add a comment, not an `await`.

## [2026-05-04] — Reuse the route/import path when replacing a feature; let the seam be the file contents
**Rule**: When deleting and replacing a feature that owns a sidebar entry, frontend route, page filename, and API path (e.g. swapping the persona-bot CRUD for a Telegram-bot CRUD that occupies the same "Bots" name), KEEP the existing identifiers — `/bots` route, `pages/Bots.tsx` filename, `/api/bots` REST path, `qk.bots` query key, `{ to: "/bots", label: "Bots" }` sidebar entry. Replace ONLY the file contents. The router (`App.tsx`), the sidebar (`Sidebar.tsx`), and the query-key barrel become zero-edit files.
**Why**: Editing the router and sidebar to point at a renamed file/route doubles the diff for no behavioral gain and makes the rollback bigger. Reusing the names also lets the dashboard's bookmark/URL stability survive the swap — a user who had `/bots` in browser history still lands on the same page, just with new contents. The trade-off is git blame on the page file shows a sudden total rewrite (good — that IS the change), and the internal server filenames keep a longer prefix (`telegramBots.ts`) to disambiguate from the deleted persona-bot file in version-control history.
**Apply when**: Any feature swap where the new feature occupies the same conceptual slot as the old one. Keep the user-facing names (URL, sidebar label, query key) identical; let the new behavior live entirely inside the file contents. Add longer disambiguating prefixes only on internal server filenames where git-blame archaeology matters.

## [2026-05-04] — Atomic delete-then-recreate phases: do NOT typecheck or commit between
**Rule**: When a multi-phase plan deletes a file in an early phase and recreates it (with new contents at the same path) in a later phase — e.g. Phase 0 deletes `pages/Bots.tsx` and Phase 7 writes a new `pages/Bots.tsx` — treat the delete-and-recreate phases as ONE atomic change set. Do NOT run `tsc --noEmit` between them, do NOT commit between them, do NOT push between them. The `App.tsx` import will be dangling for the duration of the gap. If parallelism is desired, do the recreate step IMMEDIATELY after the delete step so the import is never broken on disk.
**Why**: A typecheck run between Phase 0 and Phase 7 will fail loudly with "Cannot find module './pages/Bots'", which is technically correct but uselessly distracting — the plan acknowledged the gap and the recreate is queued. A commit between the two phases creates a build-broken commit in history, which (a) breaks `git bisect` going forward and (b) blocks anyone who checks out that SHA from running the dev server. The atomic-set rule sidesteps both. Document the atomicity in the plan ("treat Phases 0–7 as one change set; do not commit between them") so a future operator doesn't split them across PRs by accident.
**Apply when**: Any plan where a file is deleted in one step and a same-pathed replacement is written in another. The atomicity declaration belongs in the plan's Risks section AND in a comment on the deletion step itself, not just in the executor's head.

## [2026-05-04] — Webhook + CRUD route registration ordering vs catch-all plugins
**Rule**: When Fastify has `setNotFoundHandler` or a static-file plugin that catches `/*`, register every explicit route — including the public webhook — BEFORE those catch-all registrations. The webhook route at `/webhook/:botId` is no exception, even though it lives outside `/api/*`: the static plugin's catch-all will swallow it if registered first. Order in `api/index.ts` should be: explicit API routes → explicit public routes (webhooks) → static plugin → `setNotFoundHandler`.
**Why**: Fastify resolves routes in registration order for catch-all conflicts. A `staticPlugin` mounted at `/` with `wildcard: true` will eat any request that doesn't match an already-registered route. If you register webhook routes AFTER the static plugin, the webhook POST gets a 404 (or worse, a "not found" HTML page returned to Telegram, which retries forever because non-200). The fix is registration order, not a `before:` hook or a `priority` config — Fastify doesn't have those. Put a comment near both the static plugin and the webhook registration noting the ordering dependency.
**Apply when**: Any new public route in a Fastify app that uses static-file serving or a global not-found handler. Always register explicit routes first; never assume route specificity wins over registration order with catch-all plugins.

## [2026-05-04] — `BIGINT` vs JS Number: Telegram chat ids fit in 2^53, but persist as text-safe form anyway
**Rule**: Telegram chat ids are `int64` on the wire and exceed `2^31`, but currently fit comfortably in `2^53` (the JS-safe integer range). Persist them as `BIGINT` in Postgres and read them back as `string` (most pg drivers do this by default to preserve precision). When passing them BACK into a third-party API (GramJS `EntityLike`, Bot API `chat_id` JSON field), `Number(idStr)` is safe today — but prefer `String(idStr)` if the API accepts string form, since the cost of a future Telegram id-space expansion is "all your existing code is silently wrong" with no error.
**Why**: The Bot API actually accepts `chat_id` as either number OR string in JSON, and GramJS accepts string for `EntityLike`. Defaulting to string-form propagation through the stack is the forward-compatible choice for ~zero cost. The only place you MUST convert to number is when feeding the id into a JS `Map` key alongside numeric keys (don't mix), or comparing to a value the SDK has already coerced to number. The lesson is to prefer the string form whenever the API allows it — a one-character change today saves a debug session in 2030.
**Apply when**: Any code path that pulls a Telegram chat id from the DB and hands it to an external API. Default to string. Only call `Number(id)` at the explicit API boundary where the SDK demands a number, with a comment naming the SDK and version. Never use `BigInt(id)` for GramJS — the type rejection is documented in 2026-04-28's lesson; for Bot API the string-or-number flexibility makes BigInt unnecessary.

## [2026-05-04] — Webhook secret rotation: explicit delete + recreate, not in-place mutation
**Rule**: For per-row webhook secrets (or any rotateable credential stored alongside other config), document that rotation requires DELETING the row and recreating it — do NOT add a "rotate secret" PATCH endpoint or auto-regenerate on update. The secret is a one-time set at create-time; the dashboard's edit form does not expose it as editable.
**Why**: A "rotate" endpoint introduces a window where the new secret is registered with the third party (Telegram `setWebhook`) but the old secret is still in flight on a pending request — opening a brief auth-failure gap. More importantly, it's surface area no one asked for: rotation is rare (compromise, lost backup), and delete-and-recreate is a 30-second operator action that handles the registration cleanly. The simpler model also keeps the row's identity tied to its secret — useful for audit logs ("secret X was used during this incident" maps to a single row, not "row Y at time Z had secret X1, then X2").
**Apply when**: Any per-row credential where rotation frequency is "almost never" and the operator is a human, not an automated service. If rotation needs to be automated (e.g. compliance-mandated quarterly rotation), the model is different: PATCH endpoint, dual-secret window, callbacks accept both for a grace period. For the single-user dev-tool case, "delete + recreate" wins on simplicity.

## [2026-05-07] — `botAuthToken` MTProto client beats webhook for single-user bot integration
**Rule**: When adding a Telegram bot to an existing GramJS user-account app, run a SECOND `TelegramClient` in the same process authenticated via `client.start({ botAuthToken: () => Promise.resolve(token), onError })` instead of registering a webhook with `setWebhook`. The bot client speaks MTProto over the existing GramJS connection plumbing, listens via `NewMessage` and `CallbackQuery` event handlers, and requires zero public URL, zero TLS termination, zero webhook secret. Persist its session to a SEPARATE file (`data/bot-session.txt`) — never share with the user-account `data/session.txt`.
**Why**: The webhook path forces a `PUBLIC_URL`, a path-prefix carve-out for unauthenticated traffic, a `secret_token` rotation strategy, `setWebhook` lifecycle calls, and the "always 200 except 401" handler discipline (lessons-2026-05-04). All of that disappears when the bot client uses the same GramJS primitives the user-account already does — one extra module-level `TelegramClient`, one extra session file, one extra event handler pair. The trade-off is the bot is only online while the server process is alive (no message buffering during downtime), which for a single-user dev tool is a feature not a bug.
**Apply when**: Adding a Telegram bot to any app that already runs GramJS for a user account. Default to the second-client MTProto pattern; only escalate to webhook if you need (a) message delivery while the server is down, (b) horizontal scaling across multiple processes, or (c) you don't already have GramJS in the dependency tree. For greenfield bot-only apps without a user-account client, webhook is still the right default.

## [2026-05-07] — Two GramJS clients in one process: isolate session files, share nothing else
**Rule**: When running both a user-account `TelegramClient` and a bot `TelegramClient` in the same Node process, give each its own `StringSession` file (`data/session.txt` vs `data/bot-session.txt`), its own module-level singleton (`_client` in `client.ts`, `_botClient` in `botClient.ts`), and its own start/stop lifecycle. Do NOT try to share a `StringSession` instance, do NOT cross-import the singletons, and do NOT register handlers on one client that call into the other's state. Both clients can share the same `apiId`/`apiHash` from config — those are app credentials, not session state.
**Why**: GramJS `StringSession` holds the auth-key for a specific account. Sharing it between clients corrupts the session file (last-writer-wins on `client.session.save()`) and crashes one or both clients with auth errors. Separate singletons also let the bot client be stopped/restarted independently when the user toggles the dashboard `enabled` flag, without affecting the user-account client's connection. The shared `apiId`/`apiHash` is fine because Telegram allows one app's credentials to authenticate multiple sessions concurrently — that's the entire point of session strings.
**Apply when**: Any multi-account GramJS setup in one process — bot + user, multiple users, user + multiple bots. The pattern generalizes: one module per client, one session file per client, one lifecycle pair (start/stop) per client. If the count exceeds ~3, consider a `Map<accountId, TelegramClient>` registry instead of N singleton modules — but for the bot-plus-user case, two singletons keep the import graph trivially readable.

## [2026-05-07] — `chat_type='bot'` + composite unique `(tg_chat_id, chat_type)` for dual-channel chats
**Rule**: When the same Telegram user can talk to BOTH the user-account client AND a bot client in the same app, the `chats` table needs to allow two rows for the same `tg_chat_id` — one for each channel. Drop the single-column `UNIQUE(tg_chat_id)` constraint, add a composite `CREATE UNIQUE INDEX chats_tg_chat_id_chat_type_key ON chats(tg_chat_id, chat_type)`, and widen the CHECK constraint to include `'bot'`. The `upsertChat` query becomes `ON CONFLICT (tg_chat_id, chat_type) DO UPDATE`.
**Why**: A user with Telegram id `123` who DMs your user-account is a "private" chat; the same user `123` who also messages your bot is a separate logical conversation — different message history, different unread count, possibly different display name (their bot-facing name). Forcing them into one row collapses the two histories together and makes the dashboard show a confusing mix. The composite unique index is the minimum schema change that preserves both rows; the CHECK widening is mandatory because the old constraint silently rejects the bot-channel insert at runtime with no useful error.
**Apply when**: Any chat-storage schema where one external user identity can flow through multiple ingress channels (bot + user, multiple bots, bot + email-bridge). Default to `(external_id, channel_type)` as the composite uniqueness key; never rely on `external_id` alone if multiple channels exist or might be added.

## [2026-05-07] — Singleton config tables via `CREATE UNIQUE INDEX ON ((TRUE))`
**Rule**: For a config table that should hold AT MOST one row (e.g. `telegram_bot_config`, app-wide settings, single-tenant flags), enforce the singleton at the schema level with `CREATE UNIQUE INDEX <name>_singleton ON <table>((TRUE))`. The `(TRUE)` expression evaluates to the same value for every row, so the unique index allows zero or one row but rejects the second insert with a `23505` violation. Pair with a repo `setX(patch)` function that does INSERT-or-UPDATE based on whether `getX()` returned a row.
**Why**: The alternative — a CHECK constraint like `id = 1` plus enforcing `id=1` in every INSERT — leaks the magic constant through every consumer and breaks the moment someone forgets to set it. The `((TRUE))` trick is a documented Postgres idiom that lives entirely in the schema, requires zero discipline from consumers, and produces a normal unique-violation error path the route layer already handles (lessons-2026-04-30). It also leaves the row's `id` as a normal UUID, so audit logs and FKs work without special-casing.
**Apply when**: Any table whose semantic cardinality is "exactly zero or one" — global config, singleton secrets, per-app feature toggles. Don't reach for it when cardinality is "exactly one" (that's a CHECK constraint plus a seed migration); the index allows zero rows on purpose so the app can boot before config is set.

## [2026-05-07] — `ReplyAdapter` interface lets one AI loop serve multiple egress channels without forking
**Rule**: When the same `generateAndReply(chat, text, opts)` function needs to send replies through different transports (user-account `client.sendMessage` vs bot `client.sendMessage` vs future webhook), introduce a small `ReplyAdapter { sendText, persistOutbound }` interface and accept it via `opts.replyAdapter`. The default branch (no adapter) keeps the existing user-account behavior including reactions and prefix prepending; the adapter branch sends raw text and persists via the adapter's hooks. Do NOT fork the AI loop; do NOT add an `if (isBot)` branch at every send site — funnel both through the adapter.
**Why**: Forking `generateAndReply` into `generateAndReplyForBot` duplicates the tool-loop, the system-prompt assembly, the error handling, and the message-persist logic — every future change has to be made twice. The adapter pattern keeps one source of truth for the AI loop and isolates the transport differences (which buttons are supported, which reactions to skip, which prefix to prepend) into the adapter implementation. The `opts?.replyAdapter ?? defaultBehavior` shape (lessons-2026-05-02) means existing call sites are zero-edit.
**Apply when**: Any AI/orchestration loop that currently has one egress transport baked in and needs to gain a second. The `{ sendText, persistOutbound }` minimal interface is enough for most chat transports; add `sendImage`/`sendFile`/`react` only as the second adapter actually needs them. Default: empty interface, grow only on demand.

## [2026-05-07] — CallbackQuery → synthetic `[Button: <data>]` user message re-enters AI loop
**Rule**: When a Telegram bot inline button is pressed, the `CallbackQuery` event handler should: (1) fire-and-forget `void event.answer().catch(log)` to clear the spinner (lessons-2026-05-04), (2) `insertMessage` with text `[Button: <callback_data>]` and `direction='in'` as if the user had typed it, (3) call `generateAndReply` with the same opts as a regular incoming message. Do NOT add a separate code path that handles button presses with bespoke logic — re-entering the AI loop with a synthetic message keeps the conversation history complete and lets the model react to the button as a normal turn.
**Why**: A separate "button handler" branch fragments the conversation history (the AI sees the button-driven response but no record of WHICH button was pressed), forces every tool to know about both real-message and synthetic-button ingress, and makes the chat dashboard show a mysterious bot reply with no preceding user turn. The synthetic `[Button: X]` message is the same pattern as `[Image: ...]` and `[Poll: ...]` already established in the codebase (lessons-2026-04-29) — uniform treatment of non-text turns as bracketed sentinel text. The model handles them naturally because the system prompt explains the convention.
**Apply when**: Any UI element whose interaction needs to feed back into the AI loop (inline buttons, reactions, slash menu picks, voice transcripts). Convert to a synthetic `[Type: payload]` user message; let the AI loop handle it like any other turn. The convention scales: every new ingress type adds a new bracket prefix and zero new code paths.

## [2026-05-07] — Resolve global tool dependencies at INVOCATION time, not registration time
**Rule**: When a tool definition needs access to a singleton client/connection that may be reconnected during the bot's lifetime (`getBotClient()`, `getDbConnection()`, `getCurrentUser()`), call the resolver INSIDE the tool's handler — not at `buildTools(...)` time. Captured-at-registration references go stale the moment the singleton is replaced; lazy resolution always sees the current value. Cost: one extra function call per tool invocation. Benefit: tool keeps working across `stopBotClient() → startBotClient()` cycles without rebuilding the tool registry.
**Why**: When the dashboard toggles the bot off and back on, `botClient.ts` calls `client.disconnect()` and replaces `_botClient` with a new instance. Tools that captured the old `client` reference at `buildTools` time will silently send messages on a disconnected client (or throw "client not connected"). Re-resolving via `getBotClient()` at invocation time picks up the new client. Same logic applies to any singleton that has a "rebuild" path — config-driven DB pools, session managers, feature-flag clients.
**Apply when**: Authoring tools that depend on a globally-managed connection or service singleton. The lazy-resolve pattern is the default; the early-resolve pattern is fine ONLY if the singleton is provably immutable for the process lifetime (rare). Document the lazy resolution in the tool's source with a one-line comment so a future "optimization" doesn't hoist the lookup out of the handler.

## [2026-05-07] — Boot-time bot start failures must be non-fatal — log and proceed
**Rule**: When the server boots and reads `telegram_bot_config` to auto-start the bot client, wrap `startBotClient(token)` in `.catch(err => logger.error(...))`. Do NOT let a bot-auth failure (revoked token, network blip during MTProto handshake) crash the process. The dashboard MUST stay reachable so the operator can fix or clear the config; a crash loop turns a recoverable misconfiguration into a "console-into-the-server" emergency.
**Why**: The bot is one feature among many in this app; the dashboard, the user-account client, the AI loop, and the slash commands all work fine without it. Crashing the whole process on bot-start failure inverts the priority — a single peripheral feature takes down the operator's ONLY recovery surface. The error log is enough signal for the operator to notice, and the dashboard's `/bots` page lets them fix the token, toggle off, or delete the config without restarting. The same logic applies to any optional feature initialized at boot from config — log loudly, fail soft, keep the recovery surface alive.
**Apply when**: Any boot-time initialization of an optional service (bot clients, MCP servers, scheduled-job runners, integrations) where misconfiguration is recoverable via the dashboard. Wrap the start call in `.catch(log)` and continue boot. Reserve hard crashes for the irrecoverable subset: missing required env vars, DB unreachable, migrations failed.

## [2026-05-07] — `prefix` and `sendReaction` belong inside the user-account branch, not the shared loop
**Rule**: When extending an AI loop to support a second egress channel via `ReplyAdapter`, AUDIT every side-effect that currently runs unconditionally — message prefixes (`prefix + reply`), upstream reactions (`sendReaction(...)`), read-receipts, typing indicators — and gate each on `!opts?.replyAdapter`. The bot path almost always wants none of them: bots don't typically prefix every reply, can't react with the user-account's emoji set, and don't have the same "typing" semantics. Default behavior stays unchanged for existing user-account callers; bot path gets a clean send-only experience.
**Why**: Reactions sent through a bot client may use a different emoji set or fail outright (some emoji are user-only). Prefixes that make sense for the user-account ("[bot]: " or similar disambiguators) are noise on a dedicated bot channel where every message is from the bot by definition. Failing to gate these creates subtle UX bugs (double-prefixed bot replies, reaction errors in logs) that are easy to ship and hard to notice in QA. The audit is a one-time scan of the responder; the gates add ~3 lines.
**Apply when**: Any time you split a single-channel AI loop into multi-channel via an adapter pattern. Walk through every side effect in the loop, ask "does this make sense for channel B?", and gate accordingly. Document the gates inline with a short comment ("bot channel skips reactions; user-account-only feature") so a future refactor doesn't accidentally re-enable them.

## [2026-05-08] — Default-block via DB column DEFAULT flip, not via app-layer INSERT change
**Rule**: When the requirement is "all NEW rows of table T should default to state S, but existing rows are untouched", express it as a single `ALTER TABLE T ALTER COLUMN c SET DEFAULT S` migration — NOT as an application-code change to the INSERT path. This works ONLY if the existing `upsert*` helper deliberately omits column `c` from its INSERT column list (so the DB default applies on first INSERT) and from the `ON CONFLICT DO UPDATE` SET list (so existing rows preserve their value). Verify both halves before shipping the migration.
**Why**: The schema-only change is atomic, idempotent, retroactively safe (no `UPDATE` needed), and requires zero application logic. The pre-condition — `upsertChat` not touching `is_blocked` in either INSERT columns or the conflict-update — is load-bearing: if a future refactor adds `is_blocked: false` to the INSERT defaults "for clarity", the DB default is silently overridden and new chats arrive UNblocked. Pin this coupling with a top-of-migration comment naming the upsert function and the columns it deliberately omits, so a future maintainer reading the migration sees the dependency. Same pattern works for any "default state for new rows" feature (default-archive, default-pending, default-private).
**Apply when**: Any "new rows default to X" requirement where existing rows must NOT be retroactively updated. Always check the upsert/insert path doesn't override the default; document the coupling in the migration comment AND in the upsert function with a one-line "do not add column X here — DB default applies" reminder.

## [2026-05-08] — Silent-drop on auth-style gate to prevent enumeration; reply only on success
**Rule**: When a feature gates access on a user-supplied secret/name/token (e.g. `/unblock <ai_username>`, password-protected channel, invite-code redemption), the wrong-value branch must silently drop — no reply, no error message, no "wrong code" feedback — while the correct-value branch sends a confirmation. The inbound attempt is still persisted to the dashboard so the operator can see attempts, but the sender learns nothing about whether the gate exists or what the right value would be.
**Why**: Replying "wrong username" or "access denied" lets an attacker iterate over guesses with feedback ("the gate exists; let me try `bot`, `assistant`, `ai`..."); silent drop forces them to operate blind, dramatically raising the cost of brute force. The honest-user UX cost is real (a confused human might type `/unblock bot` and see nothing) but is mitigated by communicating the right username out-of-band (via the bot's profile bio, a `/start` reply, or operator instructions). The asymmetric "loud on success, silent on failure" is the standard pattern for auth gates and should be the default unless the threat model explicitly tolerates enumeration.
**Apply when**: Any user-supplied-secret gate (passwords, invite codes, magic words, named slash commands). Always: success → confirmation reply + state change; failure → silent drop + persist the attempt for audit. Log at info-level (`"blocked, dropping unblock attempt"`) not error-level so the dashboard isn't noisy. Same pattern for HMAC verification on webhooks: 401 only when the SENDER needs the signal to back off; otherwise drop.

## [2026-05-08] — Pre-pipeline gate ordering: anti-loop > authz-bypass > authz-deny
**Rule**: When a request handler has multiple pre-pipeline checks — anti-self-loop guards (skip own outbound echoes), authz-bypass commands (`/unblock` to flip the gate), and the actual authz-deny (block check) — the ordering is: (1) anti-loop FIRST so we don't process our own outbound, (2) authz-bypass parser SECOND so the unblock command can run even on a currently-blocked chat, (3) authz-deny THIRD so it only applies when the message wasn't an unblock command. Inserting the bypass after the deny means a blocked chat can NEVER unblock itself; inserting it before the anti-loop means the bot's own echo of `/unblock x` would re-trigger the bypass.
**Why**: The order is a load-bearing invariant: each check filters out a class of messages from reaching the next. A naive implementation puts the new bypass at the bottom near the AI call ("smallest blast radius") and finds the gate is unbypassable. Document the ordering with a numbered comment block at the top of the dispatcher (`// 1) skip self-echoes 2) try unblock 3) check block 4) AI`) so future additions know where to slot in. The same ordering applies to slash-command dispatchers, mute/unmute pairs, and any "command that affects whether the next step runs" feature.
**Apply when**: Adding any pre-pipeline check to a router/dispatcher that already has gates. Identify which existing checks the new one must come BEFORE (because it counters them) vs AFTER (because it depends on them passing). Anti-loop almost always stays first; authz-bypass goes between anti-loop and authz-deny; logging/metrics typically come last.

## [2026-05-08] — Pure parser + caller-decides-side-effects keeps gate logic testable across channels
**Rule**: For a command-parser used by multiple ingress paths (user-account router AND bot event handler), the parser MUST be a pure function returning a discriminated result (`{ matched: boolean; correct: boolean }`) and the CALLER decides what to do — emit events, send confirmations, update DB, log. Do NOT bake `setChatBlocked()` or `eventBus.emit()` into the parser. Each caller can vary its side effects (user-account uses `sendReply` with `source='manual'`; bot uses `getBotClient().sendMessage` + `insertMessage`) without forking the parser.
**Why**: A parser that does its own DB writes ties every consumer to the same side-effect order, the same event payload, the same confirmation text. The user-account and bot paths legitimately differ — user-account has `sendReply` infrastructure for source-tagging; bot has the bot client and a different persist path. Returning the discriminated result and having both callers handle the three cases (`!matched`, `matched && correct`, `matched && !correct`) keeps one source of truth for the parsing rule and one source of truth per channel for the side effects. Also makes the parser unit-testable with no mocks.
**Apply when**: Any cross-channel command parser (slash commands, mention triggers, hashtag handlers) where multiple ingress paths need to dispatch the same syntax. Always: parser returns a result; caller acts. Mirror the discriminated-return shape from lessons-2026-05-02 (`{ handled, type }`) — both serve the same goal of "parser informs, caller decides".

## [2026-05-08] — Async UI prompt round-trip via DB-token + synthetic re-entry message
**Rule**: When the AI needs to ask the user a discrete question and resume on their answer (inline keyboards, polls, choice prompts), do NOT block the AI loop waiting for the response. Instead: (1) the tool issues the prompt and returns IMMEDIATELY with `{ ok, sent }`; (2) the AI's turn ends; (3) the question's lookup state lives in a DB table (`pending_choices` here) keyed by a short token embedded in the callback payload (`c:<token>:<idx>`); (4) the user's tap fires a callback handler that atomically claims the token via `UPDATE ... SET consumed_at = now() WHERE token=$1 AND consumed_at IS NULL AND expires_at > now() RETURNING ...`; (5) the handler inserts a synthetic `[Choice: <label>]` user message into the SOURCE chat (where the AI originally asked) and re-enters the AI loop via the normal `generateAndReply` path. The AI sees the answer as a fresh turn, just like any other user message.
**Why**: A "wait for user input" tool that blocks the AI's runToolLoop holds the LLM API connection open for hours, breaks streaming, and leaks resources. The token+re-entry pattern decouples the issue from the answer cleanly: the AI's process can crash, restart, scale across machines — the token in Postgres survives. The synthetic `[Choice: …]` reuses the existing `[Button: …]`, `[Image: …]`, `[Poll: …]` bracket convention so dashboard rendering, conversation history, and AI context all stay uniform with no special cases. Source-chat-id stored in the token row matters: future versions might deliver the prompt to a different channel than the source (e.g. a dedicated approvals channel), but the AI continues the conversation where it started.
**Apply when**: Any AI-tool that needs a structured user response to drive the next turn — choice prompts, approval gates, scheduling pickers, multi-step forms. Default to fire-and-resume via DB token + synthetic message; reserve "blocking tool" semantics for fast deterministic computations (calculator, lookup, format conversion). The token TTL (24h here) protects against stale taps; the `consumed_at` column makes the claim idempotent.

## [2026-05-08] — Reserved callback_data prefix for system-issued buttons coexisting with AI-authored freeform buttons
**Rule**: When a callback handler must distinguish system-issued buttons (where the system owns the round-trip semantics) from AI-authored freeform buttons (where the AI invents the callback_data), reserve a short prefix for the system's format (e.g. `c:<token>:<idx>` for `ask_user_choice`) and BRANCH on the prefix BEFORE the freeform fallback. The decode is a strict regex (`^c:([A-Za-z0-9_-]{1,32}):(\d+)$`) — on miss OR on token-lookup-fail, fall through to the freeform `[Button: <data>]` synthetic-message path. NEVER fall through after a regex match — a stale system token tap should silently drop, not inject `c:expired:0` into the AI's history as a freeform button.
**Why**: Two concurrent button systems share the same Telegram callback channel — there is exactly one `CallbackQuery` event type. The prefix is the only at-rest signal that lets the handler route correctly. Strict regex (not just `startsWith('c:')`) prevents an AI-authored freeform `c:approve:42` from accidentally hitting the system path; the freeform tool's description should ALSO warn the model to avoid the reserved prefix, but the regex strictness is the primary defense. The asymmetric "fall through on prefix-miss but DROP on prefix-match-with-bad-token" is load-bearing: a matched-but-stale token represents a UI element the system once owned but no longer does, and treating it as freeform would create a phantom AI turn from a user's accidental re-tap.
**Apply when**: Any time a single callback/event channel is shared between system-managed and user/AI-managed payloads. Reserve a prefix for the system; document it in the freeform tool's description; enforce strict regex matching in the dispatcher; drop silently on match-but-stale rather than falling through. Same pattern works for any shared event channel — slash commands vs typed text, mention syntax vs URL paths, etc.

## [2026-05-08] — "End your turn" steering needs both the tool description AND the system prompt
**Rule**: When a tool has fire-and-resume semantics (returns immediately, model must NOT continue generating in the same turn), put the "END YOUR TURN" directive in BOTH the tool's description string AND a Guidelines bullet in the system prompt. The tool description alone is insufficient — the model sees it only when calling the tool, but the inhibitory instruction needs to also live in the persistent system context so the model plans the turn correctly from the start. Conversely, the system-prompt bullet alone is insufficient — by the time the model is generating after the tool response, the system prompt's instruction may not be foregrounded; the tool description (echoed in tool definitions every turn) reinforces it.
**Why**: Gemini's behavior on tool-call sequencing depends on both the structured tool definition and the natural-language instructions. A tool description saying "returns immediately" without "END YOUR TURN" can be read as "this is fast, you can immediately do more" — exactly the wrong inference. A system prompt bullet without a corresponding tool-description directive relies on the model carrying global context into a tool-specific decision, which is unreliable across model versions. The doubled instruction is cheap (two sentences) and the cost of getting it wrong is duplicate AI replies — the buttons appear AND the model emits trailing "I asked you to pick…" text, confusing the user.
**Apply when**: Authoring any AI tool with fire-and-resume / async-prompt semantics. Always pair the directive in two places. If observed empirically that the model still emits trailing chatter after such a tool, the deeper fix is post-tool early-exit (drop trailing text in the responder when the last tool was a fire-and-resume tool) — but the doubled instruction handles 90% of cases at zero code cost.

## [2026-05-08] — Source chat in token row, NOT callback chat — for cross-channel question/answer flows
**Rule**: When a system-issued button can be DELIVERED to a different chat than the one where the AI was responding (e.g. AI is talking in user-account group X, but the prompt is delivered via the bot to that same group X — same chat today, but design for divergence), persist BOTH `source_chat_id` (where the AI was responding) AND `delivery_chat_id` (where the button was sent) in the token row. The callback handler resolves the SOURCE chat from the token (via `getChatById(claimed.source_chat_id)`), inserts the synthetic `[Choice: <label>]` message into THAT chat, and calls `generateAndReply` on the source chat — NOT the callback's sender chat. In v1 source and delivery are equal, but the columns are separated for future divergence (e.g. delivering to a dedicated "decisions" channel while the conversation continues in the original source).
**Why**: The "obvious" implementation pulls chat info from the callback event itself (`event.chatId`), which works as long as the prompt was delivered to the same chat the conversation is in. The moment that assumption breaks — operator wants approvals routed to an admin channel, multi-channel orchestration, etc. — the conversation history fragments into the wrong chat. Storing source_chat_id in the token row makes the conversation-continuity rule explicit at the schema level: "this token, when claimed, belongs to THIS conversation regardless of where the button physically lives." The cost is one extra column and one extra `getChatById` lookup per claim — trivial. The benefit is the cross-channel design space stays open without a future migration.
**Apply when**: Any UI element (button, form, poll) that is issued by an AI in one logical conversation and rendered in a possibly-different physical channel. Persist source AND delivery separately even if they're equal in v1; resolve the source at claim time; insert any synthetic re-entry message into the source. Same pattern applies to OAuth callback flows, email-link confirmations, SMS reply codes — wherever the "where to resume" answer differs from the "where the click happened" answer.

## [2026-05-08] — `PASS-by-inspection` is a legitimate verifier verdict — name the unrunnable boundary explicitly
**Rule**: When verification of a behavior requires a live external service (Telegram round-trip, real LLM call, third-party webhook delivery) that the verifier cannot run in-environment, the right verdict for that criterion is `PASS-by-inspection` — meaning the code path was traced against the plan and each branch maps correctly, but the live behavior was not directly observed. Name the unrunnable criteria in a "Caveats / not-runnable in this environment" section AT THE BOTTOM of the verify report so the operator knows what to smoke-test before declaring production-ready. Do NOT mark as PASS without qualification (overstates confidence) and do NOT mark as FAIL (the code is correct, the test infrastructure is the gap).
**Why**: A binary PASS/FAIL forces the verifier to either lie (PASS for unverified live paths) or block on infrastructure they don't control (FAIL for paths that look correct). The `PASS-by-inspection` middle verdict is honest: it says "I read the code, it matches the plan, I cannot run the integration test." The Caveats section pairs it with an actionable handoff — the operator knows EXACTLY which criteria need a manual smoke test. Without naming the boundary explicitly, a future reader of the verify report has no idea which "PASS" was observed live and which was inferred from inspection. The discipline also prevents scope creep: the verifier doesn't try to mock the entire Telegram + Gemini stack just to convert a PASS-by-inspection into a PASS — they document the limitation and move on.
**Apply when**: Any verification task where some acceptance criteria require live external integrations the verifier can't reproduce. Use the trichotomy `PASS / PASS-by-inspection / FAIL`; pair PASS-by-inspection items with a Caveats section listing what the operator must smoke-test. Pattern generalizes to any pre-prod gate: code review approval ("approved subject to smoke test of the OAuth flow"), security review ("approved; runtime verification of CSP requires staging deploy"), performance review ("approved; load test requires real traffic profile").

## [2026-05-08] — Logger wrap is the zero-edit hook for cross-cutting metrics side-effects
**Rule**: When adding cross-cutting instrumentation (error counters, request counters, structured event emission) that should fire at every existing `logger.error`/`logger.warn` site, modify the LOGGER itself — wrap the existing `error`/`warn` body so it ALSO calls the side-effect AFTER the original console/file write — instead of editing every call site. The added call MUST be wrapped in a `try { sideEffect(...) } catch {}` so a bug in the side-effect can never crash the logger. Combine with TARGETED `incCounter` calls at hot paths that need stable named buckets (`router.message_received`, `gemini.call.ok`) — the logger wrap covers the long tail; the named counters cover the dashboard-prominent buckets.
**Why**: Hand-instrumenting every `logger.error` site is N edits with high miss rate (some files get touched, some don't); the dashboard then shows a partial picture and operators don't know which gaps are "no errors" vs "instrumentation missing". The logger wrap captures EVERY `logger.error` automatically — including future ones added after the instrumentation lands. The try/catch is load-bearing: the file's existing "logger must never throw" comment means any side-effect that bubbles an exception out is a regression of that invariant. The hybrid (logger-wrap + targeted named counters) gives both completeness and naming discipline without forcing every call site to opt in.
**Apply when**: Adding any "fire at every existing log call" instrumentation — error metrics, structured-log shipping, OpenTelemetry span attributes derived from log fields, audit trails. Always: wrap inside try/catch, document the invariant inline (`"metrics must never throw out of the logger"`), and confirm no circular import (the side-effect module must not transitively import the logger).

## [2026-05-08] — Metrics module must use `console.*`, not `logger.*`, to keep the import graph acyclic
**Rule**: A metrics/instrumentation module that the logger calls into (per the logger-wrap pattern) MUST NOT import the project logger — it would create a cycle (`logger.ts → metrics.ts → logger.ts`). Use bare `console.warn`/`console.log` for the rare diagnostics the metrics layer itself needs to emit (config-missing notices, write-failure callbacks). Document the constraint at the top of both files: in `logger.ts` ("metrics module imported below; that module must not import this one back") and in `metrics.ts`/`influx.ts` ("uses console.* instead of project logger to avoid circular import with logger.ts").
**Why**: The cycle is silent at compile time (TS allows it) but fragile at runtime — module-init order can leave one of the two modules with `undefined` exports during the brief window the cycle resolves, producing bizarre "X is not a function" errors only at startup. The `console.*` escape hatch costs nothing (the noise is bounded — config notices fire once at boot; write callbacks are infrequent) and keeps the import graph a DAG. Same pattern applies to any "logger calls into X, X needs to log too" situation: X uses `console`, not the logger.
**Apply when**: Designing any module that the logger transitively depends on (metrics, telemetry buffers, panic handlers). Identify the cycle BEFORE writing the code; pick the module to break the cycle (usually the lower-level one); use `console.*` there with a documenting comment.

## [2026-05-08] — Synchronous coherent snapshot before any `await` for in-process metric capture
**Rule**: When persisting an in-process metrics state (Maps of counters/gauges/histograms) to an external store, capture the entire snapshot SYNCHRONOUSLY in one JS tick BEFORE the first `await` — `const snap = { counters: Object.fromEntries(counters), gauges: Object.fromEntries(gauges), histograms: getHistograms(), errors: [...errorBuckets.values()] }` — then build serialization payloads from `snap`. JavaScript is single-threaded, so any concurrent `incCounter` calls that arrive during the subsequent `await writePoints(...)` land in the NEXT snapshot, not a half-written current one. NEVER iterate a Map and `await` per item — the second iteration's values will not match the first's.
**Why**: Iterating `for (const [name, value] of counters) { await write(...) }` exposes the writer to concurrent mutation between iterations: a counter incremented after iteration N starts but before iteration N+1 reads it produces a corrupt snapshot where some counters reflect a later state than others. The `Object.fromEntries(map)` capture takes microseconds (single tick, no yield) and produces a frozen-in-time view. Persistence proceeds against the snapshot; the live maps keep ticking. No locking, no copy-on-write, no atomic primitives needed — JS's single-threaded model is the locking primitive when used correctly.
**Apply when**: Any "drain in-process state to external store" code — metrics persistence, log batching, event buffer flushing, queue draining. Always: synchronous capture FIRST, then serialize/await against the captured snapshot. The cost (one extra map allocation) is trivial; the alternative (subtle inter-iteration drift bugs that only show up under load) is not.

## [2026-05-08] — Fire-and-forget batch writers: empty batch early-return is correct, document the `lastWriteAt=null` consequence
**Rule**: For a fire-and-forget batch writer (`writePoints(points: Point[])`), early-returning when `points.length === 0` is the right behavior — there is literally nothing to write, no point in calling the underlying SDK. The consequence is that "last write timestamp" tracking (e.g. `lastWriteAt` updated by a `writeSuccess` callback) stays NULL when the snapshot is empty (no counters incremented yet). The dashboard tile reading `server.snapshot_at` will show "never persisted" even though the persistence interval IS running and would write if there was data. Document this in BOTH the metrics persistence function AND in the verify-results doc — "empty snapshots are no-ops by design; `snapshot_at` stays null until first non-empty snapshot."
**Why**: The "always write a heartbeat" alternative (push at least one dummy point per snapshot to update `lastWriteAt`) pollutes the time-series with synthetic data and confuses any downstream Flux query that treats the bucket as "real metrics only". The empty-snapshot no-op preserves the bucket's semantic cleanliness at the cost of one observable quirk: a freshly-booted server with no traffic shows `snapshot_at: null` even after the interval has fired. Operators reading the dashboard in this state get confused ("is persistence broken?"); the inline doc / verify note prevents the wasted debugging cycle. Same pattern applies to any "drain queue" interval that legitimately has nothing to drain on most ticks.
**Apply when**: Implementing any periodic-flush writer where the batch size can legitimately be zero. Choose: (a) early-return + document the observable consequence (preferred), or (b) emit a synthetic heartbeat point + accept the bucket pollution. Default to (a); reserve (b) for cases where downstream consumers REQUIRE a "still alive" signal (heartbeat-driven liveness checks).

## [2026-05-08] — Restore counters/gauges/errors on boot, but never restore histogram rings
**Rule**: When persisting in-memory metrics to an external store and restoring on boot, restore COUNTERS (monotonic totals — losing them resets the operator's long-running count), GAUGES (last-known scalar — useful to show "subscribers: 3" immediately on boot rather than 0), and ERROR BUCKETS (the operator's "what was failing" view should survive restart). Do NOT restore HISTOGRAM RINGS — they're a sliding window of recent samples; restoring stale samples produces misleading p95/p99 values that mix yesterday's data with today's. Histograms start fresh; the SUMMARIES are still queryable from the time-series store for charts.
**Why**: Counters and gauges are summary statistics where the historical value IS the truth (total messages received since boot, current subscriber count). Histograms are sample distributions where freshness IS the value — a p95 latency from 30 minutes ago combined with the last 5 minutes of new samples produces a meaningless mixed distribution. The persistence layer can still write histogram summaries (count, p50, p95, p99, max, mean) per snapshot for time-series CHARTS, but the live ring buffer must not be reseeded from those summaries. Document the asymmetry inline (`// histograms intentionally start fresh; rings would mix old and new samples`).
**Apply when**: Designing the boot-restore path for any in-memory metric system. Categorize each metric type: monotonic totals → restore; current scalars → restore; sliding-window samples → start fresh. Document the rule in the restore function's docstring; don't let a future "restore everything for completeness" refactor smash the asymmetry.

## [2026-05-08] — Single Flux composite query restores multiple measurements; iterate and switch on `_measurement`
**Rule**: When restoring multiple metric kinds (counters, gauges, errors) from a time-series store at boot, run ONE composite Flux query with an OR-filter across measurements — `from(bucket) |> range(start: -1d) |> filter(fn: (r) => r._measurement == "tele_counter" or r._measurement == "tele_gauge" or r._measurement == "tele_error") |> last()` — and SWITCH on `_measurement` while iterating the row stream to dispatch each row to its target Map. Do NOT issue three separate queries; do NOT issue one query per metric.
**Why**: Each Flux query is a network round-trip and a server-side compile + execute. Three separate queries triple the boot-time restore latency for no benefit; one query per metric is pathological at hundreds of metrics. The composite query lets InfluxDB's TSI index do the work in a single scan; the client-side switch on `_measurement` is microseconds. The `|> last()` aggregator returns one row per unique tag combination, so the result size stays bounded by the cardinality (counter count + gauge count + error-bucket count) regardless of how much history exists. Same pattern works for any time-series store with a discriminator column (Prometheus `__name__`, OpenTSDB metric name).
**Apply when**: Any boot-restore or batch-load against a time-series store with multiple "kinds" of metrics. Default to one composite query + client-side dispatch; only split into separate queries if a kind needs a fundamentally different time range or aggregation (e.g. errors want `last 100 per bucket` while counters want `last 1 per name`).

## [2026-05-08] — Counter naming discipline is the load-bearing precondition for time-series persistence
**Rule**: Before adding ANY external time-series persistence layer (InfluxDB, Prometheus, Datadog), audit the counter/gauge/histogram naming convention to confirm cardinality is BOUNDED at the source. The rule: `<subsystem>.<event>[.<variant>]`, depth ≤ 3, and ZERO dynamic high-cardinality segments — no `chat_id`, no `user_id`, no UUID, no per-message label as part of the metric NAME. Tags follow the same rule. Document the cardinality budget in the persistence design ("expect ~80 unique counter names, ~50 histogram names, ≤100 error buckets — well under any free-tier rate limit"). Before adding a new counter that interpolates a runtime value into the name, ask: "what is the upper bound on the distinct values this can produce?" If it's "unbounded" or "scales with users/messages/data", reject the design and find a different shape (a tag with a known small enum, or a separate gauge per fixed category).
**Why**: Time-series stores price by series cardinality, not by point volume — a few thousand series writing 1 point/min each is cheaper than ten series writing 1 million points/min. A single bug like `incCounter("tool.invoked." + chatId + ".ok")` can blow up the series count from ~80 to "one per chat ever seen" within hours, breaking the rate limit and silently dropping all metrics until the bug is caught. The naming discipline must be a CODE REVIEW CHECK, not just a documentation aspiration — every PR adding `incCounter(name)` should have the name's cardinality justified inline. Pre-flight cardinality audit during persistence design catches the existing offenders BEFORE the persistence layer goes live; without the audit, the first production day is a quota-exhaustion incident.
**Apply when**: Adding any external metrics persistence layer to an existing app, OR adding any new metric to an app that already has external persistence. Always audit cardinality before shipping; reject any name that interpolates an unbounded runtime value; pin the naming convention in a top-of-file comment in the metrics module so a future contributor sees the rule before adding their counter.

## [2026-05-08] — Pure-SVG charts beat chart libraries for non-interactive dashboards
**Rule**: For a dashboard with charts that have NO interactivity requirements (no zoom, no brush, no tooltips with cross-hair, no click-to-filter, no animation, no responsive auto-resize beyond what CSS provides), build the charts as pure inline SVG components — `<polyline>` over a `viewBox` for sparklines, CSS `width: ${pct}%` divs for bar charts, two `<polyline>` paths for dual-line charts, vertical ticks on a track for percentile bars. Do NOT add `recharts`, `victory`, `chart.js`, `d3`, or any equivalent library. Each chart is 30-60 lines of declarative TSX with no internal state. Verify the constraint with a `package.json` grep at the end (`grep -iE "chart|recharts|victory|d3"` returns no matches).
**Why**: A chart library adds 80-250 KB of gzipped JS plus a transitive dependency tree to audit, plus a future upgrade-or-pin decision when the library drops support for the current React version. For four chart types totaling ~150 lines of component code, the bundle/maintenance cost is enormous relative to value. Pure SVG components are maintenance-light by construction: no internal state to debug, no animation timing to tune, no theming layer, no version churn. The "I'll need a real library when requirements grow" hedge is wrong — when the dashboard genuinely needs zoom or brush, the SVG components are 30-line throwaways and the library swap-in is the same effort it would have been on day one. The "what if I need axes with auto-formatting?" hedge is also wrong — three hard-coded label positions cover the actual need.
**Apply when**: Any internal/admin/dashboard UI with read-only charts. Default to pure SVG; only add a chart library when the requirements EXPLICITLY include zoom, brush, animation, or interactive cross-filtering. The "we might need it later" justification is a fallacy — the cost is incurred today, the value is hypothetical. Apply the same logic to date pickers, dropdowns, and other "I bet there's a library" reflexes for read-only UIs.

## [2026-05-08] — Ask the vendor LLM for its own pricing via grounded web search instead of hard-coding rates
**Rule**: For "compute USD cost per LLM call" features where the vendor publishes pricing only in human-readable docs (no machine-readable price API), don't scrape, don't hard-code, don't pin a JSON file in the repo. Make ONE periodic call to the vendor LLM with the search-grounding tool enabled (`tools: [{ googleSearchRetrieval: {} }]` for Gemini), a strict-JSON prompt naming the model id and the official docs URL, `temperature: 0.0`, and a zod-validated parser. Cache the result in a dedicated `model_pricing` table; the cost-counter site reads the cached row synchronously. Refresh every 24h via `setInterval`; pin a 7-day staleness threshold for the post-boot one-shot so a fresh dev environment self-heals within 2 minutes.
**Why**: Hard-coded rates drift the moment Google's pricing page changes (and operators won't notice). Scraping the page is fragile (HTML churn, Cloudflare). ## [2026-05-08] — Cost accumulation in micro-USD integers avoids float drift across thousands of calls
**Rule**: When accumulating per-call USD cost into a long-running counter, store the counter as `Math.round(prompt_tokens * input_per_1m_usd + completion_tokens * output_per_1m_usd)` — this resolves to MICRO-USD ($0.000001 units) as integers, NOT float USD. The dimensional sanity check: `tokens * (USD per 1M tokens)` = `tokens × USD × 10^-6 × 10^6 / 10^6` = micro-USD per call. The dashboard divides by `1_000_000` only at render time. Counter type stays `number` (JS); 2^53 ≈ $9 quadrillion of headroom is plenty.
**Why**: Floating-point accumulation drifts: $0.0001234 added 10,000 times produces noticeable error. `Math.round` to nearest micro-USD per call bounds the drift at 1 micro-USD per call regardless of how many calls accumulate. Storing in integer-units in the counter Map preserves the JS-`number` arithmetic shape (no BigInt complexity) while sidestepping the FP issue. The render-time divide by 1e6 produces clean `$0.0000`-formatted strings without `.toFixed(20)` rounding artifacts. Same trick (integer base unit + render-time scale) works for any small-fractional accumulator: micro-bytes, milli-events, microseconds.
**Apply when**: Any per-call accumulator where the per-event value is small and the accumulated total spans many events. Choose the smallest integer unit that fits 1+ digits per event (micro-USD for cost, microseconds for fast latency, bytes for size); round once at the per-event boundary; scale once at the render boundary. Document the unit in the counter NAME (`gemini.cost_micro_usd`, NOT `gemini.cost`) so a future reader can't misinterpret the magnitude.

## [2026-05-08] — Stale-gated post-boot one-shot + long-period interval is the right cadence shape for slow-changing external state
**Rule**: For features that periodically refresh external state where the external state changes slowly (vendor pricing, GeoIP DB, currency rates), pair a long-period `setInterval(refresh, 24*60*60*1000)` with a STALE-GATED post-boot `setTimeout(refresh, 2*60*1000)` — the one-shot only fires if the cached state is missing OR `fetched_at < now - <staleness threshold>` (e.g. 7 days). Don't refresh on every boot (wastes vendor API calls when the operator restarts frequently); don't skip the one-shot entirely (a fresh dev environment never gets data); don't tighten the interval below the actual change cadence (noise + cost without benefit).
**Why**: A naive "refresh on every boot" pattern punishes operators for restarting — every dev-loop iteration burns a vendor API call. A naive "interval-only with no boot one-shot" leaves a fresh database with no data until the first interval tick (24h later). The stale-gated one-shot reconciles both: long-running prod processes don't re-fetch on restart, fresh dev environments self-heal within 2 minutes. The 7-day threshold tolerates a missed daily refresh (process crashed during the timer window) and forces a re-fetch on truly stale data. The 2-minute delay (not immediate) lets the rest of the boot sequence settle before competing for the network.
**Apply when**: Any boot-time + periodic refresh of slow-changing external state. Tune the three knobs to the change cadence: interval ≈ how often it actually changes; staleness ≈ how long is "too old"; one-shot delay ≈ how long to let boot settle. Wrap the whole block in try/catch so a Flux blip returns `null` for that field rather than crashing the metrics endpoint. If the older sample is missing (server uptime < window), the field still falls back to `null` cleanly because `first` will be `null`.
**Why**: A single composite Flux query that returns both `first` and `last` in one pipeline is possible but more brittle (uses `pivot()` or `union()` that's harder to debug). Two queries are ~100ms total, dispatched serially — negligible on a 5s dashboard poll. Subtracting in JS is one line vs. learning the Flux subtraction syntax. The `Math.max(0, ...)` guards against the edge case where the counter was reset between samples (operator reset, DB wipe) — without it, a reset produces a negative "delta" which renders as gibberish. The try/catch + null fallback follows the standard "external query failure shouldn't crash the endpoint" pattern.
**Apply when**: Any windowed delta computation on a monotonic counter from a time-series store. Default to two queries + subtract in app code; reserve composite Flux pipelines for when the windowing logic is genuinely complex (multiple groupings, derived rates). The "try strict first, fallback once, log+skip" pattern preserves observability: the warn-log raw-text-prefix tells the operator EXACTLY what the model returned, making prompt-tuning iterations debuggable. **Apply when**: Any LLM-as-structured-data-source. Always: strict prompt explicitly forbidding markdown, primary parse on raw text, ONE fallback for fence-stripping, log+skip on double failure. zod validation runs AFTER successful parse to catch shape errors (NaN, negative numbers, missing fields, wrong types). **Apply when**: Any module-level `let cached = ...` that's refreshed async and read on hot paths. ## [2026-05-11] — Flux operator selection by metric kind: counters use `derivative`, histograms use `aggregateWindow(last)`, gauges use `aggregateWindow(last)`, errors use `derivative` grouped by tag

## [2026-05-15] — Hard-coded `.ts` extension in dynamic import becomes a deploy-time gotcha — flag it explicitly

## [2026-05-15] — Two-layer enabled check (SQL filter + in-loop re-fetch) for global-OR-assigned resolution

Re-fetching each row immediately before use closes the window: the operator's toggle takes effect on the very next AI call, not "the next AI call after this one finishes." ## [2026-05-15] — Global-default + per-chat opt-in/opt-out resolved in one SQL `OR EXISTS` predicate
**Rule**: When a feature has both a "global default for all chats" toggle AND a "per-chat explicit assignment" override, encode both in one resolver query: `SELECT * FROM applications a WHERE a.enabled = TRUE AND (a.is_global_default = TRUE OR EXISTS (SELECT 1 FROM application_chats ac WHERE ac.application_id = a.id AND ac.chat_id = $1 AND ac.enabled = TRUE))`. Always: one SQL query with OR-composed predicates; document the precedence in the resolver's docstring; never compose in JS what SQL can compose with `EXISTS`. ## [2026-05-20] — Insert row FIRST, copy file AFTER: no rollback, document the orphan-row failure mode
**Rule**: When an install handler must both INSERT a DB row AND copy a file to disk, do the INSERT first, then the file copy, and DO NOT add a rollback path. If the copy throws after the insert succeeds, the row exists without a corresponding file on disk; the responder's per-plugin loader catches this at runtime and warns (per the existing per-item-isolation lesson), and the operator can delete-and-reinstall to recover. The reverse order (copy first, then insert) is WORSE: a copy that succeeds before an insert failure leaves an orphaned file with no DB record, and there's no operator UI to surface "this file is orphaned." The DB row is the source of truth for "is this plugin installed"; the file is the implementation; orphan rows are recoverable, orphan files are invisible.
**Why**: Rollback in this shape requires either a distributed transaction across Postgres + file system (overkill for a single-user dev tool) or a try/catch around the copy that calls `deleteApplication(row.id)` on failure (adds a second failure mode where the rollback itself can fail, leaving the operator in a worse state than no rollback). The "insert then copy, accept the orphan-row risk" policy is the minimal correctness story: every other branch of the install handler runs against a row that definitely exists in the DB; the only edge case is "copy failed after insert," and the runtime loader's warn-on-missing-file behavior makes this self-healing on the next AI call (warn logged → operator notices → deletes and reinstalls). Document the order and the rationale in the route comment so a future "I'll add transactional rollback for safety" refactor sees the deliberate trade-off.
**Apply when**: Any install/setup handler that combines a DB write with a file system write (or any other non-transactional resource). Always: write the system-of-record (DB) first; perform the side effect (FS, external API call, webhook registration) second; never add rollback unless the side effect is also transactional. Document the failure mode and the operator recovery path inline. Same pattern for "create user row then send welcome email" — the row is the truth, the email is best-effort.

**Apply when**: Any refactor that deletes a symbol used in 3+ files OR changes a shared interface's shape. Mark the atomic block explicitly in the plan (`> ATOMIC BLOCK: Steps X-Y form one change set`). The executor and verifier both treat the block as one unit — only the final step gets verification. ## [2026-05-21] — Vite IIFE micro-frontend: classic JSX runtime is the ONLY safe path when externalizing `react` to `window.React`
**Rule**: When building a Vite IIFE plugin bundle that must share the host page's React instance via `window.React`, set `@vitejs/plugin-react`'s `jsxRuntime: "classic"` and externalize ONLY `react` (+ `react-dom`) with globals `{ react: "React", "react-dom": "ReactDOM" }`. Do NOT externalize `react/jsx-runtime` mapped to `React` — the modern JSX transform emits `import { jsx, jsxs } from "react/jsx-runtime"` and those functions are NOT properties of the main `react` namespace object. Mapping `react/jsx-runtime` → `window.React` produces `React.jsx is not a function` at runtime. Classic runtime emits `React.createElement(...)` which IS on `window.React`, so a single externalization is sufficient. Bundle-size cost of classic runtime is negligible (~1–2 KB) compared to the alternative (bundling a second React copy or hand-injecting `window.ReactJSXRuntime` from the host page). Required: every `.tsx` file in the plugin must `import React from "react"` (default import, NOT `import * as React`) — classic runtime relies on `React` being a value in scope at JSX call sites.
**Why**: The "obvious" plan reads "externalize react and react/jsx-runtime, both globals point at React" and looks symmetric. It fails at runtime because the modern JSX runtime is a SEPARATE export surface — `react/jsx-runtime` is a distinct entry point in React's `package.json` that exports `jsx`, `jsxs`, `Fragment`, and nothing else; it shares no live binding with the main `react` namespace. There is no "just expose `jsx` on `window.React`" workaround because React doesn't export it on the namespace. The two real options are (a) classic runtime → one external mapping, host page sets `window.React` once, or (b) host page exposes `window.ReactJSXRuntime = await import("react/jsx-runtime")` AND plugin externalizes `react/jsx-runtime` with `globals: { "react/jsx-runtime": "ReactJSXRuntime" }`. Option (b) requires a second pre-injection contract that's easy to forget; option (a) collapses to a single rule the host enforces. Document the choice at the top of the plugin's `vite.config.ts` so a future author who reflexively flips to automatic JSX transform sees the warning before reinventing the trap.
**Apply when**: Any micro-frontend / plugin / extension architecture where (a) a separately-built JS bundle is loaded into a host page via `<script>` injection, and (b) the bundle must share React with the host to avoid hooks errors. Always: classic runtime + `react`/`react-dom` external + `import React from "react"` (default) in every `.tsx`. Verify with `grep -E "from \"react/jsx-runtime\"" dist/ui.js` (expect zero hits) and inspect `dist/ui.js` size + content for any `useState`/`createContext` strings (host React leaks would manifest as React-internals shipped in the bundle).

## [2026-05-21] — Runtime `<script>` injection beats build-time glob for true install-without-rebuild, but `onload` cannot signal readiness — poll the window registry
**Rule**: To install a new plugin's UI without rebuilding the host bundle, the host injects a `<script id="tele-plugin-ui-${slug}" src="/api/.../ui.js">` tag at mount time and the plugin self-registers on `window.__TELE_PLUGIN_UI__[slug] = Component`. The host polls that global for up to N attempts (e.g. 50ms × 60 = 3s) to detect readiness. CRITICAL: do NOT use `script.onload` as the readiness signal — `onload` fires exactly once per element insertion, so on a remount (when the script tag is already in `document.head`), the second `appendChild` is skipped (idempotent re-use) and `onload` never fires again, leaving the host stuck. The poll loop is the SOLE source of truth for "plugin ready"; it works on first mount AND remount because it just reads the global. Contrast with `import.meta.glob` (lessons-2026-05-21 build-time): glob bundles every matched file into the host's deploy, so install-time addition of a new plugin folder requires rebuilding + redeploying the host. Script injection + window-registry polling makes install a real runtime operation: drop a row in the DB, the API streams the new `ui.js`, the dashboard picks it up on next mount.
**Why**: The intuitive plan is "inject the script tag, listen for `onload`, then render." That works exactly once. On the second mount (user navigates away and back), the existing script tag is reused (correct, avoids duplicate `<script>` evaluation), so `appendChild` either short-circuits or no-ops — and the new `onload` listener attached to the existing tag never fires because the load already happened. The first mount sees a normal "ready in 50ms" timing; the second mount sees an infinite spinner. The poll loop sidesteps both states: if the global is already set (second mount), the very first poll tick (50ms) returns the component; if it's not yet set (first mount), the poll waits exactly as long as the network/parse takes. No event-listener bookkeeping, no race between "did onload fire before my listener attached." Document at the top of the loader component that the poll loop is the readiness signal and explain why `onload` is intentionally absent — otherwise a well-meaning future author will add it back as a "fast path" optimization and reintroduce the remount bug.
**Apply when**: Any host that runtime-injects external JS bundles and expects them to self-register a value/component on a shared global. Always: idempotent `getElementById` check before `appendChild`; poll the registry, not the script element's events; document the contract at the loader call site. The same pattern applies to web component registration (`customElements.get(name)` poll instead of `onload`), Web Worker self-registration, and any other "the bundle announces itself via globals" pattern where the registration is the real signal and the script-load event is incidental.

## [2026-05-21] — Auth allowlist must live in the global hook for unauthenticated static assets served by an authenticated API
**Rule**: When a route must be publicly fetchable (e.g. a JS bundle the browser loads via `<script>` injection into an already-authenticated page) but the API's global `onRequest` hook returns JSON 401 BEFORE the route handler runs, the "route handler returns a JS comment on 401" workaround is unreachable. Fix: add a URL-pattern allowlist directly in the onRequest hook itself, returning early for whitelisted paths BEFORE the auth check. UUID in the path is the unguessability primitive — same trust posture as `/api/health`, no separate auth needed. Document the carve-out inline in the hook with the trust model rationale (UUID is unguessable + single-user tool + content is operator-installed code already trusted on the host).

This fails because the global onRequest hook owns the request lifecycle: if it sends a response, the route handler NEVER executes. Adding an exception inside the route handler is logically dead code. The mirror trap: putting per-route auth bypass at the route level looks colocated and DRY ("each route knows its own auth posture") but the global hook can only be bypassed by an EXPLICIT allowlist in the global hook itself. **Apply when**: Any API with a global auth gate that needs to serve unauthenticated static resources (CSS, JS, images, fonts, public manifests). Always: bypass at the global hook layer, not the route layer; use URL-pattern allowlists (regex on the path) rather than per-route flags; document the trust model inline. ## [2026-05-21] — Host/plugin React-sharing contract: set globals BEFORE script injection, plugin uses default import + classic runtime
**Rule**: For a plugin bundle to safely reuse the host page's React instance (so hooks work, contexts cross the boundary, and only one React ships), the host MUST execute `(window as any).React = React; (window as any).ReactDOM = ReactDOM` BEFORE injecting the plugin's `<script>` tag, inside the same effect/render path that mounts the plugin loader. The plugin's `vite.config.ts` MUST list `external: ["react", "react-dom"]` with `globals: { react: "React", "react-dom": "ReactDOM" }`. The plugin's `.tsx` source MUST use `import React from "react"` (default import) — NOT `import * as React`, because the IIFE bundle's externalization replaces `react` with `window.React` which is the module-namespace object, not a default-export. Violating ANY of the three (host doesn't set globals → plugin's `React` is undefined; plugin uses namespace import → IIFE captures the wrong shape; plugin doesn't externalize → ships a second React copy → "Invalid hook call" at runtime) breaks the integration with subtle, slow-to-diagnose symptoms.

The failure modes are insidious: a second React instance produces "Invalid hook call" with no stack-trace hint at the cause; a missing host-global produces `Cannot read properties of undefined (reading 'createElement')` with the offending file in `dist/ui.js` (a one-line minified blob); a namespace import in the plugin works fine in isolated dev but crashes in IIFE form because Rollup's IIFE wrapper assumes the external IS the default export. Document the contract at THREE places: the host's `window.React = React` line (with `// MUST run before plugin script injection`), the plugin's `vite.config.ts` externals block (with `// host provides window.React; see host/PluginSlot.tsx`), and the plugin's authoring README. Triple documentation is intentional — the failure modes are too far apart to debug from a single hint.
**Apply when**: Any plugin/extension architecture that runtime-loads a separately-built bundle and shares framework runtime (React, Vue, Svelte) with the host. Always: host sets globals before injection, plugin externalizes the framework, plugin uses the import style that matches its build's IIFE-wrapper assumptions (default import for React's UMD-style global). Verify with `grep -E "createElement|useState\\s*=" dist/ui.js` (definitions absent, usages present) and a DevTools identity check (`window.React === host's React`). The same shape applies to Vue (`window.Vue`), MapboxGL, Three.js, and any other library where two instances on the same page break encapsulation invariants.

## [2026-05-21] — `git clone` shell-exec MUST `rm -rf dest` before throwing on failure — partial-clone dirs poison subsequent install attempts
**Rule**: A spawn-based `git clone url dest` wrapper MUST `await rm(dest, { recursive: true, force: true })` BEFORE throwing in ALL three failure paths: (a) timeout/SIGKILL via `setTimeout` (lessons-2026-05-02 primitive), (b) `spawn` error event (binary missing, permission denied), (c) non-zero exit code (auth failure, network error, invalid URL). Otherwise a partial-clone dir survives on disk; the next install attempt's "if dir exists, skip clone" branch loads garbage (empty `.git/`, no working tree, or a half-written tree from a network drop). The cleanup MUST be `await`ed before the throw — fire-and-forget would let the next call race the rmdir. Capture stderr in the spawn handle and surface it in the thrown error message so the operator sees the actual git failure ("Permission denied (publickey)", "Could not resolve host"), not just "clone failed."
**Why**: The "dest exists → skip clone" branch is the right idempotency primitive for the success case (operator reinstalls, plugin source already on disk, just rebuild the manifest from it). It becomes a poison primitive in the failure case: a half-cloned dir LOOKS like a successful install to the dir-check but loads zero working files. The symptom is "first install attempt fails with clone error; second attempt silently 'succeeds' but the plugin's `src/hook.ts` is missing." The operator's instinct is to debug the plugin, not the clone wrapper. The fix is one line per failure path (`await rm(dest, { recursive: true, force: true })`) and one careful audit — ALL three paths must clean up, not just the exit-code path most people remember. The reason fire-and-forget cleanup is unsafe: the throw immediately unwinds the install handler, which returns a 500 to the operator, who clicks Install again within milliseconds — if the rmdir hasn't completed, the dir-check sees the partial dir and loads garbage. The `await` ensures the dir is gone before the response is sent. Document the cleanup at the throw site so a future refactor that splits failure handling doesn't drop one of the three paths.
**Apply when**: Any wrapper around `spawn`/`exec` that creates filesystem artifacts as a side-effect and is called from an idempotent caller. Always: cleanup before throw, in ALL failure paths (timeout, spawn-error, non-zero exit); `await` the cleanup; capture stderr for the error message. The same pattern applies to `tar -xf` (partial extract), `wget -O` (partial download), `npm install` (partial node_modules), `docker pull` (partial image layers) — anything where "the dest exists" is the caller's success signal AND the failure path can leave a partial dest.

## [2026-05-21] — Clone-before-insert is the right ordering for git-sourced plugins: you need the manifest to validate, accept the orphan-dir failure mode
**Rule**: For git-cloned plugin install: resolve the installed path (which includes the actual `git clone`) FIRST, then read+validate `manifest.json`, then validate the `src/hook.ts` exists and `manifest.slug === registry.slug`, and ONLY THEN INSERT the `applications` row. This REVERSES the prior "insert row FIRST, copy file AFTER, no rollback" rule (lessons-2026-05-20) — that rule was for the case where the manifest was already on disk and the install was just a DB row. For git-cloned plugins, you cannot insert before clone because (a) you need the manifest to fill in the row (name, type, slash_commands), (b) you need to verify slug-equality before claiming the slug in the UNIQUE-constrained DB column. The honest failure mode is: clone succeeds but INSERT fails with `23505` (slug already exists from a concurrent install or DB-state drift) — the cloned dir lingers on disk as an orphan. The alternative ordering (insert-then-clone) creates a WORSE orphan: a dangling DB row pointing at an empty/missing dir, which subsequent reads have to special-case. Operator recovery for orphan-dir: `rm -rf data/applications/<slug>` manually or retry install with a different slug. Document this as the intentional v1 failure mode.
**Why**: The instinct from lessons-2026-05-20 ("insert row FIRST") is right for cheap-side-effect installs where the file copy is fast, idempotent, and doesn't need to be validated for content (it was authored by the same operator who wrote the registry manifest). For git-cloned plugins, the cost structure is inverted: the clone is the expensive, slow, validation-rich operation; the INSERT is the trivial last step. Inserting before clone means you've claimed the slug in the DB before knowing whether the plugin is even readable, and rollback on clone failure means deleting the DB row, which races with the operator hitting Install again. Cloning before insert means the slug isn't claimed until validation passes; the only failure mode left is the DB-side UNIQUE violation, which is rare (operator's slug typo or true concurrent install). The orphan-dir is uglier than orphan-row only at first glance: orphan-row keeps loading attempts going (every list-applications call tries to read `installed_path` → fails with "no manifest" → warn-log spam), while orphan-dir is invisible until the operator notices `data/applications/` is fatter than expected. The operator can `rm -rf` an orphan-dir; cleaning an orphan-row requires SQL access.
**Apply when**: Any install/registration flow where the "materialize the artifact" step is the validation step (git clone, npm install, docker pull, container image build). Clone/build first, then validate, then claim the unique slot in the DB. The orphan-artifact-on-disk is the right failure mode to accept; the orphan-row-in-DB is the failure mode to avoid. Document the ordering decision in the install function's top-of-file comment with a reference to BOTH this lesson AND lessons-2026-05-20 (the contrasting case) so a future author understands which rule applies to which install type.

## [2026-05-21] — Filesystem registry of plugin manifests becomes obsolete once the registry catalog moves to the DB
**Rule**: When migrating from "registry is a folder of manifest.json files maintained inside the host repo" to "registry is a DB table whose rows point at git URLs or local paths," the OLD registry folder MUST be deleted in the same change set — leaving it behind creates a second source of truth that drifts. The DB row is now authoritative for "what plugins exist in the catalog"; the actual plugin source (manifest + hook + UI source) lives at the resolved `installed_path` (operator's git checkout or local path), and the host reads the manifest fresh from that path on every list operation. The deletion includes: the per-plugin folders under `<host>/applications/registry/<slug>/`, the `.gitkeep`, and the surrounding empty parent. KEEP only `<host>/applications/.gitkeep` and the pre-installed `<host>/applications/<slug>/hook.ts` files (uninstall-preserves rule, lessons-2026-05-20). The migration must also break dangling references: `UPDATE applications SET registry_slug = NULL WHERE registry_slug IN (<deleted slugs>)` so the dashboard's "is this slug installed?" logic doesn't misreport.
**Why**: The filesystem-registry approach (manifest as source of truth, copied/linked from a separately-maintained plugin author repo) requires a sync step every time a plugin changes — author edits their repo, runs `copy-ui`/`copy-hook` scripts to mirror into the host's registry folder, host's bundler picks up the change. The sync step is forgettable, breaks during merge conflicts, and produces two copies that can drift (lessons-2026-05-21 plugin-isolation-contract). DB-row + git/local source path eliminates the mirror: the host's `loadManifestFrom(installedPath)` reads the manifest fresh from the plugin's actual repo every time, so author edits are visible immediately on the next dashboard render — no sync needed. The dangling-ref UPDATE is the often-missed second step: if you delete the filesystem registry but leave `applications.registry_slug = 'kundali-match'` rows pointing at a now-nonexistent registry entry, the dashboard's "Browse tab → is this slug installed?" query returns wrong answers and the Install button on the rehydrated DB-backed row is in the wrong state. The hardcoded-host backfill (`UPDATE applications SET installed_path = '/specific/path' WHERE registry_slug = 'X'`) is the honest failure mode for the migration: it works on the original-author's host, silently no-ops elsewhere, and the operator's recovery on a fresh host is to re-install from the new Browse tab. Document this at the top of the migration file so a fresh-host operator sees the explanation when their kundali install row has `installed_path=NULL`.
**Apply when**: Any refactor that moves a filesystem-based registry to a DB-backed one (plugins, themes, integrations, datasource configs). Always: delete the OLD folders in the SAME change set (atomic-block rule); add a migration UPDATE that nulls dangling foreign-key-style references; document the hardcoded-host backfill caveat at the top of the migration for fresh-host operators. The opposite migration (DB → filesystem) is rarely right — filesystem registries scale poorly past N=10 and require git commits to add/remove entries, which is a worse UX than a dashboard form. Prefer DB-backed catalogs from day one for any registry that's expected to be operator-mutable at runtime.

## [2026-05-23] — Internal nav in a plugin = parent owns `view`, children own everything else; accept per-tab reset for v1
**Rule**: When adding tab-style internal navigation inside a single-component plugin UI (e.g. splitting one form into "Settings" + "Matching" tabs), keep the parent component tiny: ONLY hold `const [view, setView] = useState<"a" | "b">("a")` and a 2-button nav bar with active-tab styling. If users actually complain, that's the signal to hoist — not a speculative concern.
**Apply when**: Any tab/panel split inside a single-page component. Default to: parent holds `view` only; children are self-contained; conditional render unmounts the inactive one; document the per-tab reset as accepted behavior. ## [2026-05-23] — One-line `narrowGender`-style helpers belong at file scope, not inlined or imported
**Rule**: When an external API returns `string` for a field that the consumer's type system requires as a discriminated union (`"male" | "female" | "other"`), write a one-line narrowing helper at file scope and call it at every construction site. Pattern: `function narrowGender(raw: unknown): "male" | "female" | "other" { const s = typeof raw === "string" ? raw : ""; return s === "male" ? "male" : s === "female" ? "female" : "other"; }`. Use it BOTH at the API-to-domain boundary (building Person A from `fetch` JSON) AND at the UI-input-to-domain boundary (Person B's `<select onChange>` value, because `e.target.value` is typed as `string`, not the union). Do NOT inline the ternary at the call sites — duplication invites drift (someone adds a `"non-binary"` case in one place but not the other). Do NOT extract to a shared utility module — the helper is one line, depends on the local domain's union, and adding cross-file coupling for one line is worse than the duplication it would avoid.
**Why**: The naive approaches both fail in different ways. Inline ternary at every site: works initially, but when the union grows (e.g. add `"prefer_not_to_say"`), you must find and update N sites and one will be missed. Cast `as "male" | "female" | "other"`: lies to the type system — a malformed API response like `gender: "MALE"` (uppercase) or `gender: 42` (number) survives the cast and crashes at the next consumer (e.g. `mangalDosha(p)` reading `p.gender.toLowerCase()`). Extract to `@tele/shared/narrowers`: forces a cross-package dependency for one line, plus the shared package can't know which subset of values the local domain treats as valid. The file-scope one-liner is the right granularity: same module sees the union, narrowing is total (returns `"other"` for anything unexpected so downstream code never sees an invalid value), and the call sites read like assertions of intent — `gender: narrowGender(profile.gender)` self-documents. The `unknown` parameter type (not `string`) is deliberate: the helper accepts ANY input, even numbers/null/undefined, and produces a valid union value — this is the "make illegal states unrepresentable" contract at the boundary.
**Apply when**: Any place where an `unknown`/`string`/`any`-typed external value (HTTP JSON, form input, localStorage, URL query param) must be widened-then-narrowed into a closed string-literal union the codebase has chosen. Always: file-scope helper (not inlined, not imported); accept `unknown` parameter type; return a total mapping (never throw, never `undefined` — pick a sensible default for the residual case); call at EVERY boundary, not just the most-obvious one. The cost is one helper per union per file; the win is no `as X` casts, no surprise crashes downstream, and the helper itself becomes the single place to update when the union grows. Skip this pattern only for trivial booleans (`x === "true"`) or where TypeScript can already prove the narrowing (e.g. a `zod` parse result).

## [2026-05-23] — Hold raw pieces in component state, derive the rich view-model at render time
**Rule**: When a domain library exports a rich result type (e.g. `MatchResult { total, max, kootas[], summary, disclaimer, fixedMangal, candidateMangal, nadi, rajju }`), the component's `useState` should hold only the IRREDUCIBLE pieces needed across renders — typically the raw outputs that the user's choice (compute click) produced — and DERIVE the display fields (summary text, verdict color, ratio threshold) at render time from those pieces. For the kundali matching UI: `useState<{ total, max, kootas, fixedMangal, candidateMangal, nadi, rajju } | null>(null)` instead of `useState<MatchResult | null>(null)`. The `summary` and `disclaimer` fields the library produces are static strings derivable from the same inputs, so storing them in state is redundant — and worse, if the library's text changes between compute and render (e.g. a translation update), the displayed summary is stale until the user re-clicks Compute.
**Why**: The "just store the whole library result" instinct is seductive (one type, no destructuring), but it conflates "what the user produced by clicking" with "how we choose to display it." The display layer should be free to add a new section (e.g. "Show ratio as percentage") without requiring re-compute or a state-shape migration. Holding raw pieces also makes the state-shape obvious at the `useState` declaration site — a reader sees exactly which fields the UI cares about, not a generic "MatchResult" they have to drill into. The downside (slightly more verbose `setResult({ total, max, kootas, ... })`) is paid once per compute path and is trivial. The win is that the render function is the SOLE owner of how to present the result, and the library type is a transport contract, not a UI contract. Same principle applies to API response shapes: don't put the raw response in state if you're going to massage it on every render — massage once at the API-to-state boundary, store the massaged form, render straight from it. Or, conversely, store the raw form and massage at render. Either is fine — what's WRONG is storing the rich library type AND re-massaging at render, because then the library's evolution (new fields, renamed fields, deprecated fields) leaks into the component's state shape.
**Apply when**: Any component that holds a third-party library's result type in `useState`. Audit which fields are actually used in the render tree vs derivable. If the derivable fields outnumber the stored ones, store only the inputs and derive on render. If the stored fields outnumber the derivable ones, store the whole type. Tipping point in this task: 4 of 9 fields were trivial derivations (summary/disclaimer were unused, ratio was one division), so the leaner shape won. Document the choice with a one-line comment at the `useState` (e.g. `// raw pieces only; verdict/ratio derived at render`) so the next contributor doesn't add `summary` back "for completeness."

## [2026-05-23] — Two error severities = two visual treatments: top banner for blocking, inline rose-400 for transient
**Rule**: When a single panel surfaces two distinct error classes — (a) a BLOCKING error that disables the panel's primary action (e.g. "Person A profile incomplete; you must fix this in another tab"), and (b) a TRANSIENT error from the user's current action (e.g. "Person B name is required"; "compute failed because field X is malformed") — use VISUALLY DISTINCT treatments so the user can tell at a glance which kind they're looking at. Blocking error: full-width top banner with prominent background (`border-rose-700 bg-rose-900/40 p-3 rounded`), positioned ABOVE the form, accompanied by disabling all form inputs (`disabled={formDisabled}` driven by `const formDisabled = !personA`). Transient error: inline text below the submit button (`text-rose-400 text-sm`), no background, no disable — the form stays interactive because the user is mid-flow and the error is "what you just did needs fixing." Using `text-rose-400` for BOTH is the bug: the user can't visually distinguish "I need to leave this tab and fix something else" from "I made a typo just now."
**Why**: The visual-treatment uniformity instinct ("all errors look the same; one error style is consistent") loses to the user-task-clarity principle: errors that require different USER ACTIONS should look different, because the user's eye is doing pattern-matching, not reading text. A top banner with a colored background says "stop, this is a precondition problem" without the user having to read the text — they SEE that something blocking is in their way before they parse what it says. An inline rose-400 line below a button says "your submit had a problem; fix and retry" without the user having to wonder if the whole page is broken. The form-disabling reinforcement (`disabled={!personA}` cascaded to every input) is the second half of the contract: the visual treatment AND the interaction state agree that the panel is blocked. Without the disabling, a user reading the banner might still try to fill out the form and get a confusing second error from `handleCompute` when they submit. With it, the form's grayed-out state matches the banner's "you can't proceed" message. The same two-tier pattern applies to: form validation (field-level red border for transient, top-of-form red bar for "this entire flow is locked"), API errors (inline retry for 5xx, top banner for 401/403 that requires re-auth), and CSP/network errors (top banner because the user can't fix them by retrying the form).
**Apply when**: Any panel that can surface both "precondition blocking" errors (something elsewhere is wrong) and "current action" errors (your most recent input or click). Always: top banner + disable inputs for blocking; inline text near the action for transient. If the panel only has one error class, ONE treatment is fine — the rule kicks in only when the user must distinguish two kinds. The disable-the-form reinforcement is the critical second step that's easy to skip: without it, the visual "blocked" signal contradicts the interactive "still clickable" signal, and the user trusts the click over the color.

## [2026-05-23] — Verify bundled-engine inlining via grep for a UNIQUE-enough string fingerprint
**Rule**: When a Vite IIFE bundle is supposed to inline a set of pure-TS engine modules (e.g. `ashtakoot.ts`, `doshas.ts`) into the output `dist/ui.js`, verify the inlining post-build by greping for a UNIQUE string that ONLY appears in the engine source — never check for the module path or filename (those are stripped by minification) and never just check bundle size (size grew = could be anything; size shrunk = could be tree-shaking working too well). For the kundali engine, `"Graha Maitri"` (a specific koota name, exact string, not stemmed or split) is the fingerprint: `grep -c "Graha Maitri" dist/ui.js` returns `1`. If the grep returns `0`, the engine was NOT inlined (a misconfigured `vite.config.ts` external, a wrong relative import path that silently resolved to nothing, or an aggressive tree-shake that decided the symbol was unused). Pair the unique-string check with TWO other greps: `grep -c "react/jsx-runtime"` must be `0` (proves classic JSX runtime, no second React shipped), and `grep -c "__TELE_PLUGIN_UI__"` must be `1` (proves the self-registration block survived minification).
**Why**: Size-based verification is too noisy: "bundle is 15.5 KB, expected 10–15 KB, looks fine" doesn't tell you WHICH code is in there. A bundle could be 15 KB because the engine inlined correctly AND a stray dependency leaked in — the right size for the wrong reason. The unique-string grep is the smallest signal that says "this specific code path made it into the output." The string MUST be unique enough that it can't appear from unrelated content: a generic word like `"compute"` or `"score"` could appear in any other library's prose. A domain-specific phrase like `"Graha Maitri"` is the kundali engine's signature — if it's present, the engine inlined; if it's absent, only the engine is missing. Pair it with the negative grep (`react/jsx-runtime` absent) to catch the classical JSX-runtime regression, and the positive grep (`__TELE_PLUGIN_UI__` present) to catch a registration-block removal by accident. Three greps, three signals, all easy to add to a verify checklist and all impossible to game by mutation.
**Apply when**: Any post-build verification of a bundle that's supposed to inline N specific modules and exclude M others. For each module that MUST be inlined, pick a unique-enough string (a domain term, a long literal, a specific error message) and grep for it. For each module that MUST NOT be in the bundle (host-provided externals, dev-only helpers), pick a unique-enough string from THAT module and grep for ZERO hits. For the IIFE self-registration contract, grep for the exact registration call (`__TELE_PLUGIN_UI__["slug"]`) and the slug literal. The whole set takes under a second to run, catches misconfiguration that size-checks miss, and provides specific failure messages ("engine missing" vs "second React leaked" vs "registration block stripped") instead of a vague "bundle wrong size."

## [2026-05-23] — `voiceNote: true` + `DocumentAttributeAudio({ voice: true })` + `mimeType` is the GramJS contract for voice messages
**Rule**: To send a Telegram voice message via GramJS `client.sendFile`, ALL THREE pieces must be present and consistent: (a) `voiceNote: true` in the sendFile opts (tells GramJS to use the voice-message wire format), (b) `mimeType` set to the actual audio mime (`audio/wav`, `audio/ogg`, `audio/mpeg` — honest, not aspirational), (c) `attributes: [new Api.DocumentAttributeAudio({ duration: Math.max(1, Math.round(durationSec)), voice: true })]` (the `voice: true` on the audio attribute is what makes Telegram render the waveform-style voice bubble vs. an audio-with-playback bar). **Why**: Telegram's wire format has two distinct ways to send an audio file: as a music/podcast track (DocumentAttributeAudio with `voice: false`, renders with title/performer/scrub bar) and as a voice note (`voice: true`, renders with waveform + play button). GramJS's `sendFile` exposes both through different combinations of flags, and the combination is not intuitive — `voiceNote: true` is necessary but not sufficient. The three-piece contract is the documented-by-experimentation shape; missing any one piece produces a subtly wrong render that's easy to ship without noticing in tests. The `audio/wav` mime is the honest choice when the TTS produces WAV; encoding to OGG/Opus would give the iconic waveform render but adds a transcode dependency (ffmpeg) that's heavier than the feature warrants for v1. The duration-floor matters because TTS for very short text (one syllable) can produce sub-second audio that rounds to 0; the Telegram error for `duration: 0` is `[400 MEDIA_INVALID]` with no further hint.
**Apply when**: Sending voice messages from any GramJS-based Telegram integration. Always: `voiceNote: true` + honest `mimeType` + `DocumentAttributeAudio({ duration: max(1, round(seconds)), voice: true })`. Mirror the same three-piece shape in both the user-account sender and any bot-side `ReplyAdapter.sendVoice`. If the iconic waveform render is required (vs. the audio-playback bar), the path is OGG/Opus transcoding — that's a separate feature, not a flag change. ## [2026-05-23] — WAV header is a 44-byte literal-offset blueprint; verify each offset with a runtime byte-test, not just by code review
**Rule**: When writing a raw WAV header generator (`buildWavHeader(pcmLength, { sampleRate, numChannels, bitsPerSample })`), the implementation is a fixed sequence of `header.write("RIFF", 0)`, `header.writeUInt32LE(36 + pcmLength, 4)`, ... etc. through offset 40. The 44-byte total and every offset are SPEC-DEFINED — there is no flexibility. Code review of the implementation is insufficient verification: a transposed offset (`writeUInt16LE(bitsPerSample, 32)` instead of `34`) compiles, runs, and produces a header that EVERY decoder rejects with a generic "invalid WAV" error pointing at no specific field. The right verification primitive is a runtime byte-test: write a tiny Node script after the build that constructs the header with known inputs (sampleRate=24000, channels=1, bitsPerSample=16, pcmLength=0), then asserts each named-spec offset matches its expected bytes (`bytes 0-3 === "RIFF"`, `bytes 24-27 === [0xC0, 0x5D, 0x00, 0x00]` for 24000 LE, `bytes 34-35 === [0x10, 0x00]` for 16 LE). The script takes 10 lines and catches every class of offset/endianness/byte-width bug at verify time, before the first end-to-end call.
**Why**: WAV header bugs are uniquely punishing: the file LOOKS correct (right size, parses as bytes), every modern decoder rejects it with a vague error, and the troubleshooter has no idea whether the bug is in the header generator, the PCM encoder upstream, the audio capture pipeline, or the consumer. The runtime byte-test inverts the diagnostic flow: if every named offset matches its spec value, the header is correct by construction and the bug is downstream. The test must use a KNOWN sample rate (24000 here, common for speech) whose little-endian bytes are recognizable (`0xC0 0x5D` = 24000); using something opaque like 22050 means a reader of the test has to mentally compute `22050 = 0x5622` which obscures intent. The PASS criterion is byte-exact equality, not structural similarity — `0x5D 0xC0` (big-endian) is NOT acceptable for a little-endian field. Same verification pattern works for any binary header: PNG signature, ID3v2 tags, MP4 box headers, ELF headers — anywhere a few dozen bytes encode a fixed-spec layout, a byte-test against named offsets is the verification primitive.
**Apply when**: Writing or reviewing any binary header generator. Always: pair the implementation with a runtime byte-test using a known small input; assert each named-spec offset's value byte-exact; use sample-rate / size values whose LE encoding is human-recognizable (24000 → `C0 5D`, 16 → `10 00`). The test belongs in the verify step (runs against `dist/` after build), not in the source file's tests directory — the goal is "the BUILT artifact has the right bytes," not "the source compiles." Pair with the AC ordering pattern: list the byte-offset checks as their own acceptance criterion (AC12 in this task) so the verifier knows to run the script rather than infer correctness from code review.

## [2026-05-27] — Module-scoped Map+Set persist cap enforcement across calls; function-scoped resets it every call
**Rule**: When a factory like `makeAppEmit(slug)` returns a closure that must enforce a per-slug RUNTIME cap (e.g. "max 50 unique custom counter names per plugin slug across the process lifetime"), the bookkeeping state — `customNamesPerSlug = new Map<string, Set<string>>()` and `warnedOverCap = new Set<string>()` — MUST live at MODULE scope (top of file, alongside the factory function), NOT inside the factory function body. A naive implementation declaring `const seen = new Set()` inside `makeAppEmit` gives EACH call a fresh empty set, so the cap is never reached — every invocation of `emit("name_51")` would happily increment a new counter. The closure returned by the factory MUST close over the module-scoped containers. Cheap-closure principle: the factory itself does a few field reads (lookup-or-create the per-slug Set) — microseconds per AI call; the state lives for the process lifetime as designed.
**Why**: Caps that reset every call are not caps. The hot-path call site is `makeAppEmit(slug)(name)` — `makeAppEmit` runs every getContext/handleSlashCommand invocation (could be hundreds of times per minute) and the factory function's body re-executes top-to-bottom each time. Anything declared inside (with `const`) is born fresh and dies when the closure is GC'd. Module-scoped state survives every factory invocation because it's initialized once at module load, mutated by every closure that touches it. The same trap applies to ANY "remember which X we've seen across calls" requirement — rate limiters, one-time-warn flags, dedupe sets, lazy-init caches. The fix is always the same: move the state to module scope; document the lifetime in a comment so a future refactor doesn't "encapsulate" it back into the factory. ## [2026-05-27] — Slug-with-hyphens in counter names: use startsWith + split, not regex, to extract cmd from `app.<slug>.slash.<cmd>.<status>`
**Rule**: When deriving per-app metrics from counter keys like `app.kundali-match.slash.set-profile.ok`, do NOT reach for a regex like `/^app\.([a-z0-9-]+)\.slash\.([a-z0-9_]+)\.(ok|err)$/` — slugs allow hyphens and cmd names may too, making the regex's group boundaries ambiguous and fragile. Use string-prefix + split: `const slashPrefix = \`app.${slug}.slash.\`; if (!key.startsWith(slashPrefix)) continue; const tail = key.slice(slashPrefix.length).split("."); if (tail.length !== 2) continue; const [cmd, status] = tail;`. The known-prefix (with the known slug interpolated) plus a strict `length === 2` check on the tail handles slugs and cmds containing hyphens correctly — `app.kundali-match.slash.set-profile.ok` slices to tail `["set-profile", "ok"]`, length 2, clean extraction. A regex with `[a-z0-9_-]+` greedy groups would either reject this key (wrong) or match ambiguously (worse).
**Why**: Regex over dot-separated counter names with both hyphens and underscores in the segments is a known footgun: greedy alternation can collapse two segments into one, and `\.` anchoring requires every segment to be a fixed enum that the regex author maintains by hand. The startsWith + split approach treats the prefix as a literal (known slug, known stage) and only the variable tail as data — eliminates the regex-design problem entirely. The strict `tail.length !== 2` check is the validation: any counter that doesn't decompose into exactly `[cmd, status]` is dropped silently (defensive against future framework counters that might land at deeper depth). This pattern generalizes: anywhere you have a dotted namespace with a known prefix and a variable tail, prefer `key.startsWith(prefix) + key.slice(prefix.length).split(".")` over regex. The same logic applies to `app.<slug>.custom.<name>` extraction — known prefix + tail of length 1.
**Apply when**: Parsing any structured-key string with a known prefix and a variable suffix of bounded depth. Always: build the prefix with interpolated known segments; startsWith-filter; slice + split; validate the tail's length matches your expectation; destructure. Reserve regex for genuinely free-form patterns (validating a slug shape against `/^[a-z0-9-]{1,64}$/` at the boundary IS the right use). Document the parser inline with a comment showing an example key and the resulting tail, so a future reader can verify the parser handles their new counter shape without running it.

## [2026-05-27] — Nested `<Link>` (or `<a>`) inside `<button>` is invalid HTML5; restructure with `<div role="button">` + sibling buttons
**Rule**: HTML5 forbids interactive descendants of `<button>` — `<button><Link>...</Link></button>`, `<button><a>...</a></button>`, and `<button><button>...</button></button>` are all spec violations and React logs a `validateDOMNesting(...): <a> cannot appear as a descendant of <button>` warning at mount. The fix is structural, not attribute-tweaking: change the outer interactive container to a `<div role="button" tabIndex={0} onClick={...} onKeyDown={(e) => { if (e.key === "Enter" || e.key === " ") { e.preventDefault(); onClick(); } }}>` and place the inner interactive elements as direct SIBLINGS of the row contents. Each inner interactive child MUST call `e.stopPropagation()` in its own onClick so the outer div's handler doesn't also fire. The `e.preventDefault()` inside the keydown handler is load-bearing — without it, Space scrolls the page when the outer div is focused. Do NOT try to "fix" with `event.stopPropagation()` on the inner Link alone while keeping the outer `<button>` — the DOM nesting is still illegal, and the warning still fires.
**Why**: A nested-interactive structure compiles, renders, and superficially works — clicks usually do the right thing because event bubbling routes through whichever handler the browser chose first. But (a) React's dev-mode warning is permanent noise that masks real warnings, (b) screen readers announce the inner Link as "link inside button" which is incoherent, (c) some browser/AT combinations route the click to the outer button only, breaking the link, (d) future visual-regression or a11y tests will flag the structure. The structural fix (div+role) is ~5 lines of keyboard-handler boilerplate that's worth writing once. The B1 critic on this task caught this exact pattern in `Metrics.tsx` ApplicationRow: the original plan had `<Link to="/observability/apps/${slug}">` nested inside the accordion's `<button onClick={onToggle}>`. The plan pivoted (B2) to use a tab inside the existing `/applications/:id` route AND replaced the outer button with `<div role="button">` so the "Details →" navigate button could sit as a legal peer of the chevron. Both fixes were needed: the routing pivot eliminated the orphan namespace, AND the structural fix eliminated the nesting.
**Apply when**: ANY UI element that needs both a "click the whole row to do X" affordance AND inner clickable widgets (buttons, links, icon-buttons). Always: outer container is `<div role="button" tabIndex={0}>` (NOT `<button>`); explicit keyboard handler for Enter+Space with `preventDefault`; inner widgets are siblings with `stopPropagation` in their onClick. Greppable acceptance check: `grep -E '<a\s|<Link' <file> | head` should NOT show matches inside any row container that has an outer click handler. The same rule applies to `<a>` containing `<button>` — both directions of nesting are illegal. Pair with the verifier checklist: open DevTools console on the affected page, confirm zero `validateDOMNesting` warnings.

## [2026-05-27] — Single source of truth for derived lists: iterate the storage Map's keys, NOT the cap-tracking Set
**Rule**: When a module maintains BOTH a storage Map (`tsPerSlug: Map<slug, Map<name, ring>>`) AND a parallel cap-tracking Set (`tsNamesPerSlug: Map<slug, Set<name>>`) that are SUPPOSED to mirror each other, all downstream consumers that need "the list of names for this slug" MUST iterate `tsPerSlug.get(slug)?.keys()` (the storage Map) — NEVER `tsNamesPerSlug.get(slug)` (the Set). ## [2026-05-27] — Numeric Unix ms timestamps end-to-end: no ISO conversion across server-API-client boundary
**Rule**: When emitting timestamped data points (timeseries, events, audit logs) from server to dashboard via JSON, use `t: number` (Unix milliseconds, the output of `Date.now()`) all the way through — in the storage struct (`{ t: Date.now(), v: value }`), in the shared type (`interface Point { t: number; v: number }`), in the JSON wire payload (no transformation in the route handler), and in the client consumer (`<LineChart points={...}>` reads `p.t` as a number directly). Do NOT convert to ISO strings (`new Date(p.t).toISOString()`) at the API boundary "for readability" — JSON numbers are smaller (13 bytes for `1748351923456` vs 24+ bytes for `"2026-05-27T14:38:43.456Z"`), parse-free on the client (no `new Date(s).getTime()` round-trip), and lossless. The only acceptable boundary for ISO conversion is the rendering layer where a human reads the timestamp (axis label, tooltip) — and even there, use `new Date(p.t).toLocaleTimeString()` at render time, not at fetch time.
**Why**: The V5 critic on this task flagged the original plan's `t: string` (ISO) wire format. Switching to `t: number` end-to-end (a) halves the timestamps' wire-size for a 240-point payload (saves ~2.6KB per app on a polled endpoint), (b) eliminates the `new Date(s).getTime()` parse cost on every chart render (~30µs × 240 points × N apps), (c) removes a class of bugs where the server uses one ISO formatter (`toISOString()`) and the client expects another (`toJSON()`, locale-aware, with-or-without-Z suffix), (d) makes arithmetic on timestamps (deltas, sliding windows) trivial — `p2.t - p1.t` is a single subtraction vs. two `Date.parse` calls. The "readability" argument for ISO is a developer-tool argument: when reading a curl response in the terminal, ISO is friendlier. But the API consumer is the dashboard, not a human at a terminal — the human can read the millis on demand by opening DevTools and running `new Date(1748351923456).toISOString()`. Optimize for the primary consumer (the dashboard) at the wire boundary; optimize for the human at the rendering boundary.
**Apply when**: Designing ANY API that returns timestamped points/events/log entries to a programmatic consumer (dashboard, mobile app, another service). Always: `t: number` in the type; same number in the wire payload; client reads as number. Inline comment on the shared type explaining the choice ("// Unix ms; matches Date.now() output. JSON-safe, 2x smaller than ISO, no client parse cost"). Reserve ISO strings for: (a) human-facing exports (CSV downloads), (b) cross-system handoffs where the other system mandates ISO (RFC 3339), (c) log lines indexed by Elasticsearch (where ISO is the convention). For internal server-to-dashboard plumbing, numeric ms is always the right choice. Same logic for date-only fields: prefer `t: "2026-05-27"` (ISO date) over `t: 1748304000000` ONLY when the field's semantics are "a calendar day, not an instant in time" — instants are always numeric ms.

## [2026-06-01] — `CALL: /cmd {json}` marker contract: poor-man's tool-use for standalone bots avoiding Gemini function-calling
**Rule**: When a standalone bot needs the LLM to invoke slash commands AS TOOLS but you do NOT want to import / reimplement the host's full Gemini function-calling loop (with `FunctionDeclaration` schemas, tool-loop turns, function-response parts), use a string-marker convention: the system prompt instructs the model to emit lines like `CALL: /add-exam {"exam_name":"JEE_MAIN","year":2024,"percentile":95,"category":"GEN"}` for state changes. A malformed CALL line (invalid JSON, unknown cmd) is logged and silently dropped — the model sees the next turn's `getContext` reflect the un-changed state and self-corrects on retry. Do NOT make the marker syntax escape-able or context-sensitive (no nested quoting rules); do NOT try to recover partially-malformed JSON; do NOT echo parse errors back to the user. The model uses `CALL:` as a one-way directive; failures are observable only via metrics (`bot.call_marker_parse_failed`) and the next turn's context.
**Why**: Gemini's function-calling SDK requires `FunctionDeclaration` schemas, a multi-turn loop (model emits `functionCall` → server runs tool → server posts `functionResponse` part → model continues), and the loop is tele-side infrastructure (`responder.ts` + `tools/index.ts`) that's far too heavyweight to pull into a sibling-repo standalone bot. The string-marker convention trades structural rigor for zero dependencies — the model emits free text containing marker lines, the dispatch layer scans, executes, strips, and posts the cleanup. The trade-offs are real and accepted: (a) no schema validation at the model's boundary, so the model can emit `CALL: /add-exam {bad json}` which the parser drops silently; (b) the model can't see tool RESULTS in the same turn (the result lands in the next turn's `getContext` snapshot, one round-trip later); (c) markdown rendering could accidentally produce a `CALL:` line that's actually content. All three are acceptable for a single-user dev counselor where the failure mode is "the conversation continues, the AI tries again next turn." The future migration path (when standalone bot graduates to real function-calling) is to keep the marker contract for back-compat AND add structured tool definitions in parallel — markers handle legacy prompts, structured tools handle new ones.
**Apply when**: Any LLM integration that needs the model to trigger side effects but you do NOT have function-calling infrastructure available (sibling repo, embedded LLM, vendor without tool-use). Default to: explicit marker line in the system prompt; strict regex parse in the dispatch layer (anchored, no greedy alternation per lessons-2026-05-27 startsWith+split rule); silent drop on parse failure; metric counter for parse-failure rate; strip markers from user-visible output. Document the contract in the system prompt itself (the model needs to SEE the syntax) AND in a top-of-file comment in the parser. Reserve true function-calling for when the loop has to surface tool RESULTS to the model mid-turn — markers are one-way, function-calling is round-trip.

## [2026-06-01] — `bot_config` singleton via `CHECK (id = 'default')` + seed row: when the operator UX demands "edit, never insert"
**Rule**: For a singleton config table whose ONLY consumer is an operator dashboard (no automated config writers), encode the singleton constraint as `CREATE TABLE bot_config (id TEXT PRIMARY KEY DEFAULT 'default', ..., CHECK (id = 'default'))` AND seed one row via `INSERT INTO bot_config (id) VALUES ('default') ON CONFLICT DO NOTHING;` in the same migration. The dashboard's GET always returns the seed row (never null); PUT becomes a pure UPDATE (no UPSERT/INSERT branching). Contrast with the `CREATE UNIQUE INDEX ON ((TRUE))` pattern (lessons-2026-05-07) — that lets the table start EMPTY (zero rows allowed) and requires every consumer to handle the "no row yet" case via UPSERT. **Apply when**: Any operator-only singleton config table where the row identity is known at design time and the only writes are "edit fields on the existing row." Always: CHECK constraint on the id column AND a seed INSERT in the same migration; route handlers do pure UPDATE; document the magic-string at both the seed and the handler so a future refactor doesn't confuse the rule. Use `((TRUE))` instead when the table can legitimately be empty or when multiple programmatic writers exist. Both are valid; the operator-UX shape decides.

Reserve per-sub-package deps for genuine workspace setups (`pnpm-workspace.yaml`, npm workspaces) where each sub-package is independently installable. ## [2026-06-01] — `chat.databaseUrl` is the universal "host injects per-app DB" capability: design the ctx interface to be capability-extensible, not feature-fixed

**Apply when**: Any plugin/extension framework where the host-to-plugin context interface needs to grow capabilities over time. Always: widen the existing interface with optional fields; document each field's contract in a comment at the interface declaration; host populates all fields unconditionally; plugins read only what they consume; back-compat is automatic via `?.` chaining and `??` fallbacks. Reserve interface forks ONLY when a capability has truly incompatible semantics with existing consumers (rare; usually means the framework should be redesigned, not the interface forked).
