# Lessons Learned

## [2026-04-28] — Neon serverless driver: no `unsafe`, no `query`, no multi-statement SQL
**Rule**: With `@neondatabase/serverless`, the SQL function exposes only two forms: the tagged template (`` sql`SELECT ${x}` ``) and the direct-call form (`sql(text, params)`). There is no `sql.unsafe(...)`, no `sql.query(...)`, and no built-in `sql.transaction([...])` for arbitrary multi-statement strings. A migrator that reads a `.sql` file and tries to send it as one string will throw at runtime.
**Why**: The plan's first migration runner used `sql.unsafe(fileContents)` and the `db/index.ts` helper used `sql.query(text, params)` — both compiled cleanly as `any`-typed access but failed at runtime. The verifier caught this only after running `tsc` against a real `index.d.ts`. Migrations would have crashed on the very first boot, blocking acceptance criterion 1.
**Apply when**: Writing any DB code against the Neon serverless driver — especially custom migrators, schema scripts, or any "run this raw SQL blob" helper. Split multi-statement files on `;` and execute each via `sql(stmt, [])`. Make DDL idempotent (`CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`) since you cannot wrap the whole file in a single transaction with this driver.

## [2026-04-28] — GramJS `EntityLike` rejects `bigint`
**Rule**: GramJS `client.sendMessage(peer, ...)` and similar APIs type `peer` as `EntityLike = string | number | Entity`. Passing `BigInt(chatId)` is a TypeScript error and may or may not coerce at runtime depending on the call path. Convert Telegram IDs to `Number(...)` (safe for IDs < 2^53) or to a string (`peer.toString()`) before handing them to GramJS.
**Why**: Plan stored Telegram chat IDs as `BIGINT` in Postgres (correct — they can exceed 2^31), and the natural reflex was to pass `BigInt(chatId)` back into GramJS calls. That fails to typecheck and is fragile at runtime.
**Apply when**: Bridging persisted Telegram IDs (typically `string` after coming back from a `BIGINT` column) into any GramJS call. Use `Number(id)` for entity arguments; keep the string/bigint form only for DB writes and audit logs.

## [2026-04-28] — Gemini SDK `Part` union needs a widened type predicate
**Rule**: The `@google/generative-ai` `Part` type is a discriminated union; filtering with `(p): p is { functionCall: FunctionCall } => !!p.functionCall` is too narrow and trips TS2677 ("type predicate's type must be assignable to its parameter's type"). Use `(p): p is Part & { functionCall: NonNullable<Part["functionCall"]> } => !!p.functionCall` instead, or just filter without a predicate and use a non-null assertion downstream.
**Why**: A naive narrow predicate fails compilation and cascades into "possibly undefined" errors at every downstream `fc.functionCall.name` access. The function-call loop is hot-path code; this needs to compile cleanly.
**Apply when**: Implementing a Gemini tool/function-call loop that filters `response.candidates[0].content.parts` for `functionCall` parts. Same pattern applies to filtering for `functionResponse` or `text` parts in the SDK.

## [2026-04-28] — `packageManager: "pnpm@9"` is rejected by pnpm v10
**Rule**: The `packageManager` field in `package.json` requires a full semver, not a major-version shorthand. `"pnpm@9"` causes pnpm v10 to warn `Cannot switch to pnpm@9: "9" is not a valid version` and skip the version switch. Use `"pnpm@9.0.0"` (or whatever exact version you want) instead.
**Why**: Corepack and pnpm's self-update logic both expect a parseable semver. The shorthand silently degrades behavior — install still works but the pin is non-functional, which defeats the purpose.
**Apply when**: Setting up `packageManager` in any monorepo root `package.json`. Pin to a real version (e.g. `9.15.0`) and update intentionally.

## [2026-04-28] — Build workspace packages before type-checking their consumers
**Rule**: In a pnpm workspace where `apps/server` depends on a local `packages/shared`, running `tsc --noEmit` on `apps/server` fails with `Cannot find module '@tele/shared'` until `packages/shared` has been built (its `dist/` exists with `.d.ts` files). Always build shared first.
**Why**: Workspace symlinks resolve to `packages/shared/package.json`, whose `"main"` / `"types"` fields point at `dist/`. Without a build step the compiler sees the package but no types. The error is misleading — it looks like a missing dependency, not a missing build artifact.
**Apply when**: Running per-package type checks in CI or locally in a monorepo with `tsc`-built shared packages. Either build shared packages first (`pnpm -F @tele/shared build && pnpm -F @tele/server typecheck`), or configure TS project references with `composite: true` so `tsc -b` handles the build order automatically.

## [2026-04-29] — GramJS Poll API: `TextWithEntities` for question/answers, `correctAnswers` on `InputMediaPoll`
**Rule**: In current GramJS versions, `Api.Poll.question` and `Api.PollAnswer.text` are `Api.TextWithEntities({ text, entities: [] })` objects — not plain strings. Correct answer for quiz polls goes in `Api.InputMediaPoll.correctAnswers: Buffer[]`, not in `Api.Poll` itself. `Poll.id` and `SendMedia.randomId` must be `bigInt.BigInteger` — use `helpers.generateRandomLong()` from the `telegram` package.
**Why**: Telegram migrated from plain strings to `TextWithEntities` across poll fields. Code assuming plain strings compiles if you bypass typechecking but produces a GramJS runtime error or malformed request. The `correctAnswers` placement on `InputMediaPoll` (not `Poll`) is non-obvious from docs alone.
**Apply when**: Any code that constructs `Api.Poll`, `Api.PollAnswer`, or `Api.InputMediaPoll`. Always read the installed `api.d.ts` before writing — never assume the field shapes from memory or older docs.

## [2026-04-29] — Persisting poll state as synthetic outbound message avoids schema migration
**Rule**: When adding a Telegram poll-send feature, persist the poll as a plain text message `[Poll: <question>] opt1 | opt2 | ...` using the existing `messages` table and `insertMessage` helper. This lets the dashboard render it without schema changes or new UI components.
**Why**: Adding a structured `polls` table would require a migration, new repo functions, new API routes, new dashboard components, and vote-tracking infrastructure — all out of scope for an "AI can send polls" feature. The `[Poll: ...]` convention follows the established `[Image: ...]` / `[File: ...]` patterns already in sender.ts.
**Apply when**: Adding new Telegram media types (stickers, polls, location shares, etc.) where the primary goal is AI-initiated sending, not vote/response tracking. Use the synthetic message pattern; defer structured tracking to when vote results must feed back into the AI context.

## [2026-04-30] — FK to internal `chats(id)` UUID, not raw `tg_chat_id` BIGINT
**Rule**: When a new table needs to reference a "person/chat" (e.g. `kanban_tasks.assignee_chat_id`, `reminders.target_chat_id`), make the FK reference `chats(id)` (UUID) — never `chats.tg_chat_id` (BIGINT). Resolve display names via a `LEFT JOIN chats c ON c.id = t.assignee_chat_id` and a `COALESCE(NULLIF(TRIM(CONCAT_WS(' ', c.first_name, c.last_name)), ''), c.username, c.tg_chat_id::text)` expression.
**Why**: `chats.id` is the established internal PK pattern; using it makes joins trivial and avoids any GramJS coercion (the BIGINT path forces `Number(id)` / `String(id)` conversions in every consumer). It also lets `ON DELETE SET NULL` / `CASCADE` semantics work cleanly. The same pattern is in `reminders.target_chat_id` — follow it for parity.
**Apply when**: Adding any new relational column that points at a person, chat, or contact. Always FK to `chats(id)`, never to `tg_chat_id`. Pre-validate the FK in the AI tool handler (`getChatById(id)` → `{ ok: false, error: "unknown contact id; call lookup_contacts first" }`) so the model learns in-loop instead of hitting a 500 on the server.

## [2026-04-30] — Distinguish "key absent" from "key present, value null" in PATCH/PUT repos
**Rule**: For an `updateTask(id, patch)` repo that supports clearing a nullable FK (e.g. `assignee_chat_id: null` to unassign), check `Object.prototype.hasOwnProperty.call(patch, "assignee_chat_id")` — not `patch.assignee_chat_id !== undefined` and not a truthy check. The presence of the key (regardless of value) means "write this field"; absence means "leave it alone". Pair the repo with a zod schema like `z.string().uuid().nullable().optional()` so `null` survives validation.
**Why**: Truthy checks (`if (patch.assignee_chat_id)`) silently drop `null`, making "unassign" impossible. `!== undefined` works only if the JSON parser preserves the distinction (it does), but `hasOwnProperty` is the explicit, intent-revealing form. Also: if the dynamic SET builder finds no real fields it must reject the update rather than running `UPDATE ... SET updated_at = now()` with nothing else — Gemini will sometimes call `update_task(id)` with no other args.
**Apply when**: Writing any partial-update repo where some fields are nullable FKs/strings the caller may want to clear. Same pattern applies to AI tool handlers receiving partial JSON args from Gemini.

## [2026-04-30] — Tool-call response payloads must be trimmed to keep Gemini context small
**Rule**: When a Gemini tool returns a list (`list_tasks`, `list_chats`, etc.), trim each row to the minimum fields the model needs to act — typically `(id, title, status, assignee_id, assignee_name, comment_count)` — and explicitly drop heavy fields like `description`, `body`, attachments, full timestamps. Document this in the tool description so the model knows to call a separate `get_*(id)` tool for the full row when needed.
**Why**: Tool responses are echoed back into the model's context on every turn. A board with 200 tasks × full descriptions can blow past token budgets and degrade response quality. Trimming at the tool boundary (not in the repo) keeps the REST API rich while keeping the AI loop lean.
**Apply when**: Authoring any AI tool that returns a list. Default to a trimmed projection; only return full rows from singular `get_*(id)` tools. If filters could be expressed in SQL (e.g. `assignee_chat_id`, `status`), prefer SQL over post-fetch JS filtering once the table grows past a few hundred rows.

## [2026-04-30] — `WsEvent` discriminated union: verify consumers have no silent `default` branches
**Rule**: When extending the `WsEvent` union in `packages/shared/src/types.ts` with a new `{ type: "..."; payload: ... }` variant, audit existing `useWsEvent` / switch consumers to confirm none have a `default:` branch that would silently swallow the new type. The `useWsEvent<T extends WsEvent["type"]>` generic picks up new variants automatically — but only if no consumer has a catch-all that drops them.
**Why**: TypeScript's exhaustiveness checking only fires on `switch` statements with a `never`-typed default. A `default: return;` or missing `default` in a hand-written dispatcher will compile fine and silently drop the new event at runtime — invisible until QA notices the dashboard doesn't refresh.
**Apply when**: Adding any new variant to a discriminated union consumed in multiple places. Grep for the union name and audit each consumer; prefer the generic `useWsEvent<"specific:type">(handler)` pattern over a hand-rolled switch so new variants are opt-in by name.

## [2026-04-30] — Split AI tool pairs into trimmed-list + full-get; never one tool that does both
**Rule**: When the AI needs both "discover what's available" and "load the full thing", expose two tools: `list_X` returns only `(id, name, description)` for every row, and `load_X({ id?, name? })` returns the full payload for one row. Don't add a `verbose: true` flag to `list_X` — the model will pass it.
**Why**: A single `list_X(verbose=true)` lets Gemini one-shot a 200-row × full-content response into context, blowing the budget. Two distinct tools force the model to choose `id` first, then load only what it needs. The discoverability tool stays cheap; the load tool is intentionally heavy but scoped to one item. This is the deliberate counterpart to the existing trim-list rule — same goal, structural enforcement instead of convention.
**Apply when**: Designing any tool surface for content the model needs to *select among, then read in full* (skills, prompts, documents, templates, runbooks). The list/get split is the right shape; resist the temptation to merge them.

## [2026-04-30] — AI tool args: accept both `id` and `name`, prefer `id`, error if neither
**Rule**: For lookup tools where the model might know either an opaque `id` or a friendly `name` (skills, contacts, chats, channels), accept both as optional args, prefer `id` when both are present, and return `{ ok: false, error: "provide id or name" }` when neither is given. Don't make either required.
**Why**: Gemini often remembers a friendly name from earlier in the conversation (`"grail-explorer"`) but not the UUID. Forcing `id`-only makes the model call `list_*` again to look up the id — wasted turn. Forcing `name`-only breaks when the user renames. Accepting both with a documented precedence is the friction-free path. The explicit "provide id or name" error teaches Gemini in-loop instead of silently returning an empty result.
**Apply when**: Any AI tool that resolves a single record by some user-meaningful identifier. Same pattern works for `chat`, `contact`, `template`, etc. Document the precedence in the tool description so the model can plan around it.

## [2026-04-30] — Path-takes-precedence-over-inline at load time, not at write time
**Rule**: When a record can hold either inline content or a filesystem path pointing at the source of truth (e.g. `skills.content` vs `skills.path`), store both columns but resolve at *load* time: if `path` is non-null, read the file fresh; otherwise use `content`. Do NOT sync `content` from `path` on write — leave `content = ''` when `path` is set.
**Why**: Editing a SKILL.md file on disk should automatically be reflected the next time the AI loads the skill. If you eagerly cache the file's contents into `content` on write, the row goes stale the moment the file changes. Storing `content = ''` alongside a non-null `path` makes the precedence visually obvious in the DB and prevents drift.
**Apply when**: Any "ref-or-inline" data model — skills, prompt templates, config blobs, image-by-URL-or-binary. Document the precedence in the schema comment and enforce at the load boundary, not the write boundary.

## [2026-04-30] — Catch Postgres unique-violation (`23505`) and return 409, never let it bubble as 500
**Rule**: When a route inserts/updates a row with a `UNIQUE` column the user controls (e.g. `skills.name`, `chats.username`), wrap the DB call in `try { ... } catch (err) { if (err?.code === "23505") return res.status(409).json({ error: "name already exists" }); throw err; }`. Apply on both POST (create) and PUT (update — rename collisions hit the same constraint).
**Why**: Postgres throws `23505` for unique violations; without the catch, Express returns 500 with a stack trace and the dashboard shows "Server error" — useless to the user, who just typed a duplicate name. A targeted 409 with a friendly message lets the form display the real problem ("name already exists") inline. Forgetting to apply this on PUT is the common miss — only POST gets remembered.
**Apply when**: Any route writing to a column with a `UNIQUE` constraint that's user-supplied. Audit both create AND update paths. Same pattern for FK violations (`23503`) → 400 "referenced row does not exist".

## [2026-04-30] — Mirror create-time validation in update routes — partial PATCH/PUT often skips invariants
**Rule**: When a CREATE route enforces "at least one of A or B must be provided" (e.g. `content` or `path` for skills), the corresponding UPDATE/PUT route MUST re-check the invariant after applying the patch. A naive `PUT { content: "", path: null }` will happily clear both and leave the row in a state CREATE would have rejected. Read the current row, apply the patch in memory, then re-validate before persisting.
**Why**: The verifier flagged this exact gap on the skills PUT route — POST rejected `{content: "", path: null}` but PUT accepted it, leaving a skill that loads to empty instructions. Partial-update endpoints inherit none of the create-time invariants for free. Either re-validate after merging the patch, or document the invariant as "create-only" and reject the field in the update schema.
**Apply when**: Any UPDATE/PATCH/PUT route on a table whose CREATE has cross-field validation (mutual exclusivity, "at least one of", "exactly one of"). Walk through every CREATE-time check and confirm it still holds after the partial update.

## [2026-04-30] — `enabled`/visibility checks must be re-applied at every consumption point, not just at list time
**Rule**: When a row has an `enabled` flag and the dashboard hides it from a `list_*` tool by filtering on `enabled = true`, the corresponding `load_*` / `get_*` / `invoke_*` consumer MUST re-check `enabled` after the lookup and refuse if disabled. Hiding from list does not protect against stale ids — the model may have remembered an id from earlier in the conversation, before the user toggled it off.
**Why**: Verifier flagged this on `load_skill`: `list_skills` filters `enabled=true`, but `load_skill(id)` does not, so a model with a stale id from earlier in the same conversation can still load a now-disabled skill. The same applies to MCP servers, prompt templates, scheduled jobs — anything with a soft-disable flag. Defense-in-depth at every consumer, not just the discovery surface.
**Apply when**: Any soft-disable flag (`enabled`, `active`, `archived = false`) on a table consumed by multiple tools or routes. Treat the flag as an authorization check that must run at every read path, not just the listing path.

## [2026-04-30] — `pnpm -F <pkg> typecheck` fails silently when no `typecheck` script exists
**Rule**: `pnpm -F @tele/server typecheck` exits with `ERR_PNPM_RECURSIVE_RUN_NO_SCRIPT` if the package has no `"typecheck"` script in its `package.json`. The error is easy to mistake for "no typecheck needed" or "passed". Always either add `"typecheck": "tsc --noEmit"` to every package's `package.json`, or fall back to `cd <pkg> && npx tsc --noEmit` and check the exit code explicitly.
**Why**: A plan step that says "run typecheck" can read as PASS when it actually never ran. The verifier on this task caught it only by re-running with `npx tsc --noEmit` directly. Repo hygiene fix: add the script to every package; CI fix: assert the script exists before running.
**Apply when**: Writing or following a plan step that invokes `pnpm -F` with a script name. Either confirm the script is defined first (grep `package.json`) or use the direct `npx tsc --noEmit` form. Same trap applies to `lint`, `test`, etc. — pnpm recursive scripts are silent on absence.

## [2026-04-30] — Seed rows with machine-specific paths are fine if idempotent and user-editable
**Rule**: It's acceptable to ship a migration with a hard-coded `INSERT ... ON CONFLICT DO NOTHING` containing a developer-local path (e.g. `/Users/foo/Downloads/grail-explorer/SKILL.md`) provided: (1) the seed is idempotent so re-runs on a clean DB don't error, (2) the dashboard exposes a way to delete or re-point the row, and (3) the path is documented as "edit me on a fresh install" in the plan. Don't gate the migration on the path's existence at boot.
**Why**: For a single-user tool, blocking the migration on a missing local file would break onboarding for any other developer. `ON CONFLICT DO NOTHING` plus an editable dashboard is the pragmatic middle ground — the seed is convenience, not correctness. The AI tool that consumes the path should fail gracefully ("path not readable: <path>") so the model surfaces the issue to the user.
**Apply when**: Seeding any developer-local resource (file paths, MCP server commands, local URLs) into a migration. Don't put hard-coded local paths into seeds for multi-tenant or production deployments — only for single-user dev tools where the user can fix the seed via the UI.

## [2026-05-02] — Return `{ handled, type }` (not just `boolean`) from interceptors so the caller can vary follow-up side effects
**Rule**: When an "intercept-or-fall-through" dispatcher (slash command router, middleware shim, command parser) sits in front of a pipeline that already fires its own side effects (reactions, telemetry, status updates), return a discriminated result like `{ handled: true, type: "ai_prompt" }` rather than a bare `true`. The caller uses `type` to decide whether to fire its own outer side effect or let the inner branch's side effect stand alone. Never return only `boolean` if any branch reuses the existing pipeline.
**Why**: For slash commands, the `ai_prompt` branch calls `generateAndReply`, which already fires `reaction_done` on success. If the router *also* fires `reaction_done` after seeing `handled: true`, the user gets two reactions on the same message — a visible bug. The discriminated return lets the router skip the outer reaction precisely for `ai_prompt` and fire it for `shell` / `message` (which use only `sendReply` and don't double-fire).
**Apply when**: Adding any pre-pipeline interceptor where some branches go through the existing pipeline and some don't. The signature `{ handled: boolean; type?: Variant }` is the minimum needed; richer payloads are fine if downstream needs them. Document which branches are "pass-through to inner side effects" vs "self-contained" in the dispatcher's doc comment.

## [2026-05-02] — Optional `opts` param with `??` fallback is the safe way to add per-call overrides without breaking callers
**Rule**: When you need to add a per-call override to an existing function (e.g. `systemInstructionOverride` on `generateAndReply`), append an optional `opts?: { ... }` parameter at the end and use `opts?.field ?? defaultExpression` at the use site. Do NOT change the existing parameter list, do NOT make `opts` required, and do NOT mutate any module-level state to thread the override through. Existing call sites continue to work with `opts` defaulting to `undefined`.
**Why**: Mutable module-level overrides (e.g. setting a `currentSystemPrompt` global before calling) leak across concurrent calls and create surprising action-at-a-distance. A required `opts` argument forces every existing caller to be updated in the same diff, blowing up the blast radius. The `?? default` pattern keeps the override scoped to one call and adds zero behavioral change for callers that don't pass it.
**Apply when**: Extending any existing function with a new "optional behavior tweak" used by exactly one new caller. The pattern is also the right shape for dependency injection in tests — pass `opts.clock`, `opts.now`, `opts.logger` for overrides without restructuring the signature.

## [2026-05-02] — Soft-rejecting parse failures (not just lookup failures) lets shorthand syntax coexist with real content
**Rule**: When a feature reserves a syntax prefix (e.g. `/` for slash commands, `@` for mentions, `:` for emoji), the dispatcher must distinguish "matches the prefix but fails to parse as our syntax" from "parses but no row found" — and BOTH should fall through to the default handler, not error. For slash commands, `/path/to/file` starts with `/` but does not match `^([a-z0-9_-]+)(?:\s+([\s\S]*))?$/i`; the dispatcher returns `{ handled: false }` so the AI receives `/path/to/file` as ordinary user text rather than getting a 400 or a "command not found" reply.
**Why**: Users casually type things like `/Users/foo/...`, `/etc/hosts`, or `/r/subreddit` in chat. Hard-rejecting any `/`-prefixed text as a command attempt would block normal conversation. The two-stage filter (regex parse → DB lookup → both fall through on miss) means the feature only activates on syntactically clean command names that also resolve to a configured row.
**Apply when**: Designing any prefix-triggered command/mention/macro system where the prefix character also occurs in normal content (paths, URLs, user handles). Treat the regex parse as a soft filter; never throw, never reply with an error — only intercept on a positive match all the way through.

## [2026-05-02] — `child_process.spawn(shell, ["-c", cmd])` with explicit timeout + SIGKILL is the correct shell-exec primitive
**Rule**: For server-side "run this user-authored shell command and capture output" features, use `spawn("zsh", ["-c", cmd])` (or `/bin/sh`) with a manual `setTimeout` + `child.kill("SIGKILL")` for hard timeouts, accumulate stdout/stderr in string buffers via `data` events, and resolve a `{ code, stdout, stderr, timedOut }` shape from the `close` event. Do NOT use `exec` / `execSync` (no streaming, no clean timeout) and do NOT use `execFile` (no shell expansion — defeats the point for user-authored commands). Always handle the `error` event in addition to `close` to catch spawn failures (e.g. zsh missing on the host).
**Why**: `exec` buffers everything in memory with a default 1MB cap and silently truncates beyond that; its `timeout` option sends SIGTERM, which a stuck process can ignore. `spawn` + `setTimeout` + SIGKILL guarantees the process dies on the deadline. The `code` may be `null` (process killed without exit) — coalesce to a sentinel like `-1` so the caller can detect it. Truncate the stdout/stderr at the *output* boundary (e.g. 3500 chars for a Telegram reply), not the buffer boundary, so you keep the tail of the output for short-but-useful commands and only clip the noisy ones.
**Apply when**: Implementing any "run a shell command from a config row" feature (slash commands, custom hooks, webhook actions). The `{ timedOut: true }` branch is the easy-to-forget one — make sure the reply explicitly says "Command timed out after Ns" rather than reporting an empty stdout as success. For untrusted input contexts, this whole pattern is unsafe — that's a different problem (sandboxing/allowlisting), not a different exec API.

## [2026-05-02] — `String.replaceAll(literal, value)` is fine for trusted templates; resist over-engineering to `{{handlebars}}`
**Rule**: For a single-token substitution like `{args}` in a user-authored template, `template.replaceAll("{args}", value ?? "")` is the right primitive. Don't reach for Handlebars, Mustache, or a hand-rolled tokenizer unless the feature needs conditionals, loops, or escaping. Document the limitation ("the literal string `{args}` is unsupported in output") in the dashboard help text and move on.
**Why**: A real template engine adds a dependency, a security surface (HTML escaping defaults vary), and a learning curve for the user — all to support features no one asked for. `replaceAll` is one line, has no escaping behavior to misconfigure, and the "you can't emit a literal `{args}`" trade-off is acceptable for a single-user tool. If the feature later needs `{user}` and `{date}` too, a 5-line `Object.entries` loop still beats pulling in Handlebars.
**Apply when**: Adding any single-or-few-token substitution in a config string the user authors. Default to `replaceAll` per token; only escalate to a real engine when you need conditionals or escaping. Same logic for env-var-style `${VAR}` substitution in non-shell contexts.

## [2026-05-02] — `as any` cast for SDK fields not in published types — anchor to a comment naming the SDK version
**Rule**: When a Google/OpenAI SDK accepts a field at the REST boundary that is not yet in the TypeScript types (e.g. `generationConfig.responseModalities` in `@google/generative-ai` v0.21 for the image-preview model), the right escape hatch is a localized `as any` cast on just the offending object — `generationConfig: { responseModalities: ["TEXT", "IMAGE"] } as any`. Always co-locate a one-line comment naming the SDK version and stating that the SDK forwards the field to the REST body unchanged. Do NOT widen the cast to the whole call, do NOT `// @ts-expect-error` (it'll flip the wrong way on SDK upgrades), and do NOT fork or re-declare the SDK's types.
**Why**: Vendor SDKs lag the REST surface by weeks, especially for preview features. A bare `as any` with no comment becomes a mystery in three months when an SDK upgrade tightens the type and someone wonders why we had to dodge it. The comment is the documentation. A `@ts-expect-error` is worse: when the SDK adds the field, the directive becomes the new error and breaks the build, but a plain `as any` keeps working silently — which is what you want for a forward-compatible cast.
**Apply when**: Any time you're calling a vendor SDK with a preview/beta field that is documented in the REST API but missing from the typed surface. Cast at the smallest possible scope (the single object literal, not the function call), comment with the SDK version, and ideally gate the field behind a config flag so a future operator can flip it without code changes if the SDK rejects it.

## [2026-05-02] — Gate fast-changing vendor model names behind an env var with a sane default
**Rule**: For vendor model identifiers that are known to drift (Gemini image-preview model has gone through `gemini-2.0-flash-exp-image-generation` → `gemini-2.0-flash-preview-image-generation` → `gemini-2.5-flash-image-preview` in months), expose the model name as an env-var with a default in the config schema (`GEMINI_IMAGE_MODEL: z.string().default("gemini-2.0-flash-preview-image-generation")`) and reference `config.GEMINI_IMAGE_MODEL` at the call site. Do NOT hard-code the string in the tool file.
**Why**: When the model is renamed or deprecated, the user can flip the env var and restart — no code change, no redeploy of a new build. Hard-coding forces a code change on every drift event. The default keeps the zero-config experience intact for the common case. Same pattern applies to embedding models, vision models, and any preview-tier name that the vendor hasn't committed to long-term.
**Apply when**: Adding any new vendor model integration where the model id is in a preview/experimental tier, or where the vendor has historically renamed models. Always pair the env var with a default; document the rename history in a comment near the schema entry if it's especially churny.

## [2026-05-02] — Temp-file write-then-send pattern: `try/finally` with `unlink().catch(() => {})`
**Rule**: For a tool that materializes binary data (image bytes from a model, downloaded blob) just long enough to hand off to a transport that needs a path (Telegram `sendFile`, ffmpeg, image-resize CLIs), the shape is: write to `os.tmpdir()` with a randomized filename (`Date.now()`-`randomBytes(4).toString("hex")`-`.ext`), wrap the transport call in `try { await transport(tmpPath); return success; } finally { await fs.unlink(tmpPath).catch(() => {}); }`. The `.catch(() => {})` on unlink is intentional — failures to clean up should never mask the real error from the transport.
**Why**: A naive `await fs.unlink(tmpPath)` after the transport will throw if the file is already gone (e.g. on EBADF, or if a previous cleanup ran), and an unhandled throw from `unlink` would replace the meaningful error from `sendFile` with a confusing "ENOENT on cleanup". The randomized filename avoids collisions if two concurrent tool calls race. `os.tmpdir()` is OS-managed so leaked files (process killed between write and unlink) get reaped on reboot — don't over-engineer a custom GC for a single-user dev tool.
**Apply when**: Any tool/handler that produces a blob and feeds it to a path-based downstream API. The `try/finally + swallow on unlink` shape is the minimum-surprise pattern; the cost of a leaked tmp file on a process crash is acceptable, the cost of a confusing stack trace on every cleanup hiccup is not.

## [2026-05-02] — Multimodal SDK responses: trust `mimeType` for extension, default safely on unknown
**Rule**: When extracting binary parts from a multimodal LLM response (`response.candidates[0].content.parts[].inlineData`), pick the file extension from `inlineData.mimeType` (`image/png` → `.png`, `image/jpeg` → `.jpg`, `image/webp` → `.webp`) and default to `.png` for any unknown or missing mime. Don't sniff magic bytes; don't enforce a single output format on the model. Telegram (and most consumers) renders all three correctly regardless of extension, so the default is forgiving.
**Why**: The model picks the format based on its own heuristics — forcing `.png` always works, but uses larger files for photographic content where JPEG would be 1/10th the size. Trusting `mimeType` keeps the file small and the content-type accurate. Defaulting to `.png` on unknown is safe because PNG is the universal "lossless image" format and Telegram doesn't actually look at the extension — but downstream tools (S3, browsers via Content-Disposition, image processors) do.
**Apply when**: Any code extracting `inlineData` from a Gemini/OpenAI multimodal response, or any handler that writes vendor-supplied bytes to disk and needs an extension. Don't add a magic-byte sniffer; the mime field is reliable when present, and the safe default covers the rest.

## [2026-05-02] — Prefer extending the existing `ai_prompt` slash branch with a system-prompt override over adding a new dispatcher branch
**Rule**: When adding a new slash command that should trigger AI behavior with a tweaked instruction (e.g. `/draw` should force a `generate_image` tool call), seed the row with `type='ai_prompt'` and put the directive in the `action` column as a system-prompt override. Do NOT add a new dispatcher branch (`type='draw'`, `type='generate'`) — the `ai_prompt` branch already passes `action` as `systemInstructionOverride` to `generateAndReply`, which is exactly the right hook. Each new slash command becomes one migration row, zero code changes to the dispatcher.
**Why**: Adding a new `type` value would require: (1) a CHECK constraint migration to widen the allowed types, (2) a new dispatcher branch, (3) new tests, (4) a new entry in the `{ handled, type }` discriminated return. Versus: one INSERT row that uses the existing `ai_prompt` plumbing. The system-prompt override is the universal "make the AI do this specific thing" lever — resist the urge to specialize.
**Apply when**: Any new slash command whose behavior is "have the AI do X with this instruction." Seed it as `ai_prompt` with the action being the override prompt. Only add a new `type` if the behavior genuinely cannot be expressed as "AI with this system override" — e.g. a pure shell-out, a webhook trigger, a synchronous DB query that must short-circuit the AI loop entirely.

## [2026-05-04] — Public webhook routing via auth-hook scope, not per-route allowlist
**Rule**: When adding a public (unauthenticated) HTTP endpoint to a Fastify app whose auth is enforced by a global `onRequest` hook, register the route OUTSIDE the hook's URL prefix scope (e.g. mount at `/webhook/:id` when the hook only fires on `/api/*`). Do NOT widen the hook to add a `PUBLIC_PATHS` allowlist; do NOT decorate the route with a "skip auth" flag. The path itself is the auth boundary.
**Why**: A `PUBLIC_PATHS` allowlist is an action-at-a-distance config that someone editing the route forgets to update. The path-prefix split is enforced by URL routing — impossible to mis-configure on the route side. The trade-off is the public route can't share the `/api/*` namespace, but for webhooks called by external services (Telegram, Stripe, GitHub) that's the right separation anyway. ALSO: add a top-of-file comment in the public route noting the dependency on the auth hook's scope, so a future operator widening the hook to `/*` gets a breadcrumb.
**Apply when**: Any new endpoint a third party calls without your auth cookie/header (webhooks, OAuth callbacks, health checks for external monitors). Mount outside the protected prefix; comment the dependency on the hook's scope at both ends (the route and the hook).

## [2026-05-04] — `timingSafeEqual` needs length-mismatch handling — wrap, don't try/catch
**Rule**: `crypto.timingSafeEqual(a, b)` THROWS `RangeError: Input buffers must have the same byte length` when buffers differ in size — it does not return `false`. Wrap calls in a helper like `safeEqualHeader(actual, expected)` that converts both inputs to Buffers, returns `false` on length mismatch BEFORE calling `timingSafeEqual`, and returns `timingSafeEqual(a, b)` otherwise. Do NOT try/catch the throw — pre-checking the length is faster, more readable, and avoids the perception that any other crypto error is being swallowed.
**Why**: A naive `timingSafeEqual(Buffer.from(headerVal), Buffer.from(expected))` crashes the request handler with an uncaught throw the moment an attacker (or a misconfigured client) sends a header of the wrong length — turning what should be a 401 into a 500. The pre-length-check is the cleanest fix; try/catch works but loses the chance to detect genuine crypto failures. The helper also normalizes the type — header values can arrive as `string | string[] | undefined` from Fastify.
**Apply when**: Any time you compare a user-supplied secret/token header against a stored value. The pattern is identical for HMAC signatures, webhook secrets, API keys passed in headers. Default to a small `safeEqualHeader` helper in a shared util file rather than inlining at each call site.

## [2026-05-04] — Webhook handlers: always 200 except for the ONE error code the sender uses to back off
**Rule**: For an HTTP webhook handler, return `200` for nearly every outcome — bad request body, unknown bot/resource id, disabled row, parse failure, internal exception during async work — and reserve non-200 codes for the one signal the sender actually acts on. For Telegram, that signal is `401` on `secret_token` mismatch (Telegram stops retrying); every other condition uses `200` with silent drop. Wrap the AI/business logic in `.catch(err => logger.error(...))` AFTER the 200 has been returned so a Gemini failure or DB hiccup does not leak into the HTTP response.
**Why**: Webhook senders (Telegram, Stripe, GitHub) retry on non-200 with exponential backoff. Returning `400` for "unknown bot id" makes Telegram retry the same broken update for hours, flooding logs and possibly racing with cleanup. Returning `500` on a transient downstream error (Gemini timeout) makes Telegram resend the same user message — duplicating the AI response and any side effects (sent messages, DB writes). The "always 200" rule sounds wrong from a REST hygiene standpoint, but webhooks have inverted semantics: you are the *consumer*, the sender is the *server*, and your job is to acknowledge receipt fast.
**Apply when**: Any inbound webhook handler. Map error responses to: 401 = "stop retrying, you're misconfigured" (the signal); 200 = literally everything else. Document the chosen non-200 codes inline so a future maintainer doesn't add a 500 path on instinct.

## [2026-05-04] — Fire-and-forget ACK before async work — `void` + `.catch`, never `await`
**Rule**: When an inbound message has a UI-acknowledgment requirement that the upstream service uses to clear a loading state (Telegram `callback_query.id` → `answerCallbackQuery`, Slack `response_url`, Discord interaction tokens), call the ACK with `void someApi(...).catch(err => logger.warn(...))` — explicit `void` prefix, `.catch` tail to swallow errors — and IMMEDIATELY proceed to the actual work. Do NOT `await` the ACK before the AI/handler logic runs.
**Why**: `await ack(); await ai();` serializes a network round-trip in front of the slow path, leaving the user staring at a spinner for hundreds of extra milliseconds. The ACK has no return value the handler needs and its failures are non-fatal (the spinner just stays a few extra seconds). The `void` prefix is load-bearing — without it, lint rules like `no-floating-promises` will flag the line and the next person "fixes" it by adding `await`, regressing the latency. The `.catch` is not optional — an unhandled promise rejection in a `void`-discarded promise will crash Node 15+.
**Apply when**: Any handler whose upstream protocol has a "clear the spinner" or "I got it" signal separate from the actual response. Always: `void ack().catch(log); doRealWork();`. If TS or lint complains, the lint config is the bug — add a comment, not an `await`.

## [2026-05-04] — Reuse the route/import path when replacing a feature; let the seam be the file contents
**Rule**: When deleting and replacing a feature that owns a sidebar entry, frontend route, page filename, and API path (e.g. swapping the persona-bot CRUD for a Telegram-bot CRUD that occupies the same "Bots" name), KEEP the existing identifiers — `/bots` route, `pages/Bots.tsx` filename, `/api/bots` REST path, `qk.bots` query key, `{ to: "/bots", label: "Bots" }` sidebar entry. Replace ONLY the file contents. The router (`App.tsx`), the sidebar (`Sidebar.tsx`), and the query-key barrel become zero-edit files.
**Why**: Editing the router and sidebar to point at a renamed file/route doubles the diff for no behavioral gain and makes the rollback bigger. Reusing the names also lets the dashboard's bookmark/URL stability survive the swap — a user who had `/bots` in browser history still lands on the same page, just with new contents. The trade-off is git blame on the page file shows a sudden total rewrite (good — that IS the change), and the internal server filenames keep a longer prefix (`telegramBots.ts`) to disambiguate from the deleted persona-bot file in version-control history.
**Apply when**: Any feature swap where the new feature occupies the same conceptual slot as the old one. Keep the user-facing names (URL, sidebar label, query key) identical; let the new behavior live entirely inside the file contents. Add longer disambiguating prefixes only on internal server filenames where git-blame archaeology matters.

## [2026-05-04] — Atomic delete-then-recreate phases: do NOT typecheck or commit between
**Rule**: When a multi-phase plan deletes a file in an early phase and recreates it (with new contents at the same path) in a later phase — e.g. Phase 0 deletes `pages/Bots.tsx` and Phase 7 writes a new `pages/Bots.tsx` — treat the delete-and-recreate phases as ONE atomic change set. Do NOT run `tsc --noEmit` between them, do NOT commit between them, do NOT push between them. The `App.tsx` import will be dangling for the duration of the gap. If parallelism is desired, do the recreate step IMMEDIATELY after the delete step so the import is never broken on disk.
**Why**: A typecheck run between Phase 0 and Phase 7 will fail loudly with "Cannot find module './pages/Bots'", which is technically correct but uselessly distracting — the plan acknowledged the gap and the recreate is queued. A commit between the two phases creates a build-broken commit in history, which (a) breaks `git bisect` going forward and (b) blocks anyone who checks out that SHA from running the dev server. The atomic-set rule sidesteps both. Document the atomicity in the plan ("treat Phases 0–7 as one change set; do not commit between them") so a future operator doesn't split them across PRs by accident.
**Apply when**: Any plan where a file is deleted in one step and a same-pathed replacement is written in another. The atomicity declaration belongs in the plan's Risks section AND in a comment on the deletion step itself, not just in the executor's head.

## [2026-05-04] — Webhook + CRUD route registration ordering vs catch-all plugins
**Rule**: When Fastify has `setNotFoundHandler` or a static-file plugin that catches `/*`, register every explicit route — including the public webhook — BEFORE those catch-all registrations. The webhook route at `/webhook/:botId` is no exception, even though it lives outside `/api/*`: the static plugin's catch-all will swallow it if registered first. Order in `api/index.ts` should be: explicit API routes → explicit public routes (webhooks) → static plugin → `setNotFoundHandler`.
**Why**: Fastify resolves routes in registration order for catch-all conflicts. A `staticPlugin` mounted at `/` with `wildcard: true` will eat any request that doesn't match an already-registered route. If you register webhook routes AFTER the static plugin, the webhook POST gets a 404 (or worse, a "not found" HTML page returned to Telegram, which retries forever because non-200). The fix is registration order, not a `before:` hook or a `priority` config — Fastify doesn't have those. Put a comment near both the static plugin and the webhook registration noting the ordering dependency.
**Apply when**: Any new public route in a Fastify app that uses static-file serving or a global not-found handler. Always register explicit routes first; never assume route specificity wins over registration order with catch-all plugins.

## [2026-05-04] — `BIGINT` vs JS Number: Telegram chat ids fit in 2^53, but persist as text-safe form anyway
**Rule**: Telegram chat ids are `int64` on the wire and exceed `2^31`, but currently fit comfortably in `2^53` (the JS-safe integer range). Persist them as `BIGINT` in Postgres and read them back as `string` (most pg drivers do this by default to preserve precision). When passing them BACK into a third-party API (GramJS `EntityLike`, Bot API `chat_id` JSON field), `Number(idStr)` is safe today — but prefer `String(idStr)` if the API accepts string form, since the cost of a future Telegram id-space expansion is "all your existing code is silently wrong" with no error.
**Why**: The Bot API actually accepts `chat_id` as either number OR string in JSON, and GramJS accepts string for `EntityLike`. Defaulting to string-form propagation through the stack is the forward-compatible choice for ~zero cost. The only place you MUST convert to number is when feeding the id into a JS `Map` key alongside numeric keys (don't mix), or comparing to a value the SDK has already coerced to number. The lesson is to prefer the string form whenever the API allows it — a one-character change today saves a debug session in 2030.
**Apply when**: Any code path that pulls a Telegram chat id from the DB and hands it to an external API. Default to string. Only call `Number(id)` at the explicit API boundary where the SDK demands a number, with a comment naming the SDK and version. Never use `BigInt(id)` for GramJS — the type rejection is documented in 2026-04-28's lesson; for Bot API the string-or-number flexibility makes BigInt unnecessary.

## [2026-05-04] — Webhook secret rotation: explicit delete + recreate, not in-place mutation
**Rule**: For per-row webhook secrets (or any rotateable credential stored alongside other config), document that rotation requires DELETING the row and recreating it — do NOT add a "rotate secret" PATCH endpoint or auto-regenerate on update. The secret is a one-time set at create-time; the dashboard's edit form does not expose it as editable.
**Why**: A "rotate" endpoint introduces a window where the new secret is registered with the third party (Telegram `setWebhook`) but the old secret is still in flight on a pending request — opening a brief auth-failure gap. More importantly, it's surface area no one asked for: rotation is rare (compromise, lost backup), and delete-and-recreate is a 30-second operator action that handles the registration cleanly. The simpler model also keeps the row's identity tied to its secret — useful for audit logs ("secret X was used during this incident" maps to a single row, not "row Y at time Z had secret X1, then X2").
**Apply when**: Any per-row credential where rotation frequency is "almost never" and the operator is a human, not an automated service. If rotation needs to be automated (e.g. compliance-mandated quarterly rotation), the model is different: PATCH endpoint, dual-secret window, callbacks accept both for a grace period. For the single-user dev-tool case, "delete + recreate" wins on simplicity.
